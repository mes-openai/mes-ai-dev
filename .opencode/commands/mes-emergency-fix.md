# mes-emergency-fix

## 功能说明

处理生产环境紧急问题的快速修复流程。绕过正常开发流程，以最短时间恢复服务，事后补充完整文档。

## 使用方式

```
/mes-emergency-fix
```

**适用场景**：
- 生产环境核心功能异常，需要立即修复
- 发现紧急缺陷需要 hotfix
- 需要快速回滚到上一版本

**前置条件**：无（紧急情况可绕过正常前置条件）

**预期耗时**：30分钟 - 4小时（视问题复杂度）

**步骤级强制门禁**：每个步骤的产出在进入下一步骤前，必须先执行步骤级门禁审查；未通过时当前步骤打回重做，不得将未通过结果注入下一步骤。

**图谱/TDD Skill 继承说明**：当本命令编排的 Skill 命中 GitNexus / graphify / TDD 单元测试场景时，默认继承 `.opencode/references/mes-ai-reference/rules/governance/skill-graph-tdd-consumption-standard.md`；图谱能力仅作为证据导航或导读，不替代事实证据、阶段结论或门禁判断；TDD/Mockito/路径兼容规则以该统一标准为准。

**阶段完成产物报告**：
- `/mes-emergency-fix` 完成后必须输出一份阶段完成产物报告
- 报告必须说明：紧急修复阶段标准产物、已生成文件、各文件作用、未生成文件及未生成原因
- 报告未生成或内容不达标，不得通过紧急修复阶段退出门禁

**阶段详细审查报告**：
- `/mes-emergency-fix` 完成后必须输出 `emergency-review-report.md`
- 落盘目录：`mes-ai-dev/workspace/emergency/EMG-YYYYMMDD-XXX/`
- 报告必须满足详细审查报告最小字段集，并包含明确审查时间与完整证据链
- 报告未生成或内容不达标，不得通过紧急修复阶段退出门禁

## GSD 执行增强说明

`/mes-emergency-fix` 默认优先采用 GSD 执行增强模式，以“先恢复服务、再补治理”为基本原则。

### 一、适用原则

紧急修复阶段必须优先明确当前目标，例如：

- 恢复服务
- 缩小影响范围
- 阻止问题继续扩散
- 形成临时保护措施
- 产出可上线热修包

不得在目标未明确前直接进入长链路文档补齐。

### 二、何时必须局部切回 Strict

命中以下情况时，必须在对应步骤切换到 Strict 子流程：

- 数据库不可逆变更
- 高风险安全修复
- 关键状态模型修正
- 关键发布路径重大风险
- 当前修复动作可能引入新的全局收口风险

### 三、blocker 规则

所有 blocker 必须先判断其是否阻碍“恢复服务”这一主目标。

- 若阻碍恢复服务，则视为硬阻塞
- 若不阻碍恢复服务，只影响完整治理，则视为软阻塞或后补项
- 若受外部环境/协同限制，但存在临时缓解措施，则视为外部依赖阻塞

紧急修复中不得因非关键 blocker 整体阻断恢复动作。

### 四、最小可交付定义

紧急修复阶段的最小可交付至少包含：

- 服务已恢复或影响显著收敛
- 当前修复动作已明确
- 当前风险与观察项已记录
- 后补治理动作已明确
- 下一步建议已输出

### 五、GSD Continue Exit 条件

进入 GSD Continue Exit 至少必须满足：

1. 当前故障已恢复或显著缓解
2. 当前修复路径明确
3. 当前风险与观察项已记录
4. 后补治理动作已明确
5. 推荐下一步已输出
6. 已完成收尾扫描

### 六、Completion Sweep

紧急修复结束前至少检查：

- 恢复状态是否真实成立
- 是否仍存在高风险未控制项
- 是否已明确 refresh / retrospective / 正式修复动作
- 是否已明确观察期与回滚关注项

### 七、输出要求

本命令的最终输出至少应包含：

- 当前模式
- 当前目标
- 当前修复状态
- blocker 状态
- 当前最小可交付状态
- 风险与观察项
- 后补动作
- 推荐下一步

### 八、模板绑定说明

- 命中 blocker 时，应使用 `.opencode/references/mes-ai-reference/templates/governance/blocker-record-template.md` 记录 blocker 分类、代偿动作与后补动作
- 以 GSD Continue Exit 继续时，应使用 `.opencode/references/mes-ai-reference/templates/governance/minimum-deliverable-template.md` 说明当前最小可交付
- 当前修复路径的完成状态，应映射到 `.opencode/references/mes-ai-reference/templates/governance/definition-of-done-template.md` 的 GSD 完成定义
- 紧急修复结束前，应使用 `.opencode/references/mes-ai-reference/templates/governance/completion-sweep-template.md` 执行收尾扫描
- 若需独立输出下一步建议，应使用 `.opencode/references/mes-ai-reference/templates/governance/next-step-recommendation-template.md`

## 多Agent并行编排策略

| Phase | 可并行 | 说明 |
|-------|--------|------|
| Phase 1: 问题登记 | 否 | 必须先确定策略 |
| Phase 2: 执行 | 否 | 由 `mes-backend-developer` / `mes-frontend-developer` 最小面修复，速度优先 |
| Phase 3: 事后补充 | ✅ 部分并行 | `mes-review-auditor` 审核 + 回归测试和文档补充可并行 |

## Agent职责与交接规则

| Phase | 主负责Agent | 说明 |
|------|-------------|------|
| Phase 1 | `mes-sisyphus` | 事件登记、爆炸半径评估、选择回滚/修复策略 |
| Phase 2 | `mes-backend-developer` / `mes-frontend-developer` | 按受影响面最小修复，不扩散修改 |
| Phase 3 | `mes-review-auditor` | 审核事故报告、复盘结果和知识补录质量 |

### 交接文件

每次紧急修复至少应生成以下交接文件：
- `incident-report.md` — 事件报告
- `fix-record.md` — 修复记录与修改清单
- `postmortem.md` — 复盘与预防措施
- `refresh-hints.md` — 知识库刷新提示

若 Phase 2 由后端/前端修复Agent执行，必须先写 `fix-record.md`，主控 Agent 再决定是否进入 Phase 3。

### 失败升级规则

1. 若紧急修复两次尝试仍失败，立即升级至技术负责人，不允许继续盲修
2. 若回滚与修复都不可行，主控 Agent 必须输出阻塞摘要并升级至项目经理
3. 若知识补录未在24小时内完成，状态标记为 `paused`，并持续保留问题记录直至补齐

## 人工审核门禁

| 节点 | 审核内容 | 必须确认的问题 |
|------|---------|--------------|
| Phase 1完成后 | 处理策略 | 回滚还是修复？修复方案是否可行？ |
| Phase 2完成后 | 修复验证 | 冒烟测试是否通过？服务是否恢复？ |
| Phase 3完成后 | 事后补充 | 事故报告是否完整？预防措施是否明确？ |

## 执行计划模板

```markdown
## 紧急修复计划

**事件编号**：EMG-YYYYMMDD-XXX
**严重程度**：P0/P1/P2

**处理策略**：回滚 / 紧急修复 / 临时规避

**步骤**：
1. 登记事件，评估影响范围
2. [回滚：执行回滚方案 / 修复：定位并修复问题]
3. 快速验证（冒烟测试）
4. 快速部署
5. 24小时内补充完整文档

**风险评估**：
- 修复可能影响的关联服务
- 数据一致性风险
```

## 阶段完成产物报告要求

紧急修复阶段退出前，必须基于正式产物清单生成 `stage-output-report.md` 或等价阶段完成产物报告，至少覆盖：

- `incident-report.md`
- `fix-record.md`
- `postmortem.md`
- `refresh-hints.md`
- 紧急修复相关门禁记录 / 快速验证记录

报告中必须明确：
- 每个文件或产物组的作用
- 哪些文件已生成
- 哪些文件未生成
- 未生成是否因“紧急模式下延后补齐 / 条件未命中 / 待人工补录 / 被门禁阻断”等原因
- 未生成是否影响紧急修复完成结论

## 输出目录结构

```
mes-ai-dev/workspace/emergency/EMG-YYYYMMDD-XXX/
├── incident-report.md     # 事件报告（含root cause分析+爆炸半径）
├── postmortem.md          # Postmortem复盘（5Why+预防措施+检测改进）
├── fix-record.md          # 修复记录（含修改清单）
├── refresh-hints.md       # 知识库紧急刷新提示
└── emergency-review-report.md # 阶段详细审查报告（强制产物）
```

## 注意事项

1. **最小修改**：紧急修复只修问题本身，不做优化
2. **速度优先**：可跳过正常流程的部分步骤
3. **事后必补**：24小时内必须补充完整文档和测试
4. **必须记录**：所有紧急修复记录到 status-tracker.md
5. **不可省略验证**：冒烟测试是底线
