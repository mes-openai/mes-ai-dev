# mes-deliver-release

## 功能说明

执行完整的发布交付流程，确保经过测试验证的功能安全发布到生产环境。包括部署规划、验收检查、部署执行、发布说明和交付交接五个阶段。

## 使用方式

```
/mes-deliver-release
```

**适用场景**：
- 测试验证完成后，需要发布到生产环境时
- 需要制定部署计划和回滚方案时
- 需要生成发布说明和交接文档时

**前置条件**：
- 测试验证已完成（执行过 `/mes-test-verify`）
- 测试报告确认通过
- 若当前发布涉及统一响应、错误码、SDK 模型、认证/MQ 契约，应优先确认相关契约变更已完成回归验证
- 若当前发布涉及多仓、多 provider、现有接口复用或空模板阻断问题修复，应优先确认专项回归结果已具备可追溯证据

> **交付前验证说明**：
> 发布交付阶段不重新定义契约、provider 或仓边界，而应基于测试与回归结果判断：
> - 契约相关风险是否已验证闭环
> - 多 provider 路径是否符合冻结设计
> - 既有接口复用是否未引入平行实现
> - 占位态文档误消费问题是否已被阻断

**预期耗时**：约30-60分钟（不含人工审批时间）

**步骤级强制门禁**：每个步骤的产出在进入下一步骤前，必须先执行步骤级门禁审查；未通过时当前步骤打回重做，不得将未通过结果注入下一步骤。

**图谱/TDD Skill 继承说明**：当本命令编排的 Skill 命中 GitNexus / graphify / TDD 单元测试场景时，默认继承 `.opencode/references/mes-ai-reference/rules/governance/skill-graph-tdd-consumption-standard.md`；图谱能力仅作为证据导航或导读，不替代事实证据、阶段结论或门禁判断；TDD/Mockito/路径兼容规则以该统一标准为准。

**阶段工作目录命名说明**：本命令必须遵循 `.opencode/references/mes-ai-reference/rules/governance/stage-workspace-directory-standard.md`。发布交付阶段开始前必须复用上游 `{REQ-ID}`，并重新计算当前输出目录为 `mes-ai-dev/workspace/delivery/{REQ-ID}/`；不得把交付、发布说明或交接文档写入需求、设计、开发或测试目录。

**审查报告强制要求**：本命令中的验收检查、Go/No-Go 决策、交付审计和步骤级门禁结论，必须输出详细审查报告，不得仅保留审批流水、检查清单或口头结论。

**阶段完成产物报告**：
- `/mes-deliver-release` 完成后必须输出一份阶段完成产物报告
- 报告必须说明：发布交付阶段标准产物、已生成文件、各文件作用、未生成文件及未生成原因
- 报告未生成或内容不达标，不得通过发布交付阶段退出门禁

**阶段详细审查报告**：
- `/mes-deliver-release` 完成后必须输出 `delivery-review-report.md`
- 落盘目录：`mes-ai-dev/workspace/delivery/{REQ-ID}/`
- 报告必须满足详细审查报告最小字段集，并包含明确审查时间与完整证据链
- 报告未生成或内容不达标，不得通过发布交付阶段退出门禁

## GSD 执行增强说明

当测试阶段已形成明确的可交付倾向、最小上线包可定义、关键发布前置条件大体满足时，`/mes-deliver-release` 可采用 GSD 模式，优先形成 Go/No-Go 判断与发布后动作清单。

### 一、何时必须维持 Strict

命中以下情况时必须维持 Strict：

- 无法形成 Go/No-Go 倾向
- 回滚策略缺失
- 关键发布前置条件未满足
- 关键风险未明确
- 当前交付结论无法支撑真实发布判断

### 二、最小上线包定义

最小上线包至少包含：

- 当前发布对象与范围
- 当前 Go/No-Go 倾向
- 关键风险说明
- 回滚思路
- 发布后观察项
- refresh / retrospective / 后补治理建议

### 三、风险分层规则

交付阶段必须区分：

- 阻断发布风险
- 可带发布风险
- 发布后观察项

### 四、GSD Continue Exit 条件

交付阶段进入 GSD Continue Exit 至少必须满足：

1. 当前最小上线包已形成
2. Go/No-Go 倾向明确
3. 关键风险已说明
4. 回滚思路明确
5. 发布后动作已输出
6. 已完成收尾扫描

### 五、Completion Sweep

交付阶段结束前至少检查：

- 是否已形成明确结论
- 是否仍有未识别的发布阻断
- 是否已明确回滚与发布后动作
- 是否已给出 refresh / retrospective 建议

### 六、输出要求

本命令的最终输出至少应包含：

- 当前模式
- 当前交付目标
- 最小上线包状态
- Go/No-Go 倾向
- 阻断发布风险
- 可带发布风险
- 发布后动作
- 推荐下一步

### 八、关键交付补充

- 若当前发布依赖本轮骨架整改结果，应优先引用专项回归计划/清单/结果产物，而不是只引用代码测试结论
- 若当前发布涉及业务仓外契约变更，应检查来源类型、版本一致性说明与回归证据是否齐备
- 若当前发布涉及多 provider / 复用主路径，应检查是否仍有“技术可达但架构禁止”的残留实现

### 七、模板绑定说明

- 命中 blocker 或关键发布风险时，应使用 `.opencode/references/mes-ai-reference/templates/governance/blocker-record-template.md` 记录阻断发布风险与后补动作
- 以 GSD Continue Exit 继续时，应使用 `.opencode/references/mes-ai-reference/templates/governance/minimum-deliverable-template.md` 说明最小上线包与当前发布判断
- 交付完成状态，应映射到 `.opencode/references/mes-ai-reference/templates/governance/definition-of-done-template.md` 的 GSD 完成定义
- 发布交付结束前，应使用 `.opencode/references/mes-ai-reference/templates/governance/completion-sweep-template.md` 执行收尾扫描
- 若需独立输出下一步建议，应使用 `.opencode/references/mes-ai-reference/templates/governance/next-step-recommendation-template.md`

## 多Agent并行编排策略

| Phase | 可并行 | 说明 |
|-------|--------|------|
| Phase 1: 部署规划 | 否 | 必须先确认测试结果并制定计划 |
| Phase 2: 验收检查 | 否 | 依赖部署计划中的变更范围 |
| Phase 3: 部署执行 | 否 | 需人工确认，串行执行 |
| Phase 4: 并行产出 | ✅ 两路并行 | 发布说明和交付交接可同时进行 |

Agent分配:
- Phase 1-3: 单 Agent 串行执行
- Phase 4: 两个 Agent 并行

task()调用示例（Phase 4）:

```
task(category="deep", load_skills=["mes-deliver-release-note"], run_in_background=true,
     prompt="生成发布说明。\n输入: workspace/delivery/{REQ-ID}/ 下所有产出\n输出: workspace/delivery/{REQ-ID}/release-note.md")

task(category="deep", load_skills=["mes-deliver-handover"], run_in_background=true,
     prompt="生成交付交接文档。\n输入: workspace/delivery/{REQ-ID}/ + workspace/ 下所有阶段产出\n输出: workspace/delivery/{REQ-ID}/handover-doc.md")
```

结果合并:
- 汇总 `release-note.md` 和 `handover-doc.md`

共享文件与局部成功规则:
- `release-note.md` 与 `handover-doc.md` 可并行生成，但最终交付结论、锁释放和发布状态更新必须由主控Agent串行执行
- 单项交付产出成功时必须保留，不因另一项失败而回滚
- 部署日志、Go/No-Go 决策和交付审计记录只允许主控Agent统一写入

错误处理:
- 部署执行失败立即触发回滚
- 回滚失败立即告警转人工处理
- 单项产出失败不阻塞另一项

## 人工审核门禁

本命令在以下节点**必须等待用户确认**后才能继续。AI不得自动跳过任何门禁。

### 门禁点

| 节点 | 审核内容 | 必须确认的问题 |
|------|---------|--------------|
| Phase 1完成后 | 部署计划 | 部署策略是否合理？涉及服务是否完整？回滚方案是否可行？ |
| Phase 2完成后 | 验收报告 | 验收标准是否全部满足？未通过项是否可接受？ |
| Phase 3执行前 | 部署确认 | ⚠️ 是否确认执行部署？（不可跳过） |
| Phase 3完成后 | 部署日志 | 部署是否成功？冒烟测试是否通过？ |
| Phase 4完成后 | 发布说明+交接文档 | 变更记录是否完整？交接文档是否清晰？ |

### 门禁执行规则

1. 每个门禁点完成后，主控Agent必须**暂停并输出审核摘要**，等待用户明确回复"确认"或"继续"
2. Phase 3 的部署确认是**硬性门禁**，必须用户回复"确认部署"才能执行
3. 用户可以回复"确认"继续，或提出修改意见
4. 用户未确认时，不得进入下一阶段
5. 审核摘要应包含：本阶段产出清单、关键决策点、发现的问题

## 阶段完成产物报告要求

发布交付阶段退出前，必须基于正式产物清单生成 `stage-output-report.md` 或等价阶段完成产物报告，至少覆盖：

- `deploy-plan.md`
- `acceptance-report.md`
- `deploy-log.md`
- `go-nogo.md`
- `release-note.md`
- `handover-doc.md`
- `operations-manual.md`
- `user-guide.md`
- `post-release-checklist.md`
- `delivery-package-audit.md`
- `delivery-audit-record.md`
- `release-retrospective.md`（若本次已进入复盘补录范围）
- 步骤级门禁记录 / 验收审查记录

报告中必须明确：
- 每个文件或产物组的作用
- 哪些文件已生成
- 哪些文件未生成
- 未生成是否因“不适用 / 条件未命中 / 由复盘阶段承接 / 待补证 / 被门禁阻断”等原因
- 未生成是否影响交付完成结论

## 交付包完整性审计

> Phase 4新增步骤：在部署执行后，对交付包进行完整性审计，确保所有必要产物已齐备。

### 审计维度

| 审计维度 | 审计项 | 验证方式 |
|---------|--------|---------|
| 产物清单核对 | 所有阶段产出文件已生成 | 文件存在性检查（对照预期产出清单） |
| 依赖包完整性 | 后端jar/war包可编译可运行 | mvn package 成功 + 包大小合理 |
| 依赖包完整性 | 前端dist包可构建可部署 | npm run build 成功 + dist目录完整 |
| 配置文件检查 | 所有环境配置已准备 | application.yml 各环境配置存在 |
| 配置文件检查 | 配置项无硬编码敏感信息 | grep检查无IP/端口/密钥硬编码 |
| 回滚包验证 | 回滚脚本已准备 | rollback-template.md 存在且完整 |
| 回滚包验证 | 回滚包可执行 | 回滚脚本语法检查 + 手动验证步骤可行 |
| 变更清单核对 | change-list.md 与实际变更一致 | 文件列表比对 |
| API变更核对 | 新增/修改/删除API已记录 | api-registry.md 更新 + 前端API适配 |

### 审计执行步骤

1. **产物清单核对**：使用预期产出清单逐项检查文件存在性
2. **依赖包验证**：执行构建命令验证包完整性
3. **配置文件检查**：检查配置文件完整性和安全性
4. **回滚包验证**：检查回滚脚本完整性和可执行性
5. **变更清单核对**：比对change-list.md与实际变更文件
6. **记录审计结果**：填写delivery-package-audit.md

### 审计结果判定

| 审计结论 | 判定条件 | 后续动作 |
|---------|---------|---------|
| ✅ 完整 | 所有审计项通过 | 可继续发布流程 |
| ⚠️ 部分缺失 | 有非关键项缺失 | 补充缺失项后继续，记录豁免 |
| ❌ 关键缺失 | 回滚包/配置文件等关键项缺失 | 阻止发布，返回补充 |

### 审计产出模板

使用 `templates/delivery-package-audit-template.md` 记录审计结果。

## 注意事项

1. **测试前置**：P1用例必须100%通过才能进入交付阶段
2. **人工确认**：部署执行前必须获得人工确认，不可自动跳过
3. **回滚准备**：部署前必须确认回滚方案可行
4. **并行产出**：发布说明和交接文档可并行生成，提高效率
5. **锁释放**：交接完成后必须释放 workspace/locks/ 下的锁文件
