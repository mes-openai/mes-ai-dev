# mes-test-verify

## 功能说明

执行完整的测试验证流程，包括测试规划、单元测试、集成测试和测试报告生成。确保开发完成的功能经过充分测试验证。

## 使用方式

```
/mes-test-verify
```

**适用场景**：
- 代码开发完成后，需要进行测试验证时
- 需要生成测试用例和测试代码时
- 需要生成测试报告时

**前置条件**：
- 代码开发已完成（执行过 `/mes-develop-code`）
- 后端和前端代码已实现
- 若当前变更涉及统一响应、错误码、SDK 模型、认证/MQ 契约，应优先确认相关契约事实源与回归范围已明确
- 若当前变更涉及多仓、多 provider 或现有接口复用，应优先确认 analyze/design 产物中相关边界与主路径已冻结
- 已存在开发阶段先行生成并经用户补充确认的 `test-cases.md`

> **回归依赖说明**：
> 测试阶段不应重新定义仓边界、provider 或契约结构，而应基于 analyze/design/develop 已形成的结论验证：
> - 契约是否被正确消费
> - provider 是否按冻结路径调用
> - 既有接口复用是否被正确实现
> - 空模板阻断是否未被绕过

> **全仓视角依赖说明**：
> 若本次测试规划依赖全仓级的调用链、热点服务/API/表或全局覆盖率判断，应优先确认已完成 `/mes-init-converge`；否则测试结论只能视为局部范围结论。

**预期耗时**：约20-40分钟

**步骤级强制门禁**：每个步骤的产出在进入下一步骤前，必须先执行步骤级门禁审查；未通过时当前步骤打回重做，不得将未通过结果注入下一步骤。

**审查报告强制要求**：本命令中的测试报告、阶段门禁结论和验收性判断，均必须具备详细审查报告结构，至少保留结论、问题、风险、整改、证据路径和闭环状态。

**阶段完成产物报告**：
- `/mes-test-verify` 完成后必须输出一份阶段完成产物报告
- 报告必须说明：测试验证阶段标准产物、已生成文件、各文件作用、未生成文件及未生成原因
- 报告未生成或内容不达标，不得通过测试验证阶段退出门禁

**TDD 闭环要求**：
- 测试阶段必须先核对 `test-cases.md` 中的 AI 初始计划、用户补充区与确认后的最终计划
- 测试阶段不得绕开已确认的 TDD 范围重新定义主验证范围；若需扩充，只能在保留既有通过用例的前提下补充

**测试与覆盖率强制要求**：
- 本轮新生成测试用例必须全部通过
- 本轮生成/修改并纳入验证范围的代码行覆盖率必须达到 100%
- 覆盖率不足时，只允许补充测试用例，不得删除已经生成且验证通过的测试用例

**阶段详细审查报告**：
- `/mes-test-verify` 完成后必须输出 `test-review-report.md`
- 落盘目录：`mes-ai-dev/workspace/testing/REQ-YYYYMMDD-XXX/`
- 报告必须满足详细审查报告最小字段集，并包含明确审查时间与完整证据链
- 报告未生成或内容不达标，不得通过测试验证阶段退出门禁

## GSD 执行增强说明

当开发阶段已形成可提测结果、核心验证目标明确、最小提测集可定义时，`/mes-test-verify` 可采用 GSD 模式，优先形成“当前是否可继续进入交付判断”的测试结论。

### 一、何时必须维持 Strict

命中以下情况时必须维持 Strict：

- 核心验证目标不明确
- 阻断问题无法识别
- 当前结果无法形成可交付性判断
- 关键测试前置条件未满足且无代偿路径

### 二、最小提测集定义

最小提测集至少应覆盖：

- 当前需求/修复的核心主路径
- 当前变更影响的关键回归点
- 当前最容易导致交付失败的阻断项

### 三、缺陷分类规则

测试阶段必须区分：

- 阻断缺陷：阻止继续进入交付阶段
- 可带风险缺陷：允许继续，但必须留痕
- 后补验证项：当前不阻断，但必须安排后补

### 四、GSD Continue Exit 条件

测试阶段进入 GSD Continue Exit 至少必须满足：

1. 核心验证已执行
2. 阻断缺陷与可带风险缺陷已区分
3. 当前可交付性已有倾向结论
4. 风险与后补动作已明确
5. 推荐下一步已输出
6. 已完成收尾扫描

### 五、Completion Sweep

测试阶段结束前至少检查：

- 是否已经形成明确测试结论
- 是否已经区分阻断/可带风险问题
- 是否已经说明需要回开发还是可进交付
- 是否已经记录后补验证项

### 六、输出要求

本命令的最终输出至少应包含：

- 当前模式
- 当前测试目标
- 最小提测集范围
- blocker 状态
- 阻断缺陷列表
- 可带风险缺陷列表
- 当前是否建议进入交付
- 后补验证动作
- 推荐下一步

### 八、关键验证补充

- 命中业务仓外契约场景时，应显式验证来源类型与契约消费结果是否一致
- 命中多 provider 场景时，应显式验证是否沿架构允许路径调用，而非仅验证“能调通”
- 命中现有接口复用场景时，应显式验证未在下游仓重造平行能力
- 命中空模板场景时，应显式验证占位态文档未被误当作正式规范输入

### 七、模板绑定说明

- 命中 blocker 或关键缺陷时，应使用 `mes-ai-dev/templates/governance/blocker-record-template.md` 记录阻断与可带风险问题
- 以 GSD Continue Exit 继续时，应使用 `mes-ai-dev/templates/governance/minimum-deliverable-template.md` 说明最小提测集与当前可交付倾向
- 测试完成状态，应映射到 `mes-ai-dev/templates/governance/definition-of-done-template.md` 的 GSD 完成定义
- 测试阶段结束前，应使用 `mes-ai-dev/templates/governance/completion-sweep-template.md` 执行收尾扫描
- 若需独立输出下一步建议，应使用 `mes-ai-dev/templates/governance/next-step-recommendation-template.md`

## 多Agent并行编排策略

| Phase | 可并行 | 说明 |
|-------|--------|------|
| Phase 1: 测试规划 | 否 | 必须先生成测试用例 |
| Phase 2: 并行测试 | ✅ 多路并行 | 单元测试、集成测试、性能测试可并行 |
| Phase 3: 测试报告 | 否 | 依赖Phase 2所有结果 |

Agent分配:
- Phase 2: 多个 mes-test-executor Agent并行

task()调用示例:

```
task(category="deep", load_skills=["mes-test-generate-unit"], run_in_background=true,
     prompt="执行单元测试。\n测试用例: workspace/testing/{REQ-ID}/test-cases.md\n输出: workspace/testing/{REQ-ID}/unit-test-results.md")

task(category="deep", load_skills=["mes-test-generate-integration"], run_in_background=true,
     prompt="执行集成测试。\n测试用例: workspace/testing/{REQ-ID}/test-cases.md\n输出: workspace/testing/{REQ-ID}/integration-test-results.md")

task(category="deep", load_skills=["mes-test-performance-analysis"], run_in_background=true,
     prompt="执行性能测试。\n测试用例: workspace/testing/{REQ-ID}/test-cases.md\n输出: workspace/testing/{REQ-ID}/performance-analysis.md")
```

结果合并:
- 汇总所有 `*-results.md` 和 `*-analysis.md`

共享文件与局部成功规则:
- `test-report.md` 只允许主控Agent统一汇总写入
- 单类测试成功结果必须保留，不因另一类测试失败而回滚
- 失败测试只重试对应 mes-test-executor Agent，不重跑全部测试

错误处理:
- P1用例失败必须修复
- 单类测试失败不阻塞其他

## 人工审核门禁

本命令在以下节点**必须等待用户确认**后才能继续。AI不得自动跳过任何门禁。

### 门禁点

| 节点 | 审核内容 | 必须确认的问题 |
|------|---------|--------------|
| Phase 1完成后 | TDD 最终测试范围 | 用户补充是否已吸收？最终计划是否还需调整？ |
| Phase 2完成后 | 测试执行结果 | P1用例是否全部通过？新生成测试是否全绿？缺陷是否需修复？ |
| Phase 4完成后 | 测试报告 | 覆盖率是否 100%？遗留风险是否可接受？ |

### 门禁执行规则

1. 每个门禁点完成后，主控Agent必须**暂停并输出审核摘要**，等待用户明确回复"确认"或"继续"
2. 用户可以回复"确认"继续，或提出修改意见
3. 用户未确认时，不得进入下一阶段
4. 审核摘要应包含：本阶段产出清单、关键决策点、发现的问题

## 阶段完成产物报告要求

测试验证阶段退出前，必须基于正式产物清单生成 `stage-output-report.md` 或等价阶段完成产物报告，至少覆盖：

- `test-cases.md`
- 单元测试代码 / 测试类（若命中单元测试）
- `unit-test-results.md`（若命中）
- `integration-test-results.md`（若命中）
- `performance-analysis.md`（若命中性能测试）
- `uat-scenarios.md`（若命中 UAT）
- `data-regression-report.md`（若命中数据回归验证）
- `smoke-and-gray.md`
- `test-report.md`
- `test-report.md`（作为测试阶段主交接文档）
- 缺陷跟踪 / 步骤级门禁记录

报告中必须明确：
- 每个文件或产物组的作用
- 哪些文件已生成
- 哪些文件未生成
- 未生成是否因“不适用 / 条件未命中 / 由后续交付阶段承接 / 待补证 / 被门禁阻断”等原因
- 未生成是否影响进入交付阶段

## 注意事项

1. **并行执行**：单元测试和集成测试可并行生成
2. **优先级覆盖**：确保P1用例100%通过
3. **缺陷记录**：发现的缺陷需记录在报告中
4. **测试报告**：测试完成后必须生成测试报告
5. **通过标准**：P1全部通过、P2通过率≥95%方可进入下一阶段
6. **TDD 对齐**：测试范围必须先对齐开发阶段已确认的 `test-cases.md`
7. **覆盖率闭环**：本轮新增/修改代码覆盖率不足 100% 时，只能补充测试用例
