# 执行步骤（2/2）

### 4A. 术语冲突检测（强制）

在解析需求要素后，**必须**与术语表进行交叉比对，识别术语冲突：

1. 读取 `mes-ai-dev/knowledge/reference/terminology-glossary.md`
2. 提取需求中使用的所有业务术语
3. 逐条比对：
   - 需求中的术语是否在术语表中存在？
   - 同一概念是否使用了不同的名称？（如：需求中用"批次号"，术语表中定义的是"Lot ID"）
   - 不同概念是否使用了相同名称？（如：需求中"工单"可能指 WorkOrder 或 ProductionOrder）
4. 输出术语冲突清单至 `raw-requirement.md` 的"术语冲突"章节：

| 需求中使用的术语 | 术语表中的标准术语 | 冲突类型 | 建议处理方式 |
|----------------|-----------------|---------|-------------|
| [需求中的叫法] | [术语表标准叫法] | 同义多名字/一词多义/未定义 | [统一为术语表标准/需澄清/需新增到术语表] |

**冲突类型说明**：
- **同义多名字**：同一概念有多个叫法（如"报工"/"工单回报"/"产出上报"）→ 统一为术语表标准名
- **一词多义**：同一词汇在不同上下文含义不同 → 需要澄清具体含义
- **未定义**：需求中使用了术语表中不存在的术语 → 需确认是否需要新增术语

**处理规则**：
- 所有"一词多义"冲突必须澄清后才能进入设计
- "未定义"术语必须补充到术语表或在需求中明确解释
- 缺失的业务规则
- 矛盾的需求描述
- 技术可行性存疑点
- 需要业务确认的边界场景

**Step Gate B**：需求要素提取不完整、模糊点识别不足或术语冲突未记录 → 打回步骤3-4重做，不得进入需求类型判断。

### 5. 判断需求类型
检查 `mes-ai-dev/workspace/requirements/` 下是否存在历史需求：
- **新需求**：全新功能，无历史需求记录
- **迭代需求**：对已有功能的增强/修改，需关联历史需求编号

检查方法：
1. 扫描 requirements/ 目录下所有需求文档
2. 对比功能描述关键词
3. 查找相似的业务模块或功能点
4. 生成需求关联图（使用 `templates/requirement-lineage-template.md` 模板），记录：
   - 本次需求复用了哪些历史方案
    - 哪些已有需求被影响
    - 是否属于二次迭代/回滚修复/补丁修补

**Step Gate C**：新需求/迭代需求判断依据不足或关联历史需求未校准 → 打回步骤5重做，不得生成 raw-requirement 文档。

### 6. 生成交付文件
输出文件：`mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/raw-requirement.md`

若命中 OpenSpec 条件产物触发条件，还必须同步生成：
- `mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/exploration.md`（命中探索触发条件时）
- `mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/proposal.md`（命中方案取舍或用户确认触发条件时）

`exploration.md` 与 `proposal.md` 不得替代 `spec.md`，其结论必须在后续规格生成阶段被吸收到 `spec.md`。

**Step Gate D**：raw-requirement 文档缺少关键章节、保留占位内容或不可被下游消费 → 打回步骤6重做，不得交付影响分析阶段。
