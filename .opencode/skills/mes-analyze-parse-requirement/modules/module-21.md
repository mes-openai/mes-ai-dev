# exploration/proposal 条件产物

## 一、定位

`exploration.md` 与 `proposal.md` 是 analyze 阶段的 OpenSpec 条件产物：

- `exploration.md`：用于探索、澄清、比较、排查边界，不承载最终规格承诺。
- `proposal.md`：用于给出推荐方案、取舍理由和待用户确认的关键决策。
- `spec.md`：最终主交接文档，必须吸收已确认的探索和提案结论。

## 二、触发条件

### 2.1 必须生成 `exploration.md`

命中任一条件时必须生成：

- 需求存在多个合理解释，直接写规格会引入猜测。
- 存在两个及以上可行实现/落仓/复用路径，需要比较边界与风险。
- 涉及多仓、跨服务、跨模块，且责任边界尚不清晰。
- 存量项目结构复杂，需要先判断目标仓结构、分层、命名和测试组织适配风险。

### 2.2 必须生成 `proposal.md`

命中任一条件时必须生成：

- 需要在多个候选方案中给出推荐方案。
- 需要用户确认方案取舍、仓落点、provider 或复用路径。
- 存在会影响 design 阶段的高风险前置决策。

## 三、最小结构与收口要求

### 3.1 `exploration.md` 最小结构
- 探索背景
- 待澄清问题
- 候选解释 / 候选路径
- 存量结构与边界证据
- 风险与不确定性
- 建议进入 proposal/spec 的结论

### 3.2 `proposal.md` 最小结构
- 候选方案摘要
- 推荐方案
- 取舍理由
- 用户待确认决策
- 对 `spec.md` 的落点要求

### 3.3 收口规则
- `exploration.md` 的结论必须进入 `proposal.md` 或 `spec.md`。
- `proposal.md` 的推荐方案和确认结论必须进入 `spec.md`。
- 不得将未确认的探索假设写成 `spec.md` 中的既定事实。

**Step Gate E**：命中触发条件但未生成条件产物，或条件产物未被 `spec.md` 吸收 → 打回条件产物生成与规格收口步骤。
