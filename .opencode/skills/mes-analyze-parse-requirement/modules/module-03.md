# 执行步骤（1/2）

## 执行步骤

### 1. 执行前输出计划（必须）
在执行任何操作前，先输出本次执行计划：
```
【执行计划】
目标：将原始需求解析为结构化需求要素
步骤：
  1. 创建需求目录结构
  2. 读取原始需求输入
  3. 解析需求要素（背景、功能、规则、角色、流程、非功能）
  4. 识别模糊点和待确认事项
  5. 判断新需求/迭代需求
  6. 判断是否命中 exploration.md / proposal.md 条件产物
  7. 生成原始需求文档与条件 OpenSpec 产物
预期产出：raw-requirement.md；命中条件时生成 exploration.md / proposal.md
风险：需求描述过于模糊导致解析偏差
```

### 2. 创建需求目录
生成需求编号 `REQ-YYYYMMDD-XXX`（日期+当日序号），创建目录：
```
mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/
```

**Step Gate A**：需求目录未正确创建或需求编号不合规 → 打回步骤2重做，不得进入需求要素解析。

### 3. 解析需求要素
从原始需求描述中提取以下要素：

#### 3.1 业务背景
- 需求来源（业务部门/客户/合规要求等）
- 业务痛点或机会
- 期望达成的业务目标
- 业务价值评估

#### 3.2 功能描述
- 核心功能点（逐条列出）
- 用户操作流程（从用户视角描述）
- 输入/输出说明
- 业务场景说明

#### 3.3 业务规则
- 数据校验规则
- 业务计算逻辑
- 权限控制规则
- 异常处理规则
- 数据完整性约束

#### 3.4 用户角色
- 角色定义（谁使用这个功能）
- 权限要求（不同角色的操作权限）
- 用户数量级（并发用户数估算）

#### 3.5 业务流程
- 主流程（正常场景）
- 分支流程（异常场景、边界场景）
- 与现有流程的集成点

#### 3.6 非功能需求
- 性能要求（响应时间、吞吐量）
- 安全要求（数据保护、访问控制）
- 可用性要求（系统稳定性）
- 兼容性要求（浏览器、设备）
- 可维护性要求

### 4. 识别模糊点
明确标注需求中的模糊、不清晰之处：

若模糊点、候选路径、责任边界不清或结构适配风险满足 `module-21.md` 的触发条件，必须进入 exploration/proposal 条件产物判断，不得直接把未冻结判断写入 `spec.md`。
