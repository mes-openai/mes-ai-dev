# 执行步骤（2/2）

### 5. 风险评估

针对选定的技术方案：
- **技术风险**：新技术引入、性能瓶颈、兼容性问题
- **业务风险**：对现有功能的影响、数据一致性风险
- **实施风险**：工作量估算、依赖关系、团队技能
- **应对策略**：为每个风险制定缓解措施

#### 5.0 数据库变更风险评估（新增）
若技术方案涉及数据库变更，必须读取Schema风险画像：

读取文件：`mes-ai-dev/knowledge/database-index/schema-xxx/index.md`（按需）

提取信息：
- **Schema风险等级**：低/中/高，影响变更审批流程设计
- **热表清单**：高频访问表，设计方案需评估性能影响
- **共享表清单**：多服务访问表，设计方案需协调机制
- **大表清单**：数据量>100万行，设计方案需分批执行策略
- **回滚敏感表清单**：设计方案需回滚策略

用途：
- 在风险评估中标注数据库变更风险等级
- 为技术方案制定数据库变更审批要求

#### 5.01 测试可测性评估（新增）
若技术方案涉及服务开发，必须读取测试可测性矩阵：

读取文件：`mes-ai-dev/knowledge/code-map/testability-matrix.md`

提取信息：
- **服务测试可测性等级**：高/中/低，判断服务是否易于测试
- **可测性评分维度**：数据可测性、接口可测性、状态可测性、依赖可测性
- **测试难点标注**：哪些功能点难以测试，需特殊测试策略
- **建议测试类型**：单元测试、集成测试、性能测试的建议比例

用途：
- 在技术方案中标注服务测试可测性
- 为测试阶段提供测试策略建议
- 影响开发任务规划中的测试任务分配

### 5A. 兼容性设计（存量变更必做）

若技术方案涉及对存量代码/接口/表结构的修改，必须进行兼容性设计：
1. 使用 `templates/compatibility-design-template.md` 模板
2. 评估向后兼容需求（老接口保留、字段默认值、废弃标记）
3. 制定灰度发布方案（灰度维度、比例、观察指标、回滚条件）
4. 设计双写/双读过渡方案（如涉及数据迁移）
5. 制定历史数据迁移策略（迁移范围、脚本、校验、回滚）
6. 产出写入：`mes-ai-dev/workspace/designs/REQ-YYYYMMDD-XXX/compatibility-design.md`

### 5B. 跨服务一致性校验

若需求涉及多个微服务，必须进行跨服务一致性校验：
1. 使用 `templates/cross-service-consistency-template.md` 模板
2. 调用 `mes-design-check-consistency` Skill 执行校验
3. 检查维度：字段命名、错误码、状态枚举、接口语义、数据格式
4. 产出写入：`mes-ai-dev/workspace/designs/REQ-YYYYMMDD-XXX/cross-service-consistency.md`

### 5C. ADR强制触发判断

以下5种场景**必须**生成ADR（架构决策记录），使用 `templates/adr-template.md`：
1. **跨服务改动** — 变更涉及2个及以上微服务的协作方式
2. **引入新表** — 新增数据库表（含索引策略、分区策略决策）
3. **接口兼容变更** — 修改已有API的入参/出参/语义
4. **权限模型变更** — 新增/修改角色、数据权限、操作权限
5. **性能方案取舍** — 涉及缓存/异步/分库分表等性能方案的取舍

产出写入：`mes-ai-dev/workspace/designs/REQ-YYYYMMDD-XXX/adr-{序号}-{标题}.md`

### 5D. 安全设计点输出

在技术方案中**必须**输出以下安全设计点（不是安全审查，是设计阶段的安全考量）：

| 安全设计点 | 必须回答的问题 |
|-----------|--------------|
| 权限变更 | 本次变更是否新增/修改了权限？哪些角色受影响？ |
| 数据脱敏 | 涉及的敏感字段是否需要脱敏？脱敏规则是什么？ |
| 审计字段 | 哪些操作需要记录审计日志？审计字段是否完整？ |
| 风险操作确认 | 哪些高风险操作（删除/批量操作/审批）需要二次确认？ |
| 越权路径 | 是否存在水平越权（访问他人数据）或垂直越权（提权操作）风险？ |

产出写入：`tech-approach.md` 的安全设计章节。

**Step Gate C**：方案对比、风险评估、安全设计点或兼容性/一致性要求未闭环 → 打回步骤4-5重做，不得进入技术决策记录。
