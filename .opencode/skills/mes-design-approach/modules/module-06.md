# 执行步骤（1/2）

## 执行步骤

### 1. 输出执行计划

在开始设计前，必须先输出本次执行计划：
- **目标**：明确本次技术方案设计的目标范围
- **步骤**：列出将要执行的详细步骤
- **预期产出**：明确将生成的文档和关键决策点
- **风险评估**：识别设计过程中可能遇到的问题

### 2. 读取前置输入

按以下顺序读取必要的输入文件：

1. **需求规格文档**
   - 路径：`mes-ai-dev/workspace/requirements/{REQ-ID}/spec.md`
   - 重点提取：功能需求、非功能需求、约束条件

2. **原始需求文档**
   - 路径：`mes-ai-dev/workspace/requirements/{REQ-ID}/raw-requirement.md`
   - 用途：理解业务背景和上下文

3. **第0层索引（系统总览）**
   - 后端总览：`mes-ai-dev/knowledge/code-map/backend-overview.md`
   - 前端总览：`mes-ai-dev/knowledge/code-map/frontend-overview.md`
   - 用途：识别需求涉及的微服务和模块

**Step Gate A**：前置输入不完整或需求/知识库基线读取不足 → 打回步骤2重做，不得进入涉及服务分析。

### 3. 涉及服务分析

基于需求规格和第0层索引：
- 列出涉及的所有微服务
- 标注每个服务的职责和边界
- 识别服务间的潜在协作关系
- 确定是否需要新增服务或模块

#### 3.1 服务架构约束读取（新增）
读取文件：`mes-ai-dev/knowledge/code-map/backend-overview.md` 的「仓级架构职责」章节

提取信息：
- **分层架构定义**：确保技术方案遵循现有分层约束（Controller→Service→DAO→Model）
- **服务职责边界**：确保技术方案不越界调用其他服务职责
- **技术栈栈位**：确保技术选型符合现有技术栈
- **架构约束与规范**：确保技术方案遵守架构约束（不硬编码、统一响应格式等）

用途：
- 在技术方案设计时明确架构约束
- 防止设计方案违反现有架构规范

#### 3.2 服务仓库画像读取（新增）
读取文件：`mes-ai-dev/knowledge/code-map/services/service-xxx/repo-profile.md`（按需）

提取信息：
- **一致性评分**：判断服务代码规范程度，影响设计复杂度预估
- **风险热力图**：识别服务内部高风险区域，设计方案需规避或标注
- **演化状态**：判断服务是否适合激进改动或需保守设计
- **AI开发适配度**：判断是否适合AI自动实现，需人工辅助区域
- **不建议AI修改区域**：设计方案需标注人工必须介入的区域

用途：
- 在技术方案中标注架构约束遵守情况
- 为风险评估提供服务内部风险热点

**Step Gate B**：涉及服务边界不清、仓库画像未校准或架构约束未识别 → 打回步骤3重做，不得进入技术方案制定。

### 4. 技术方案制定

#### 4.1 方案设计原则
- 符合现有架构规范（Jalor框架 + Vue前端）
- 优先复用现有组件和服务
- 考虑性能、可扩展性、可维护性
- 遵循最小改动原则

#### 4.2 多方案对比

为每个关键设计点提出至少2个方案：
- **方案A**：推荐方案（需说明推荐理由）
- **方案B**：备选方案
- 对比维度：实现复杂度、性能影响、风险程度、工作量评估
