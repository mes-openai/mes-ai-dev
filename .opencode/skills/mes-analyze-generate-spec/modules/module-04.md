# 执行步骤（2/2）

### 5. 按模板组织文档
按照 `requirement-spec-template.md` 的章节组织内容，重点整合：

#### 5.1 文档基本信息
- 需求编号
- 需求名称
- 需求类型（新需求/迭代需求）
- 文档版本
- 创建日期
- 创建人

#### 5.1.5 exploration/proposal 结论收口
- 哪些探索结论已被采纳
- 哪些候选方案被淘汰以及原因
- 哪些推荐方案已由用户确认
- 哪些仍是后补动作，不能作为本轮规格承诺

#### 5.2 需求概述
整合业务背景：
- 需求来源
- 业务痛点
- 业务目标
- 业务价值

#### 5.3 功能需求
整合功能描述：
- 核心功能点列表
- 用户操作流程
- 输入 / 输出说明
- 业务场景说明

整合业务规则：
- 数据校验规则
- 业务计算逻辑
- 权限控制规则
- 异常处理规则

#### 5.4 数据需求
整合数据库信息：
- 数据表列表
- 字段定义
- 表关联关系
- 数据流向

#### 5.5 接口需求
整合 API 信息：
- 前端 API 调用列表
- 后端接口定义
- 请求参数定义
- 响应数据定义

#### 5.6 非功能需求
整合非功能需求：
- 性能要求
- 安全要求
- 可用性要求
- 兼容性要求
- 可维护性要求

#### 5.7 影响分析
整合影响范围：
- 后端服务影响
- 前端模块影响
- 数据库影响
- 配置影响
- 风险评估

#### 5.8 验收标准
定义验收标准：
- 功能验收标准
- 性能验收标准
- 安全验收标准
- 用户验收标准

#### 5.9 存量功能变更说明（迭代需求）
若为迭代需求，增加此章节：
- 存量功能描述
- 变更内容说明
- 变更影响评估
- 存量功能兼容性

**Step Gate C**：规格内容组织不完整、章节遗漏或验收/NFR信息未整合 → 打回步骤5重做，不得生成交付文件。

### 6. 生成交付文件

生成以下文件：
- `mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/spec.md` — OpenSpec 需求规格主文档
- `mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/nfr-checklist.md` — NFR 非功能需求清单（强制）

NFR 清单覆盖 8 个维度：
1. 性能需求（响应时间 / 并发 / 吞吐量 / 批处理 / 数据量级）
2. 安全需求（脱敏 / 访问控制 / 加密 / 注入防护 / XSS）
3. 审计需求（操作日志 / 数据变更 / 登录审计 / 关键操作签名）
4. 数据保留需求（保留期限 / 归档策略 / 数据清理）
5. 权限需求（角色定义 / 数据权限 / 越权防护）
6. 并发需求（并发控制 / 幂等性 / 资源竞争）
7. 灰度需求（灰度维度 / 灰度比例 / 回滚条件）
8. 回滚需求（回滚策略 / 回滚耗时 / 数据回滚）

**Step Gate D**：`spec.md` 或 `nfr-checklist.md` 缺少真实内容、保留占位文本或不可被下游消费 → 打回步骤6重做，不得交付设计阶段。
