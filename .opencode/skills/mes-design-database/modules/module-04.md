# 执行步骤（1/2）

## 执行步骤

### 1. 输出执行计划

在开始设计前，必须先输出本次执行计划：
- **目标**：明确本次数据库设计的范围和目标
- **步骤**：列出将要执行的详细步骤
- **预期产出**：明确将生成的设计文档和DDL脚本
- **风险评估**：识别可能的数据安全和性能风险

### 2. 读取前置输入

按以下顺序读取必要的输入文件：

1. **需求规格文档**
   - 路径：`mes-ai-dev/workspace/requirements/REQ-YYYYMMDD-XXX/spec.md`
   - 重点提取：数据相关需求、数据实体、数据关系

2. **技术方案文档**
   - 路径：`mes-ai-dev/workspace/designs/REQ-YYYYMMDD-XXX/tech-approach.md`
   - 重点提取：涉及的服务列表、数据存储决策

3. **数据库注册表**
   - 路径：`mes-ai-dev/knowledge/dependency-graph/database-registry.md`
   - 用途：了解现有数据库Schema分配和表结构概览

4. **数据库索引**
     - 路径：`mes-ai-dev/knowledge/database-index/schema-xxx/`
     - 根据涉及的服务读取对应的Schema索引文件

5. **真实数据库对象证据**
   - 路径：目标 schema 对应 DDL、现有 Entity、现有 Mapper XML / SQL
   - 用途：核对表是否真实存在、字段是否真实存在、是否属于现有表变更而不是臆造新表

**Step Gate A**：前置输入不完整、Schema范围不清、数据库基线读取不足或真实数据库对象证据未完成采集 → 打回步骤2重做，不得进入现有表结构分析。

### 3. 现有表结构分析

根据数据库注册表和索引：
- 列出涉及的现有数据表
- 分析表结构、字段定义、约束关系
- 识别现有的索引配置
- 定表之间的关联关系

#### 3.0 表真实性核验（新增）

对本次设计涉及的每一张表，必须先形成真实性核验记录：

| 表名 | 类型 | 事实来源 | 证据路径 | 是否允许进入设计 |
|------|------|----------|----------|------------------|
| `t_xxx` | 现有表 / 新增表 / 修改表 | DDL / database-index / Entity / 已批准新增 | [文件路径] | 是/否 |

核验要求：
- **现有表**：必须至少能在 DDL、`database-index`、Entity、现有 SQL 之一中找到证据。
- **修改表**：除现有表证据外，还必须说明本次修改依据。
- **新增表**：必须明确新增依据，不能只有“需求里提到业务对象”。
- **证据路径**：必须填写到具体文件，不能只写“数据库”或“知识库”。

若任一表缺少真实性证据，或新增表缺少批准依据，当前步骤必须判定为不通过。

#### 3.1 Schema风险画像读取（新增）
读取文件：`mes-ai-dev/knowledge/database-index/schema-xxx/index.md`

提取信息：
- **Schema风险等级**：低/中/高，决定变更审批要求
- **热表清单**：高频访问表，变更需评估性能影响，建议低峰时段执行
- **共享表清单**：多服务访问表，变更需在设计中标注协调机制
- **大表清单**：数据量>100万行，变更需分批执行策略
- **触发器/存储过程清单**：有触发器时需评估数据一致性影响
- **回滚敏感表清单**：设计需包含完整回滚策略

用途：
- 在数据库设计文档中标注Schema风险等级
- 确定变更审批要求（低风险→开发自审，中风险→技术负责人审批，高风险→DBA审批）
- 设计变更执行策略（热表→低峰时段，大表→分批执行）

#### 3.2 变更审批要求判定（新增）
根据Schema风险等级确定审批要求：

| Schema风险等级 | 审批要求 | 执行要求 |
|---------------|---------|---------|
| 🟢 低风险 | 开发自审通过即可 | 可在业务时段执行 |
| 🟡 中风险 | 技术负责人审批 | 建议在低峰时段执行，需回滚脚本 |
| 🔴 高风险 | 技术负责人 + DBA审批 | 必须在维护窗口执行，需完整回滚+验证方案 |

**Step Gate B**：表真实性核验、现有表结构、Schema风险画像或审批要求判定不完整 → 打回步骤3重做，不得进入数据模型设计。
