# 执行步骤（2/2）

### 4. 数据模型设计

#### 4.1 实体关系建模
- 识别业务实体及其属性
- 定义实体间的关系（1:1, 1:N, N:M）
- 绘制ER图（可选，使用文本描述）

#### 4.2 表结构设计

为每个需要变更的表制定详细设计：

前置要求：
- 先引用步骤3的表真实性核验结果。
- 每张表都必须标注“现有 / 新增 / 修改”类型。
- 每张表都必须标注事实来源和证据路径。

**新增表设计**
- 表名（遵循命名规范：`t_`前缀）
- 字段定义（类型、长度、默认值、约束）
- 主键设计（自增ID or 业务主键）
- 外键约束（如有）
- 索引设计

**修改表设计**
- 新增字段及其定义
- 修改字段（类型变更、长度调整等）
- 删除字段（需标注是否允许数据丢失）
- 索引调整

禁止事项：
- 不得把不存在于真实数据库对象中的表写成“现有表”。
- 不得把未批准的新表写进设计文档。
- 不得只凭需求语言推测字段名和字段类型。

### 5. 索引设计

针对查询需求设计索引：
- **主键索引**：自动创建
- **唯一索引**：业务唯一约束字段
- **普通索引**：高频查询字段、外键字段
- **组合索引**：多字段联合查询（考虑最左前缀原则）

索引设计说明：
- 索引名称规范：`idx_表名_字段名`
- 组合索引字段顺序：区分度高的字段在前
- 评估索引对写入性能的影响

### 6. 数据迁移设计

如果涉及数据迁移：
- **迁移策略**：停机迁移 vs 在线迁移
- **迁移脚本**：数据转换逻辑
- **回滚方案**：迁移失败时的回滚步骤
- **数据校验**：迁移后的数据一致性验证

**Step Gate C**：表真实性、表结构、索引、迁移或性能影响设计未闭环，或存在无证据表/字段 → 打回步骤4-7重做，不得输出数据库设计文档。

### 7. 性能影响评估

评估数据库变更对系统性能的影响：
- 查询性能影响（预期提升或下降）
- 写入性能影响
- 存储空间预估
- 并发性能影响
