# 约束规则

## 约束规则

### 执行前必须输出计划
每次执行前必须输出执行计划，经用户确认后才可继续。

### 必须读取现有风格
在编写任何 API 文件前，必须：
1. 读取目标模块现有的 API 文件（至少 3 个）
2. 分析请求封装、函数命名、参数传递、返回值类型
3. 保持与现有代码风格一致

### API 路径一致性
必须严格按照 API 设计文档定义：
- URL 路径：与后端 Controller 路径一致
- HTTP 方法：POST（创建）、PUT（更新）、DELETE（删除）、GET（查询）
- 参数传递：POST/PUT 使用 data，GET 使用 params
- 资源 ID：使用 URL 路径参数 `/api/xxx/${id}`

### 函数命名规范
- 创建：`createXxx`
- 批量创建：`batchCreateXxx`
- 更新：`updateXxx`
- 删除：`deleteXxx`
- 查询详情：`getXxxById`
- 查询列表：`getXxxList`
- 分页查询：`getXxxPage`
- 校验：`checkXxxExists` 或 `validateXxx`

### 注释规范
- 文件必须有文件头注释（说明、作者、日期）
- 每个函数必须有注释（功能、参数、返回值）
- 使用 JSDoc 标准格式

### 响应处理规范
- 响应必须解构为 Result<T> 格式（如项目使用该包装）
- 成功响应：按现有请求封装规则处理
- 失败响应：由请求拦截器统一处理
- 异常响应：由请求拦截器统一处理

### 不硬编码 API 路径
API 路径使用常量或从配置获取，不硬编码具体路径（除了请求对象中的 url）。
