# 执行步骤（2/4）

### 4. 调用链设计

#### 4.1 调用链路识别

识别业务流程涉及的服务调用链：

**主流程调用链**
```
前端 → 服务A → 服务B → 服务C
```

**分支流程调用链**
```
服务A → 服务B (成功) → 服务C
      → 服务B (失败) → 服务D (降级)
```

#### 4.2 调用方式设计

为每个服务调用确定调用方式：

**同步调用（REST/RPC）**
- 适用场景：实时性要求高、需要立即返回结果
- 调用方式：HTTP REST、Dubbo RPC、Feign
- 超时设置：根据业务特性设置合理的超时时间
- 重试策略：是否重试、重试次数、重试间隔

**异步调用（消息队列）**
- 适用场景：耗时操作、解耦、削峰填谷
- 消息中间件：RabbitMQ、Kafka、RocketMQ
- 消息格式：消息头、消息体定义
- 消息可靠性：确认机制、持久化策略

使用文本描述服务调用链路：

**Step Gate B**：服务调用关系分析或调用链设计不完整 → 打回步骤3-4重做，不得进入数据流转与事务边界设计。

### 5. 数据流转设计

#### 5.1 数据流转路径

定义数据在各服务间的流转路径：

```
服务A (原始数据)
  ↓ 处理并传递
服务B (中间数据)
  ↓ 处理并传递
服务C (最终数据)
```

#### 5.2 数据传递方式

**同步传递**
- 通过API参数传递
- 通过HTTP Header传递上下文信息
- 通过请求体传递业务数据

**异步传递**
- 通过消息体传递业务数据
- 通过消息属性传递元数据
- 消息格式标准化

#### 5.3 数据一致性保证

**强一致性场景**
- 使用分布式事务（如Seata）
- 事务协调者设计
- 事务补偿机制

**最终一致性场景**
- 使用消息队列
- 消息确认机制
- 消息重试和死信处理
- 幂等性设计
