# 执行步骤

## 执行步骤

### Step 0: 存量仓模式检测（新增）

在开始提取前，检测是否为存量仓（已有人工开发的大型代码仓）：

1. 统计代码行数，判断仓规模标签（参见 `knowledge/rules/repository-scale-rules.md`）
2. 若仓规模为 `medium` 及以上，启用**存量仓模式**：
   - **优先真实边界**：从代码中提取实际的职责划分、API归属、表归属，优先于理想化分类
   - **延后抽象术语**：先提取具体实体名、API路径、表名、配置项名，再归纳为抽象术语体系
   - **标注置信度**：所有推断的知识必须标注置信度（高/中/低）和推断依据
   - **记录真实偏差**：当实际结构与理想规范不一致时，在输出中标注偏差说明
   - **参考实现优先**：找到最规范、最完整的模块标记为参考实现

### Step 1: 提取业务术语
- 扫描 Entity 类名、字段注释、枚举注释。
- 汇总术语中文名、英文名、定义、关联实体和来源服务。
- 无法确认定义时标注来源依据与置信度。
- **存量仓模式**：先收集代码中出现的所有中文注释和业务名词，建立"原始术语清单"，再归纳为统一术语表。标注实际使用的业务用语与标准术语的偏差。

输出要求：
- 必须写入 `mes-ai-dev/knowledge/reference/terminology-glossary.md`
- 不得只保留占位标题、说明文字或空章节
- 至少包含可消费的术语条目、来源依据和必要置信度标记

**Step Gate A**：术语、领域模型或基础参考信息提取不完整 → 打回步骤1-3重做，不得进入枚举/错误码/API/编码规范提取。

### Step 2: 构建领域模型
- 分析 Entity 间关联关系，包括 JPA 注解、外键引用和代码中的对象依赖。
- 识别核心实体、从属实体和聚合边界。
- 用文本方式输出实体关系，并标注推断依据。
- **存量仓模式**：优先从代码中提取实际的外键关系和对象依赖，而非理想化的聚合边界。当实际依赖关系与理想领域模型不一致时，标注偏差。

#### Step 2.1: 提取业务规则（领域模型扩展）

在构建领域模型的同时，从代码中提取业务规则：

1. **校验规则**：扫描Service实现类中的 if 条件判断、参数验证逻辑（@Valid、手动校验）
2. **计算规则**：识别Service中的计算方法（包含数学运算、百分比、比率计算的方法）
3. **流程规则**：分析状态字段（status/state）的赋值逻辑，提取状态流转规则
4. **数据约束**：从Entity注解（@NotNull、@Size、@Pattern）和数据库DDL（UNIQUE、CHECK、FK）提取

输出：追加到 `domain-model.md` 的业务规则章节

### Step 3: 生成数据字典
- 提取 Entity 字段定义、注释、类型和约束信息。
- 汇总公共字段含义与各业务域字段定义。
- 归纳字段命名规范，未检测到的模式标注“未检测到”。

### Step 4: 提取枚举注册表
- 扫描所有 enum 类，提取枚举值、含义、使用场景和来源位置。
- 同步识别状态码、类型码和标志位常量。
- 按服务分组组织，缺失部分标注“未检测到”。

### Step 5: 提取错误码
- 扫描错误码常量类和异常类。
- 提取错误码编号、HTTP 状态、描述、触发条件和来源位置。
- 若编码规则不统一，记录现状并标注置信度。

### Step 6: 推断API约定
- 分析现有 Controller 的 URL 模式、HTTP 方法、参数命名与响应包装。
- 汇总分页、排序、过滤、认证授权和版本管理规则。
- 对未出现的模式明确标注“未检测到”。

### Step 6.5: 解析业务仓外契约源（新增）
- 当统一响应、错误码、认证、MQ 或 SDK 请求/响应对象定义在业务仓之外时，必须先解析外部契约源，再生成全局规范。
- 按以下顺序解析事实源：
  1. `workspace-source`：当前 workspace 中的 `sdk/`、`common/`、`shared/`、`integration/`、`framework-extension/` 等源码目录
  2. `repo-source`：已纳入当前工作区范围的公共源码仓
  3. `sources-jar`：与当前依赖版本对应的源码附件
  4. `decompiled-jar`：仅当 1~3 都不可得时，作为受控兜底路径
  5. `unknown`：若仍不可得，则标记未知或候选结论，不得补洞
- 对每条契约知识必须记录：来源类型、来源定位、获取方式、版本一致性说明、置信度。
- 若未确认外部契约源，不得输出统一响应、统一错误码、统一认证或统一 MQ 契约全局结论。

### Step 7: 推断编码规范
- 分析现有代码的包结构、命名、注解模式、日志与异常处理方式。
- 同步观察配置管理方式与前后端共性风格。
- 输出规则说明、示例、来源依据和置信度。

### Step 8: 生成权限矩阵（新增）
- 扫描Controller层的权限注解（@PreAuthorize、@Secured等）。
- 分析角色定义（从用户角色相关代码或配置推断）。
- 提取命令与文件的访问权限关系（从业务逻辑推断）。
- 汇总为角色×命令×文件权限矩阵。

输出文件：`mes-ai-dev/knowledge/reference/permission-matrix.md`
