# 约束规则

## 约束规则

### 执行前必须输出计划
每次执行前必须输出执行计划，经用户确认后才可继续。

### 必须读取现有风格
在编写任何Service前，必须：
1. 读取目标服务现有的Service接口和实现类（至少各2个）
2. 分析命名规范、注解使用、异常处理、日志使用
3. 保持与现有代码风格一致

### 必须完成真实性校验
在写入任何 Service 代码前，必须：
1. 确认 import 对应类真实存在
2. 确认包声明与目录路径一致
3. 确认调用的方法真实存在于 DTO / VO / Entity / Mapper / 工具类 / 客户端中
4. 确认返回值类型、泛型和包装层与真实定义一致
5. 确认异常类、分页对象、响应对象来自项目真实公共能力，而不是模板默认值

### 命名规范
- Service接口：`XxxService`（功能名+Service）
- Service实现：`XxxServiceImpl`（功能名+ServiceImpl）
- 方法名：动词开头，如`create`、`update`、`delete`、`getById`

### 业务逻辑规范
- 所有写操作必须加`@Transactional`
- 所有方法必须有日志记录（入参、出参、异常）
- 参数校验放在方法开头
- 业务校验在参数校验之后
- 异常使用统一的BusinessException

### 跨服务调用规范
- 使用框架提供的RestServiceClient
- 服务名从配置获取，不硬编码
- 调用失败必须记录日志并抛出异常
- 调用超时必须合理设置
- 调用前必须核对 provider 真实方法签名、返回包装和异常语义
- 接收返回值时必须按真实契约解包，不得凭接口名称猜测 data 结构

### 不硬编码服务名
跨服务调用时，服务名必须从配置文件或常量获取，不硬编码具体服务名。
