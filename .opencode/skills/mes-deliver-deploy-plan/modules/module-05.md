# 执行步骤

## 执行步骤

### 1. 输出执行计划

在执行前，必须输出本次计划：

```
本次执行计划：
目标：制定完整的部署计划
步骤：
  1. 读取测试报告，确认测试通过状态
  2. 读取设计文档，梳理变更范围
  3. 读取服务依赖图，确定部署顺序
  4. 制定部署策略
  5. 规划环境配置
  6. 制定回滚方案
  7. 生成部署计划文档
预期产出：deploy-plan.md
风险：服务依赖遗漏、环境配置差异未覆盖
```

### 2. 确认测试状态

1. 读取测试报告，确认测试通过状态
2. 检查P1用例是否100%通过
3. 检查P2用例通过率是否≥95%
4. 如有未通过用例，输出警告并建议暂停部署

**Step Gate A**：测试状态确认不完整或存在未解释未通过项 → 打回步骤2重做，不得进入变更范围梳理。

### 3. 梳理变更范围

1. 读取设计文档，提取变更清单：
   - 新增/修改的API端点
   - 新增/修改的数据库表/字段
   - 新增/修改的配置项
   - 新增/修改的前端页面/组件
2. 读取需求规格，确认变更的业务背景
3. 读取知识库中相关服务的 `service-xxx/index.md`，确认涉及的服务范围

### 4. 确定部署顺序

1. 读取服务依赖图，提取本次涉及服务的调用关系
2. 分析依赖方向，确定部署顺序：
   - 被依赖的服务先部署（如DAO层变更先于Service层）
   - 数据库变更先于应用代码
   - 后端先于前端（如有接口变更）
3. 标注可并行部署的服务组

**Step Gate B**：变更范围、依赖关系或部署顺序分析存在缺口 → 打回步骤3-4重做，不得进入部署策略与环境规划。

### 5. 制定部署策略

根据变更范围选择部署策略：

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 蓝绿部署 | 大版本更新 | 零停机，秒级切换 | 需要双倍资源 |
| 滚动部署 | 小规模更新 | 资源占用低 | 切换期间存在版本不一致窗口 |
| 灰度发布 | 需要验证的场景 | 风险可控 | 需要流量路由支持 |

### 6. 规划环境配置

梳理各环境（开发→测试→预发布→生产）的配置差异：

1. 数据库连接配置
2. 服务注册地址
3. 外部服务端点
4. 功能开关
5. 日志级别

### 7. 制定回滚方案

1. 定义回滚触发条件：
   - 部署后健康检查失败
   - 冒烟测试不通过
   - 关键业务指标异常
2. 制定回滚步骤（逆部署顺序）
3. 确定回滚验证方式
4. 估算回滚耗时

**Step Gate C**：部署策略、环境配置或回滚方案未闭环 → 打回步骤5-7重做，不得生成部署计划文档。

### 8. 生成部署计划

1. 读取部署计划模板
2. 按模板格式填充所有内容
3. 写入 `workspace/delivery/{REQ-ID}/deploy-plan.md`

**Step Gate D**：`deploy-plan.md` 缺少部署顺序、环境差异、验证步骤或回滚方案 → 打回步骤8重做，不得交付部署执行阶段。
