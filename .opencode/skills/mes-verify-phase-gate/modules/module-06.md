# 校验规则（2/3）

### 表真实性专项执行逻辑（新增）

```
当当前阶段或步骤涉及 database-design / schema / DDL 设计时：
  1. 检查设计中每张表是否标注“现有表 / 修改表 / 新增表”
  2. 检查每张表是否标注事实来源与证据路径
  3. 若表被标记为现有表/修改表：
     - 检查是否能在 DDL / database-index / database-registry / Entity / 现有SQL 中找到证据
  4. 若表被标记为新增表：
     - 检查是否有明确新增批准依据
  5. 若任一表无法证明真实性或批准性 → must-pass 失败
```

表真实性专项校验中的 must-pass 失败示例：
- 把需求中的业务对象直接写成真实表
- 表名在真实数据库对象中找不到证据
- 新增表没有批准依据
- 设计文档未记录事实来源或证据路径

### Java 引用可解析性专项执行逻辑（新增）

```
当当前阶段或步骤涉及后端 Java 代码生成时：
  1. 检查包声明是否与目录路径一致
  2. 检查 import 对应类是否真实存在
  3. 检查调用的方法是否真实存在于目标对象中
  4. 检查接收返回值的类型、泛型和包装层是否与真实定义一致
  5. 任一项不满足 → must-pass 失败
```

### MyBatis 映射一致性专项执行逻辑（新增）

```
当当前阶段或步骤涉及 Mapper 接口 / Mapper XML 生成时：
  1. 检查 XML `namespace` 是否等于 Mapper 接口全限定名
  2. 检查 XML `id` 是否与接口方法一一对应
  3. 检查 `parameterType` / `resultType` / `resultMap` 是否真实可解析
  4. 检查表名、字段名是否存在真实数据库依据
  5. 任一项不满足 → must-pass 失败
```

### Provider 契约一致性专项执行逻辑（新增）

```
当当前阶段或步骤涉及内部接口调用或跨服务调用时：
  1. 检查是否已读取 provider 真实接口定义、client 封装或 API 注册信息
  2. 检查参数类型、返回类型、泛型、包装层是否一致
  3. 检查调用方解包方式是否与 provider 真实返回结构一致
  4. 任一项不满足 → must-pass 失败
```

### 权限校验执行逻辑（新增）

```
权限校验在阶段进入时执行：

1. 读取 permission-matrix.md
2. 确定当前用户角色（从对话上下文或默认角色推断）
3. 确定当前要执行的命令（从调用上下文获取）
4. 检查：
   - 用户角色是否在命令所需角色列表中
   - 用户角色是否对输出文件有写权限
   - 用户角色是否对输入文件有读权限
5. 标记 ✅ 或 ❌

权限不通过时输出：
❌ 权限校验阻止：[用户角色] 无权限执行 [命令]
- 需要角色：[角色列表]
- 当前角色：[用户角色]
- 请联系管理员获取相应权限。
```

**Step Gate D**：命令/文件权限校验未完成或角色判断不明确 → 打回权限校验步骤重做，不得放行阶段或步骤。
