# 执行步骤

## 执行步骤

### 1. 确定当前阶段

从调用上下文确定：
- 当前要进入/退出哪个阶段
- 需求编号是什么

### 2. 加载门禁标准

优先读取 `phase-gates/index.md` 确认阶段门禁入口，再读取 `phase-gates/common.md` 与 `phase-gates/<当前阶段>.md`，提取对应阶段的进入/退出条件，并按 must-pass / should-check / advisory 分层解释，同时服从 `phase-gate-usage-standard.md` 对结论口径、失败处理与落盘要求的统一约束。`phase-gates.md` 仅作为兼容导航入口，不应作为默认主读取对象。

### 3. 逐项执行检查

按分层门禁条目执行自动化检查。涉及人工确认的条目，标记为"待人工确认"。

若命中数据库索引相关条目，必须附加执行“数据库索引专项执行逻辑”，并在报告中单列数据库校验结果，不得只在通用 must-pass 中笼统给出“已检查”。

若命中数据库设计、后端代码生成、MyBatis 生成或内部接口调用相关条目，必须附加执行对应专项逻辑，并在报告中单列真实性 / 一致性校验结果，不得仅写“已核对”。

若命中阶段退出场景，必须附加检查阶段完成产物报告：
- 是否存在正式报告文件
- 是否覆盖本阶段标准产物清单
- 是否说明已生成文件及作用
- 是否说明未生成文件及未生成原因
- 是否说明未生成是否允许、是否影响下一阶段

同时必须附加检查本阶段详细审查报告：
- 是否存在正式详细审查报告文件
- 是否记录明确审查时间
- 是否能从检查项回溯到证据路径
- 是否存在缺少证据支撑的猜测性结论

任一项不满足，阶段退出门禁不得通过。

若命中 AI 代码生成相关阶段，还必须附加检查：
- `test-cases.md` 是否先于代码生成形成
- 是否包含用户补充区与用户确认结论
- 本轮新生成测试用例是否全部通过
- 本轮生成/修改并纳入验证范围的行覆盖率、分支覆盖率、方法覆盖率是否均达到 100%
- 是否存在通过删测、降级或跳过已通过用例换取覆盖率达标的行为
- 当前阶段 OpenSpec 主交接文档是否已生成且命名正确

### 4. 汇总结果

输出校验报告。如有 must-pass ❌ 项，**停止当前 Command 或当前步骤执行**；其余项作为提示附带输出。

### 4.5 门禁留痕规则

- 以下场景必须落盘门禁审查记录，而不是仅输出到对话：
  1. 门禁结论为 `❌不通过`
  2. 门禁结论为 `⚠️有条件通过`
  3. 任一跨阶段出口门禁
  4. 初始化收拢、发布执行、知识刷新、状态迁移校验相关门禁
- 落盘路径与文件命名：遵循 `.opencode/references/mes-ai-reference/reference/status-tracker-governance.md`
- 步骤级 gate 若需落盘，推荐命名：`step-gate-{阶段名}-step-{步骤编号}.md`
- 未落盘的强制留痕门禁结果，视为门禁未完成，不得下游消费
- 落盘的门禁记录不得仅保留“通过/不通过”单行结论，必须包含：审查目标与范围、输入与输出、检查项与结果、问题/风险、整改要求、结论、证据路径、复审/闭环状态
- 若门禁记录主文件超限，可拆分附录，但主文件必须保留关键结论、问题、风险、整改要求和证据路径

### 5. 更新状态

校验通过后，更新 `status-tracker.md`。

> `step-gate` 审查记录字段与 `status-tracker.md` 的映射关系，遵循 `status-tracker-governance.md`。

**Step Gate E**：校验报告、阻断原因或状态更新留痕不完整 → 打回汇总/状态更新步骤重做，不得将校验结果交付下游消费。
