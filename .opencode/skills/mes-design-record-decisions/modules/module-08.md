# 执行步骤

## 执行步骤

### 1. 识别决策点

读取设计文档，识别需要决策的技术点：
- 技术选型（A vs B）
- 架构模式（集中式 vs 分布式）
- 兼容性策略（兼容 vs 不兼容）
- 数据策略（实时 vs 异步）

### 2. 评估备选方案

对每个决策点，至少评估 2 个备选方案：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案A | ... | ... |
| 方案B | ... | ... |

**Step Gate A**：决策点识别或备选方案评估不完整 → 打回步骤1-2重做，不得进入 ADR 记录。

### 3. 记录决策

按 ADR 模板格式记录：
- 背景（为什么需要决策）
- 决策（选择了什么）
- 备选方案（放弃了什么，为什么）
- 影响分析（性能/安全/兼容性/运维）
- 约束与风险

### 4. 检查跨服务一致性

检查本次决策是否与其他服务的现有决策冲突：
1. 读取 `dependency-graph/service-dependencies.md`
2. 检查本次决策涉及的API变更是否影响下游服务
3. 如有冲突 → 在 ADR 中标注"需协调"

### 5. 兼容性设计

对每个涉及API或数据库变更的决策：
1. 标注是否向下兼容
2. 如不兼容 → 必须设计过渡方案（分阶段实施）
3. 过渡方案写入 ADR 的"约束与风险"章节

**Step Gate B**：ADR 记录、跨服务冲突检查或兼容性设计存在缺口 → 打回步骤3-5重做，不得交付架构决策记录。
