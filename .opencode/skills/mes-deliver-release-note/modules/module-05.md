# 执行步骤

## 执行步骤

### 1. 输出执行计划

```
本次执行计划：
目标：生成完整的发布说明
步骤：
  1. 读取所有输入文档
  2. 提取功能变更清单
  3. 提取API变更清单
  4. 提取数据库变更清单
  5. 整理已知问题和限制
  6. 确定版本号
  7. 生成发布说明
预期产出：release-note.md
风险：变更项遗漏、版本号不正确
```

### 2. 收集变更信息

从各阶段文档中提取变更信息：

| 信息来源 | 提取内容 |
|---------|---------|
| 需求规格 | 功能需求描述、业务背景 |
| 设计文档 | 技术方案、API设计、数据库设计 |
| 测试报告 | 测试覆盖情况、已知缺陷 |
| 部署日志 | 实际部署的服务和步骤 |
| 验收报告 | 验收结论、遗留问题 |

**Step Gate A**：变更信息收集不完整或来源依据不足 → 打回步骤2重做，不得进入分类整理。

### 3. 分类变更清单

按变更类型分类整理：

#### 功能变更
| 变更类型 | 描述 |
|---------|------|
| 新增 | 原来不存在的功能 |
| 修改 | 已有功能的优化或调整 |
| 废弃 | 不再支持的功能 |
| 修复 | 修复的缺陷 |

#### API变更
| 变更类型 | 端点 | 说明 |
|---------|------|------|
| 新增API | POST /xxx | 新增接口 |
| 修改API | PUT /xxx | 参数变更 |
| 废弃API | DELETE /xxx | 已废弃 |

#### 数据库变更
| 变更类型 | 对象 | 说明 |
|---------|------|------|
| 新增表 | table_name | 新增表结构 |
| 修改表 | table_name | 新增/修改字段 |
| 数据迁移 | table_name | 数据转换脚本 |

### 4. 整理已知问题

1. 从测试报告中提取已知缺陷
2. 从验收报告中提取遗留问题
3. 从代码自审报告中提取技术债务
4. 按严重程度分级：严重/一般/建议

### 5. 确定版本号

遵循语义化版本规范（SemVer）：

| 版本类型 | 格式 | 触发条件 |
|---------|------|---------|
| 主版本 | X.0.0 | 有不兼容的API变更 |
| 次版本 | X.Y.0 | 新增功能，向下兼容 |
| 修订版 | X.Y.Z | 缺陷修复，无功能变更 |

**Step Gate B**：变更分类、已知问题整理或版本号判断不准确 → 打回步骤3-5重做，不得进入升级指南与发布说明生成。

### 6. 编写升级指南

如有需要注意的升级事项：

1. 数据库迁移步骤
2. 配置项变更说明
3. 不兼容变更的适配方式
4. 依赖升级要求

### 7. 生成发布说明

1. 读取发布说明模板
2. 按模板格式填充所有内容
3. 写入 `workspace/delivery/REQ-YYYYMMDD-XXX/release-note.md`

**Step Gate C**：`release-note.md` 缺少变更清单、已知问题、版本号依据或升级指南 → 打回步骤6-7重做，不得交付交接阶段。
