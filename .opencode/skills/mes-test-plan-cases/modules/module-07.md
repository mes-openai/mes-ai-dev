# 执行步骤

## 执行步骤

### 步骤1：加载输入文档

1. 读取需求规格文档，提取功能需求点
2. 读取设计文档，提取技术实现细节
3. 读取任务计划，了解开发范围
4. 读取目标仓既有测试资产，提取测试组织方式、命名约定、夹具模式和断言风格
5. 读取测试用例模板，明确格式规范

#### 1.1 风险热点读取（新增）
读取服务仓库画像和Schema风险画像，确定重点测试区域：

**服务仓库画像**：`mes-ai-dev/knowledge/code-map/services/service-xxx/repo-profile.md`
提取信息：
- **风险热力图**：识别服务内部高风险区域，需重点测试
- **不建议AI修改区域**：人工完成的功能，需人工测试验证

**Schema风险画像**：`mes-ai-dev/knowledge/database-index/schema-xxx/index.md`
提取信息：
- **热表清单**：高频访问表，需重点性能测试
- **共享表清单**：多服务访问表，需跨服务数据一致性测试
- **大表清单**：数据量>100万行，需分页/性能边界测试
- **回滚敏感表清单**：需回滚测试用例

用途：
- 在测试用例中标注风险热点区域
- 为P1用例确定重点测试目标

#### 1.2 测试可测性矩阵读取（新增）
读取文件：`mes-ai-dev/knowledge/code-map/testability-matrix.md`

提取信息：
- **服务测试可测性等级**：高/中/低，确定自动化测试可行性
- **可测性评分维度**：
  - 数据可测性：测试数据是否容易准备（影响数据驱动测试设计）
  - 接口可测性：接口是否容易Mock（影响集成测试设计）
  - 状态可测性：状态是否容易验证（影响状态流转测试设计）
  - 依赖可测性：外部依赖是否容易隔离（影响单元测试设计）
- **测试难点标注**：哪些功能点需要手工测试或特殊测试策略
- **建议测试类型比例**：单元测试/集成测试的建议比例

用途：
- 在测试用例中标注自动化可行性（高可测性→自动化，低可测性→手工）
- 为测试类型选择提供依据
- 标注需特殊测试策略的功能点

**Step Gate A**：输入文档、风险热点或可测性矩阵读取不完整 → 打回步骤1重做，不得进入测试规划。

### 步骤1.5：形成 TDD 计划草案并等待用户补充

1. 先输出 AI 初始测试计划，不得直接进入代码生成
2. 在 `test-cases.md` 中显式预留“用户补充区”和“确认结论”
3. 若用户补充了场景、边界、历史缺陷、兼容约束或回归范围，必须并入最终计划
4. 未收到用户确认前，当前 Skill 的结论只能是“测试计划待确认”，不得宣告可进入代码生成

**Step Gate B**：未形成用户补充区、未记录用户确认结论或未贴合存量项目既有测试组织方式 → 打回步骤1.5重做，不得进入后续代码生成。
