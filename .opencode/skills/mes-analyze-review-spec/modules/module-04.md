# 执行步骤（2/2）

### 4. 五维审核

#### 4.1 完整性审核
检查需求规格文档是否包含所有必要章节：
- [ ] 文档基本信息
- [ ] 需求概述
- [ ] 功能需求
- [ ] 数据需求
- [ ] 接口需求
- [ ] 非功能需求
- [ ] 影响分析
- [ ] 验收标准
- [ ] 存量功能变更说明（迭代需求必须）

同时检查各章节内容是否完整：
- 功能需求：功能列表、详细说明、输入/输出、业务场景
- 数据需求：数据表、字段定义、表关联关系、数据流向
- 接口需求：前端 API、后端接口、请求参数、响应数据
- 验收标准：功能 / 性能 / 安全 / 用户验收标准

#### 4.2 一致性审核
检查文档内部及跨文档逻辑是否一致：
- 功能与数据一致性
- 功能与接口一致性
- 业务规则与功能一致性
- 主流程与分支流程一致性
- 与 `raw-requirement.md` 的一致性
- 与 `business-flow-trace.md` 的一致性

#### 4.3 可测性审核
检查验收标准是否可测试：
- [ ] 验收标准是否具体明确
- [ ] 是否有明确的测试方法
- [ ] 是否有可量化的验收指标
- [ ] 是否覆盖所有功能点

重点检查：
- 功能可测性
- 性能可测性
- 安全可测性

#### 4.4 可行性审核
检查技术和业务实现是否可行：
- 后端可行性
- 前端可行性
- 数据库可行性
- 业务规则可行性
- 权限控制可行性
- 性能可行性

#### 4.5 关联性审核
通过知识库验证：
- 与现有系统关联是否准确
- 影响服务 / 模块 / 数据表是否真实存在
- 迭代需求下，存量功能描述与变更影响是否准确

**Step Gate C**：五维审核任一维度未完成、问题分级不清或结论不可执行 → 打回步骤4重做，不得进入评审报告生成。

### 5. 使用评审模板生成交付文件
生成文件：
- `mes-ai-dev/workspace/requirements/{REQ-ID}/spec-review-report.md`

统一使用：
- `.opencode/references/mes-ai-reference/templates/analyze/spec-review-report-template.md`
- `.opencode/references/mes-ai-reference/templates/governance/detailed-review-report-template.md`（落盘模板）
- `.opencode/references/mes-ai-reference/rules/governance/review-report-standard.md`（统一最小字段集约束）

> **模板使用要求**：
> - 模板只提供结构化评审报告骨架，不可保留占位文本
> - 必须将五维审核结果、问题分级、评审结论和知识库验证依据替换为真实内容
> - 若评审不通过，必须明确高优先级问题与修正建议
> - 评审结果必须满足详细审查报告最小字段集：审查目标与范围、审查对象与输入、审查依据、检查项与结果、问题清单、风险评估、整改要求/建议、审查结论、证据路径、复审/闭环状态
> - 如主报告超出行数限制，必须采用“主报告 + 附录/证据文件引用”方式，不得退化为简略结论

**Step Gate D**：评审报告缺少真实结论/问题分级/修正建议/证据路径/闭环状态中的任一关键项 → 打回步骤5重做，不得交付下游使用。
