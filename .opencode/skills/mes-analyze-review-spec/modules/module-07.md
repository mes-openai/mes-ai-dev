# 约束规则

## 约束规则

### 必须遵守
1. **执行前输出计划**：在任何操作前必须输出执行计划
2. **结果审核**：评审报告输出前必须审核结论的合理性
3. **五维度审核**：必须完成完整性、一致性、可测性、可行性、关联性五个维度的审核
4. **知识库验证**：必须读取知识库验证对现有系统的描述准确性
5. **上下文预算控制**：知识库读取总量控制在 10K token 以内
6. **文件交接**：通过已有文件获取输入，不依赖对话历史
7. **详细审查报告强制**：输出必须符合详细审查报告统一标准，不得仅保留摘要式结论

### 禁止行为
1. ❌ 不跳过任何审核维度
2. ❌ 不凭主观判断，必须基于知识库验证
3. ❌ 不忽略高优先级问题
4. ❌ 不省略问题汇总和建议
5. ❌ 不给出模糊的评审结论

### 知识库读取策略
```
策略1：按需读取涉及的验证
  影响的服务列表 → 读取 backend-overview.md 验证存在性
  影响的模块列表 → 读取 frontend-overview.md 验证存在性

策略2：只读取必要的验证项
  服务存在性 → backend-overview.md（第0层）
  服务职责准确性 → service-xxx/index.md（第1层精简版，按需）
  表存在性 → database-registry.md

策略3：控制上下文预算
  优先读取第0层（总览）进行快速验证
  发现问题时再读取 detail.md（第1层完整版）进行深度验证
```
