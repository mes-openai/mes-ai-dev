# 执行步骤

## 执行步骤

### 1. 评估当前阶段需要的读取量

```
本次上下文预算评估：
阶段：[阶段名]
需求涉及服务数：X个
计划读取：
  - 第0层：2 个文件（~3K token）
  - 第1层精简版：X 个 index.md（~X×2K token）
  - 第1层完整版：Y 个 detail.md（~Y×8K token）
  - 第1.5层：Y 个 file-summaries.md（~Y×1K token）
  - 第2层：Z 个源文件（~Z×3K token）
  - 历史文档：W 个（~W×5K token）
预估总消耗：~XXXK token
预算状态：✅ 在预算内 / ⚠️ 接近上限 / ❌ 超限
```

**Step Gate A**：预算评估输入不完整或读取量估算不准确 → 打回步骤1重做，不得进入超限处理策略。

### 2. 超限处理策略

当预估消耗超过硬性上限时，按以下优先级缩减：

| 优先级 | 缩减策略 | 适用场景 |
|--------|---------|---------|
| 1 | 减少第1层精简版读取数量 | 涉及服务>8个，只读直接相关 |
| 2 | 减少第1层完整版读取数量 | detail.md 读取>3个，分批处理 |
| 3 | 减少第2层源码读取数量 | 源文件>15个，分批用 workspace 交接 |
| 4 | 将历史文档转为摘要 | workspace/ 文档>5个，只读摘要 |
| 5 | 拆分为多个子任务 | 单阶段无法完成，拆分后分批执行 |

### 3. 文档压缩规则

当需要将文档转为摘要时：

| 文档类型 | 压缩方式 | 压缩后目标大小 |
|---------|---------|---------------|
| spec.md | 提取功能点列表 + 验收标准 | ≤1K token |
| design.md | 提取API设计 + 数据模型 | ≤2K token |
| tasks.md | 提取任务列表 | ≤1K token |
| test-report.md | 提取通过率 + 未通过项 | ≤500 token |
| self-review-report.md | 提取问题清单 | ≤500 token |

### 4. 输出预算确认

在每个阶段开始时输出预算确认：

```
📊 上下文预算确认

| 预算项 | 计划消耗 | 上限 | 状态 |
|--------|---------|------|------|
| 第0层 | 3K | 5K | ✅ |
| 第1层精简版 | 8K | 15K | ✅ |
| 第1层完整版 | 16K | 24K | ✅ |
| 第2层源码 | 30K | 40K | ✅ |
| 对话+产出 | 50K | 100K | ✅ |
| **合计** | **107K** | **190K** | **✅** |

执行策略：直接执行 / 需要分批 / 需要压缩
```

**Step Gate B**：超限处理、阻断记录或预算确认缺少执行策略和恢复路径 → 打回步骤2-4重做，不得交付阶段执行前使用。
