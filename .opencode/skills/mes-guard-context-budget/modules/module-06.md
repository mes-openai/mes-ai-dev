# 硬阻断机制（强制执行）

## 硬阻断机制（强制执行）

> 以下4个条件任一成立时，**必须暂停执行**，输出阻断原因和处理建议，等待用户确认后方可继续。

### 阻断级别定义

| 级别 | 阻断条件 | 阻断动作 | 恢复方式 |
|------|---------|---------|---------|
| 🔴 硬阻断1 | 预估总消耗 > 200K token | 拒绝执行，必须拆分需求或缩减范围 | 用户确认拆分方案后重新评估 |
| 🔴 硬阻断2 | 第2层源码读取文件数 > 15 个 | 暂停执行，强制分批 | 分批方案确认后按批执行，每批≤15文件 |
| 🟡 软阻断1 | 单服务 detail.md 估算 > 10K token | 暂停执行，强制裁剪 | 只读取相关包路径章节，跳过无关章节 |
| 🟡 软阻断2 | 同时展开 > 3 个服务 detail.md | 暂停执行，强制分批 | 分批读取，每批≤3个服务 |

### 阻断执行流程

```
1. 评估预算
2. 检查4个阻断条件
3. 任一命中 → 输出：
   ╔═══════════════════════════════════════╗
   ║ ⛔ 上下文预算阻断                      ║
   ╠═══════════════════════════════════════╣
   ║ 阻断级别：🔴硬阻断 / 🟡软阻断         ║
   ║ 命中条件：[具体条件]                   ║
   ║ 当前状态：[预估消耗 vs 上限]           ║
   ╠═══════════════════════════════════════╣
   ║ 处理建议：                             ║
   ║   1. [建议1]                           ║
   ║   2. [建议2]                           ║
   ╠═══════════════════════════════════════╣
   ║ 请确认处理方案后继续                    ║
   ╚═══════════════════════════════════════╝
4. 用户确认处理方案 → 按方案调整后重新评估
5. 全部条件通过 → 输出预算确认 → 继续执行
```

### 阻断记录

每次阻断发生时，在 workspace 下记录阻断事件：

```
写入文件：workspace/{阶段目录}/budget-block-log.md
内容：
- 阻断时间
- 阻断级别
- 命中条件
- 处理方案
- 用户确认结果
```
