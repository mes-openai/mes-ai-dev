# 执行步骤

## 执行步骤

### 1. 输出执行计划

```
本次执行计划：
目标：逐项验证交付物是否满足验收标准
步骤：
  1. 读取需求规格，提取所有验收标准
  2. 读取测试报告，检查功能覆盖
  3. 读取代码自审报告，确认代码质量
  4. 逐项对照验收标准
  5. 检查非功能需求
  6. 汇总验收结论
  7. 生成验收报告
预期产出：acceptance-report.md（详细审查报告）
风险：验收标准遗漏、非功能需求未覆盖
```

### 2. 提取验收标准

1. 读取需求规格文档
2. 提取"验收标准"章节中的所有条目
3. 提取非功能需求（性能、安全、兼容性等）
4. 为每条验收标准编号，形成验收检查清单

**Step Gate A**：验收标准提取不完整、编号不清或非功能需求遗漏 → 打回步骤2重做，不得进入测试结果对照。

### 3. 对照测试结果

1. 读取测试报告，提取测试用例与功能点的对应关系
2. 检查每条验收标准是否有对应测试用例覆盖
3. 检查测试用例的执行结果：
   - P1用例是否全部通过
   - P2用例通过率是否达标
4. 标记未被测试覆盖的验收标准

### 4. 对照代码质量

1. 读取代码自审报告
2. 检查是否存在未解决的严重问题
3. 检查代码规范是否符合 `rules/coding-standards.md`
4. 确认编译是否通过

**Step Gate B**：测试覆盖、代码质量或非功能需求检查存在缺口 → 打回步骤3-5重做，不得进入逐项评定。

### 5. 检查非功能需求

逐项检查非功能需求是否满足：

| 非功能需求 | 检查来源 | 检查方式 |
|-----------|---------|---------|
| 性能要求 | 测试报告-性能分析 | 响应时间/吞吐量是否达标 |
| 安全要求 | 代码自审-安全审查 | 是否有安全漏洞 |
| 兼容性要求 | 测试报告 | 是否覆盖目标环境 |
| 可维护性 | 代码自审 | 代码质量评分 |

### 6. 逐项评定

对每条验收标准进行评定：

| 评定结果 | 含义 | 条件 |
|---------|------|------|
| ✅ 通过 | 完全满足 | 测试通过 + 代码审查通过 |
| ⚠️ 部分通过 | 基本满足但有小问题 | 核心功能正常 + 非关键问题待修复 |
| ❌ 未通过 | 不满足 | 关键功能异常或测试失败 |

### 7. 汇总验收结论

根据逐项评定结果，给出总体结论：

- **通过**：所有验收标准均为"通过"
- **有条件通过**：存在"部分通过"项，但无"未通过"项
- **不通过**：存在"未通过"项

**Step Gate C**：逐项评定与总体验收结论不一致或修复建议缺失 → 打回步骤6-7重做，不得生成验收报告。

### 8. 生成验收报告

1. 读取验收报告模板
2. 按模板格式填充所有内容
3. 写入 `workspace/delivery/{REQ-ID}/acceptance-report.md`

验收报告必须补齐：
- 审查目标与范围
- 审查依据
- 逐项检查结果
- 问题清单
- 风险评估
- 整改要求与闭环计划
- 证据路径
- 复审状态

**Step Gate D**：acceptance-report 缺少逐项结论、未通过原因、总体结论依据、风险评估、证据路径或闭环状态 → 打回步骤8重做，不得交付发布阶段使用。
