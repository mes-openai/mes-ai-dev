# 执行步骤

## 执行步骤

### 1. 输出执行计划

```
本次执行计划：
目标：评估存量测试资产的真实可用性
步骤：
  1. 读取 test-assets.md 获取测试索引
  2. 逐服务检查测试文件的可执行性
  3. 评估测试质量（是否测试了真正的业务逻辑）
  4. 识别无回归入口的服务
  5. 生成 testability-matrix.md
预期产出：mes-ai-dev/knowledge/code-map/testability-matrix.md
风险：静态分析无法确定测试是否真的能通过运行
```

### 2. 读取前置输入

- `mes-ai-dev/knowledge/code-map/test-assets.md` — 测试索引
- `mes-ai-dev/knowledge/code-map/backend-overview.md` — 服务列表
- 各服务 src/test/ 目录

**Step Gate A**：测试索引、服务列表或测试目录读取不完整 → 打回步骤2重做，不得进入逐服务评估。

### 3. 逐服务评估

对每个服务执行以下评估：

#### 3.1 存在性检查

| 检查项 | 判断方式 |
|--------|---------|
| src/test/ 目录是否存在 | 目录存在性 |
| 测试文件数量 | 文件计数 |
| 测试类是否有 @Test 注解 | 静态分析 |

#### 3.2 质量评估

| 质量维度 | 评估方式 | 分级 |
|---------|---------|------|
| 业务逻辑覆盖 | 测试方法名是否对应业务操作 | 高/中/低 |
| Mock合理度 | Mock是否过多导致测试无意义 | 合理/过度Mock/无Mock |
| 数据依赖 | 测试是否依赖特定数据才能通过 | 自包含/需准备数据/硬依赖 |
| 断言完整度 | 是否有真正的断言（非空断言） | 有业务断言/仅有非空/无断言 |

#### 3.3 可用性分级

| 等级 | 标记 | 定义 | 判断标准 |
|------|------|------|---------|
| 可执行 | `runnable` | 测试自包含，可独立运行 | 有数据准备+有业务断言+无外部依赖 |
| 需维护 | `needs-work` | 测试存在但需修复才能运行 | 有断言但数据依赖缺失或环境依赖 |
| 摆设 | `decorative` | 测试存在但无实际验证能力 | 无断言或仅有非空断言 |
| 无测试 | `none` | 无测试文件 | src/test/ 为空或不存在 |
