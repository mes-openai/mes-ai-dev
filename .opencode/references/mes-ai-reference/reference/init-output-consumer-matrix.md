# 初始化产物下游消费映射

## 一、用途
- 明确“初始化阶段什么叫已经足够读懂存量代码仓到可下游消费”。
- 明确初始化产物如何被 analyze / design / develop / test / deliver / refresh 消费。

## 二、初始化最小知识面

以下知识面属于初始化阶段应优先建立的最小结构化事实集合：

| 知识面 | 最低要求 | 允许状态 |
|---|---|---|
| 仓/服务/模块边界 | 已识别主仓、主服务、主模块及其职责边界 | 确认 / 候选 |
| 关键调用链与热点入口 | 已识别高频 API、热点服务、关键业务链入口 | 确认 / 候选 |
| 数据库 ownership | 已识别关键 Schema、关键表及其主责归属 | 确认 / 候选 |
| 契约级知识 | 已识别统一响应、错误码、认证、MQ、SDK 模型等定义源 | 确认 / 候选 / 未知 |
| 公共依赖与配置 | 已识别 SDK/common/shared/integration、关键配置与调用映射 | 确认 / 候选 |
| 测试资产可用性 | 已识别是否存在可复用测试资产、回归链路或测试缺口 | 确认 / 候选 |

说明：
- “确认”表示可直接作为下游阶段事实输入。
- “候选”表示已有局部证据，但必须带低置信度与待补证说明。
- “未知”只允许出现在契约级、外部依赖或当前 scope 无法闭合的对象上，不得伪装成已知事实。

## 四、下游阶段消费要求

### 4.1 analyze
- 若缺少仓/服务边界、关键调用链或契约定义源，默认不得输出全局责任边界结论。

### 4.2 design
- 若缺少服务依赖、私有契约或关键表归属，默认不得冻结服务链、Schema 边界或接口契约。

### 4.3 develop
- 若缺少私有契约、真实依赖或实现落点事实，必须回流 analyze/design 或降级为局部实现结论。

### 4.4 test
- 若缺少调用链、测试资产或关键表归属，必须显式标注未覆盖边界，不得默认已完成全量验证。

### 4.5 deliver
- 若缺少交付对象边界、契约口径或关键风险基线，不得直接给出全局可发布结论。

### 4.6 refresh
- refresh 只能更新已存在的初始化知识基线；若发现初始化知识面本身缺失，应记录待补初始化，而不是直接伪造全局知识。

## 五、阻断与降级原则
- 需要全局统一结论时，关键初始化知识必须达到“确认”或至少“候选且风险明确”。
- 仅具备局部产物时，只允许输出局部结论或降级推进结论。
- 若契约级知识、仓级边界或关键 ownership 为“未知”，不得继续给出全局 analyze/design/deliver 结论。
