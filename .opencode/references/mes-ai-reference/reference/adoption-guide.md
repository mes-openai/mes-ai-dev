# MES-AI-DEV 落地与试点指南

> 本文档用于承接 README 中较长的 adoption 内容，帮助团队从“理解骨架”进一步走向“开始试点”和“稳定引入”。
> 本文档用于团队引入与试点规划，不替代日常运行时入口；若需判断“当前任务该读什么、预算是否超限、哪份规则是主定义”，优先查阅 `runtime-entry.md`、`skeleton-loading-matrix.md`、`context-budget-baseline.md` 与 `rule-authority-matrix.md`。

---

## 一、典型落地路径 / 上手路径

如果团队准备正式引入 MES-AI-DEV，建议按下面的顺序推进，而不是一开始就追求“AI 全链路接管”。

### 第 1 步：先完成初始化与知识建图

目标：

- 让 AI 先知道系统的真实边界
- 建立基础代码地图、依赖图、数据库索引和路径映射

产出重点：

- code-map
- dependency-graph
- database-index
- state.yaml

适合结果：

> AI 已经能做“看哪里、读什么、先查什么”的受控分析，而不是盲搜全仓。

### 第 2 步：从需求分析与设计阶段开始纳入治理

目标：

- 不急着让 AI 直接改很多代码
- 先让 AI 在需求分析、影响范围、设计方案、ADR 记录上稳定工作

产出重点：

- requirement spec
- impact scope
- design doc
- api / db / service-chain 设计
- 阶段交接与阶段记忆

适合结果：

> AI 先成为“分析和设计执行单元”，团队开始建立对其输出的信任。

### 第 3 步：把开发与测试纳入正式链路

目标：

- 让 AI 不只会写内容，还能按阶段门禁、交接、证据和记忆工作
- 逐步把开发、测试、交付纳入正式流程

产出重点：

- task plan
- code implementation
- self review
- test plan / test report
- release / handover

适合结果：

> AI 从“局部助手”升级为“可治理的工程执行单元”。

### 第 4 步：持续沉淀阶段记忆与治理经验

目标：

- 不只是完成一个需求
- 而是让 AI 在连续需求中越做越稳

产出重点：

- pitfall
- pattern
- constraint
- governance memory
- skeleton change log

适合结果：

> 团队拥有的不只是一个会干活的 AI，而是一套会持续学习、持续避免重复错误的工程治理体系。

## 二、采用前 / 采用后

| 维度 | 采用前 | 采用后 |
|---|---|---|
| AI 使用方式 | 主要靠聊天、临时提示词、零散问答 | 进入正式阶段链路，按命令 / Skill / 模板 / 门禁执行 |
| 知识获取 | 依赖个人经验、临时搜索、反复问人 | 通过 code-map、dependency-graph、阶段记忆、全局记忆库受控消费 |
| 多阶段衔接 | 容易断层，靠人手工补上下文 | 通过 stage-output-report、handoff、memory 文件稳定接力 |
| 大仓可操作性 | AI 易失控、易爆上下文 | 分层索引、热点优先、预算守卫支持大仓 / 超大仓 |
| 历史经验复用 | 踩过的坑留在聊天记录或个人记忆里 | pitfall / pattern / constraint 可持续沉淀与复用 |
| 代码仓贴合度 | 容易按照理想结构想象实现 | 强制贴合目标仓真实服务 / 模块 / Schema 结构 |
| 审查与追溯 | 容易只看结论，不看证据链 | 报告、证据、门禁、日志形成完整审计路径 |
| 团队协作方式 | AI 更像个人助手 | AI 成为团队流程中的正式执行单元 |

## 三、最小试点建议

如果团队准备开始试点，不建议一上来就把所有能力全开。更稳妥的方式是先跑一个**最小闭环试点**。

### 推荐先选什么样的试点

优先选择：

- 范围中等、边界清晰的需求
- 涉及 1~3 个主要服务 / 模块的改动
- 有明确业务 owner 和技术 owner 的需求
- 能走完整链路，但风险可控的需求

不建议一开始就选：

- 超高风险生产改造
- 涉及大规模数据库破坏性变更的需求
- 边界极其模糊、依赖方很多的需求
- 团队本身还没有统一规则的项目

### 推荐先开启哪些能力

建议第一批先启用：

- 初始化建图与知识索引
- 需求分析
- 详细设计
- 阶段门禁
- 阶段交接与阶段记忆

### 第一轮先不要过度开启的能力

建议先谨慎推进：

- 大范围代码自动改造
- 高风险数据库迁移自动执行
- 大规模跨仓并行开发
- 高风险发布自动执行

### 一个稳妥的试点节奏

1. 先把初始化和知识消费跑顺
2. 再让 AI 先承担分析 / 设计
3. 然后在局部开发任务中试运行
4. 最后再把测试 / 交付逐步纳入

这样更容易建立团队信任，也更容易快速发现治理缺口。

## 四、常见引入失败模式

即使骨架能力完整，引入方式不当也会失败。下面是最常见的失败模式。

### 1. 一开始就想全链路接管

问题：

- 还没建立知识库和阶段治理，就希望 AI 直接从需求做到交付

结果：

- 输出不可控
- 信任快速下降
- 团队把问题归因到“AI 不可靠”

更好的做法：

- 先让 AI 在分析、设计、局部开发中稳定工作，再逐步扩大范围

### 2. 初始化不完整就进入开发

问题：

- 代码地图、依赖图、路径映射、数据库索引还没补齐，就要求 AI 做深入设计和开发

结果：

- AI 只能靠猜
- 容易脱离目标仓真实结构

更好的做法：

- 先把初始化补到足够支撑后续消费的程度，再往下推进

### 3. 只改文档，不改规则 / skill / 模板

问题：

- 想通过 README 或局部说明文档来“宣布能力存在”，但底层规则、模板、skill 没接上

结果：

- 能说不能做
- 规则与执行脱节

更好的做法：

- 规则、模板、skill、导航入口、日志要一起改，形成闭环

### 4. 只有阶段产物，没有阶段交接和记忆

问题：

- 阶段结束后只保留正式产物，不保留风险、坑点、决策、blocker 和记忆

结果：

- 下一阶段又重新踩坑
- 跨阶段容易断层

更好的做法：

- 把阶段记忆作为标准产物持续维护

### 5. 把默认示例路径当成真实结构

问题：

- 默认看到 `jalor/`、`web/`、`dbscript/` 就直接按这个结构工作

结果：

- 与目标仓真实结构脱节
- 设计与开发落点错误

更好的做法：

- 永远以初始化探测和真实路径映射为准

## 五、首个试点成功判定标准

首个试点不需要一开始就证明“AI 可以替代所有人”，更现实的目标是证明：

> **AI 已经能够在正式治理约束下稳定参与研发链路，并显著降低团队的搜索、交接和重复劳动成本。**

可以用下面这些标准判断试点是否成功：

### 1. 过程可控

- 需求、设计、开发、测试、交付至少部分阶段已经进入标准链路
- 关键产物、审查报告、交接文件和阶段记忆都能稳定落盘
- 团队能解释“AI 是怎么做出这些结果的”，而不是只能看最终输出

### 2. 交接不断层

- 上一阶段输出能够被下一阶段直接消费
- handoff 和记忆文件真正参与后续阶段，而不是只生成不使用
- 历史坑点能在后续阶段被显式规避

### 3. 目标仓贴合度足够高

- AI 的分析、设计、开发没有明显脱离目标仓真实结构
- 服务 / 模块 / 包 / Schema 归属判断基本稳定
- 团队不再需要频繁纠正“路径猜错、归属猜错、结构想象化”问题

### 4. 团队感知到真实收益

- 搜索和整理上下文的时间明显下降
- 审查与交接成本下降
- 文档、设计、测试、交接类产物输出更稳定
- 重复解释、重复踩坑、重复补上下文的情况减少

### 5. 治理成本低于收益

- 团队没有因为引入骨架而陷入额外的管理负担
- 规则、模板、skill、日志之间基本一致
- 团队愿意继续扩大试点范围，而不是试一次就放弃

如果以上 5 点大部分成立，首个试点就可以视为成功。

## 六、团队角色分工建议

引入 MES-AI-DEV 时，不同角色的关注点不同。最容易失败的方式之一，就是所有人都把它当成“只是开发工具”。

### 产品 / 业务角色

重点职责：

- 提供更结构化的原始需求输入
- 配合澄清边界、验收标准和优先级
- 审阅需求分析产物是否贴近真实业务目标

在试点初期最重要的不是让产品写很多新文档，而是：

> 帮助 AI 和研发团队减少需求歧义。

### 架构 / 技术负责人

重点职责：

- 审核技术方案和设计边界
- 确认目标仓真实结构与设计落点一致
- 决定哪些设计决策需要升级为 ADR
- 识别哪些规则、模板、skill 需要继续收敛

这是试点阶段最关键的角色之一，因为它决定骨架会不会脱离真实系统。

### 开发工程师

重点职责：

- 验证 AI 生成的实现是否贴合现有代码风格和分层结构
- 维护实现阶段的阶段记忆、坑点与 blocker 信息
- 反馈哪些 skill / 模板最容易偏离真实工作方式

试点初期，开发角色更像：

> AI 的执行校正者 + 骨架的真实可用性反馈者。

### 测试工程师

重点职责：

- 让测试阶段不只是“验代码”，而是验证阶段交接、证据链和风险说明是否完整
- 补齐测试资产、可测性矩阵、验证证据等内容
- 识别测试阶段是否还停留在 happy path 思维

### 运维 / 交付角色

重点职责：

- 验证交付边界说明、风险说明、回滚动作是否完整
- 确认 handover 是否真的可接手
- 推动“交文件”升级为“交状态、交风险、交后补动作”

### 项目负责人 / 推动者

重点职责：

- 控制试点节奏，不让团队一开始就全链路接管
- 判断试点成功标准是否达成
- 决定下一轮扩大范围时优先补哪些能力
