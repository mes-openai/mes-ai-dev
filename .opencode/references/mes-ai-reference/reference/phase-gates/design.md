# 详细设计阶段门禁

## 一、进入门禁

### must-pass
- 需求分析阶段已退出
- 需求规格、验收标准、影响范围已存在
- 设计前加载检查已完成
- 已完成 overview 与 index 层消费，且未跳过 `backend-overview.md`、`frontend-overview.md`
- 已明确后端服务名称、代码仓与 Schema
- 已明确前端模块名称、代码仓与路由路径

---

## 二、退出门禁

### must-pass
- `design.md` 已生成，或已明确与历史旧名的兼容映射
- 详细设计阶段完成产物报告已生成，且列出标准产物、已生成文件、文件作用、未生成文件及原因
- 设计文档包含技术方案、数据模型与接口设计
- 新增/变更接口具备请求响应定义
- 涉及数据库变更时，已明确 Schema / 表范围
- 命中多仓/跨服务/多 provider 场景时，服务链已冻结
- 命中私有契约场景时，已显式引用对应契约事实源
- 设计文档已显式标注参考的知识库文件
- 命中参数开关/配置取值场景时，已明确采用 `parameter-switch-patterns.md` 的哪一种模式
- 阶段评审结论为 ✅通过 / ⚠️有条件通过
- 关键设计对象已映射到开发任务、测试对象或验收标准，不存在不可验证的设计结论

### must-pass：服务链冻结
- 最终服务链已明确
- provider 选择理由已记录
- 禁止路径已记录
- 契约兼容边界已记录
- 需要复用的现有接口已优先评估

若仍停留在“多个实现方案均可”而未冻结主路径，不得进入开发阶段。

### must-pass：私有契约显式引用
- 已显式引用对应契约知识或事实源
- 关键字段与关键行为可追溯到定义点
- 未使用通用模板替代项目私有约束

未显式引用私有契约时，不得视为设计通过。

### must-pass：加载检查与知识消费顺序
- 已加载 `phase-design.md` 规定的必读入口
- 已读取 `backend-overview.md` 与 `frontend-overview.md`
- 命中参数开关/配置取值场景时，已读取 `parameter-switch-patterns.md`
- 已按 overview → index → detail/file-summaries → 精准源码 的顺序消费
- 未跳过 overview 与 index 直接生成设计文档

任一项不满足，当前设计应判定为 ❌不通过。

### must-pass：专项违规自动检测
- 若创建新的独立基表，必须说明为何无法复用现有模式；否则判定为违规
- 若未明确代码仓定位（服务名称、模块名称），判定为违规
- 若未显式标注参考了哪些知识库文件，判定为违规
- 若命中参数开关场景但未遵循 `parameter-switch-patterns.md` 的三种模式之一，判定为违规

命中以上任一违规项，不得视为设计通过。

### should-check
- ADR 已记录关键决策
- `cross-service-consistency.md` 已完成
- 如触发存量变更，兼容性设计已完成
- 如涉及 DB / 性能敏感链路，性能影响分析已完成
- 安全设计点与可观测性设计已输出
- 设计到实现映射表已补充
- 若本次设计依赖全仓视角的依赖图、热点层或跨仓一致性判断，应确认已完成 `/mes-init-converge`
- 已检查编码前思考结果，包含约束、备选方案、取舍理由、兼容性边界与回退边界
- 已检查简洁优先，未引入不必要抽象层、平行机制或平台化设计
- 已检查精准修改，设计范围未超出已确认仓、服务、表、接口、页面、配置与脚本
- 若使用 GitNexus 类能力，已说明服务链、依赖关系、Provider 路径或影响面的校验结果及其事实来源
- 若使用 graphify 类能力，已说明图谱导读不替代 `design.md` 正文结论

### advisory
- 回归范围说明与设计偏差预警已提前沉淀
