# 测试验证阶段门禁

## 一、进入门禁

### must-pass
- 开发阶段已退出
- 自审无严重未解决问题
- 需求规格与设计文档可作为测试输入

---

## 二、退出门禁

### must-pass
- 测试用例与测试报告已生成
- 测试验证阶段完成产物报告已生成，且列出标准产物、已生成文件、文件作用、未生成文件及原因
- `test-cases.md` 已保留用户补充与确认后的最终计划
- 本轮新生成测试用例全部通过
- 本轮生成/修改并纳入验证范围的行覆盖率、分支覆盖率、方法覆盖率均达到 100%
- P1 用例全部通过
- P2 用例通过率达到约定阈值
- 无未关闭 P0 缺陷
- 若本轮测试将支撑正式验收或发布判断，需求项到测试结论的追溯关系已形成
- 阶段评审结论为 ✅通过 / ⚠️有条件通过
- 本轮生成或修改的单元测试不得包含平台相关路径硬编码，且应兼容 Windows / Linux 路径差异
- 本轮生成或修改的 Mockito 测试应符合参数匹配一致性与空指针预防要求，不得通过脆弱 mock 制造虚假通过

### should-check
- 单元测试、回归测试数据、UAT 场景、冒烟测试、缺陷跟踪已完成
- 测试数据治理已落实
- 数据回归验证已完成（MES 核心链路）
- 未覆盖需求项、局部验证项与限制条件已显式记录
- `test-report.md` 已覆盖测试交接结论
- 若本次测试依赖全仓视角的调用链、热点服务/API/表或全局覆盖率判断，应确认已完成 `/mes-init-converge`
- 已检查目标驱动执行，明确测试目标、成功标准、证据路径与未覆盖限制
- 已检查简洁优先，测试用例聚焦目标行为，未为覆盖率制造复杂、脆弱或无意义断言
- 已检查精准修改，失败修复未扩大到无关代码或无关测试
- 若使用 GitNexus 类能力，已说明其辅助反推的影响面、调用链与回归路径，并与真实执行证据区分
- 若使用 graphify 类能力，已说明其需求项、测试对象、证据与风险关系图不替代 `test-report.md` 结论

### advisory
- 性能、安全、灰度等补充验证项已沉淀证据
