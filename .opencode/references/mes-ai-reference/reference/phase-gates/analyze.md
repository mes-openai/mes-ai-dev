# 需求分析阶段门禁

## 一、进入门禁

### must-pass
- 知识库已初始化
- 原始需求输入存在

---

## 二、退出门禁

### must-pass
- `spec.md` 已生成，或已明确与历史旧名的兼容映射
- `impact-scope.md` 已生成
- 需求分析阶段完成产物报告已生成，且列出标准产物、已生成文件、文件作用、未生成文件及原因
- 核心歧义已收敛，无阻断性冲突
- 验收标准已形成可执行表述
- 命中多仓/跨服务/公共契约场景时，仓级责任边界已形成
- 命中多 provider / 复用场景时，服务链与 provider 选择已形成
- 阶段评审结论为 ✅通过 / ⚠️有条件通过
- 验收标准已具备可验证表达，且可被后续设计、开发、测试或交付阶段追溯

### must-pass：仓级责任边界
- 已区分候选仓、确认仓、待补证仓
- 已明确仓级责任边界
- 已说明各仓涉及原因
- 已识别事实定义源、实现落点与消费方角色

若只列出“可能受影响仓”而未形成责任边界，不得视为通过。

### must-pass：服务链与 provider 选择
- 已串联关键业务链、服务链与数据链
- 已区分技术可达 provider 与架构允许 provider
- 已记录最终选定 provider 及理由
- 已判断现有能力复用、最小扩展或新增接口路径

未明确 provider 选择依据时，不得进入设计阶段。

### should-check
- 若命中歧义/多方案/边界不清触发条件，`exploration.md` 已生成并被 `proposal.md` 或 `spec.md` 吸收
- 若命中方案取舍/用户确认触发条件，`proposal.md` 已生成并被 `spec.md` 吸收
- `ambiguity-checklist.md`、`acceptance-criteria-graded.md`、`nfr-checklist.md` 已生成
- `acceptance-trace-matrix.md` 已生成或已说明为何当前不需要正式追溯矩阵
- 非功能需求、需求来源可信度、范围冻结点已明确
- 涉及服务/模块已列出
- 若本次分析依赖全仓视角的 overview / dependency / hot 层结果，应确认已完成 `/mes-init-converge`
- 已检查编码前思考结果，包含关键假设、隐藏歧义、范围边界与方案取舍
- 已检查简洁优先，优先复用现有能力、最小扩展或最小闭环方案
- 已检查精准修改，未把当前需求范围扩大到无关仓、服务、表、页面或配置
- 若使用 GitNexus 类能力，已说明其辅助识别的影响范围、调用链或依赖链，并标注哪些结论已被项目事实源支撑
- 若使用 graphify 类能力，已说明其图谱导读与正式规格结论之间的映射关系

### should-check：契约与消费者影响
- 是否识别响应/错误/认证等契约的定义源
- 是否识别直接消费者与间接消费者
- 是否识别契约变更对下游的影响范围
- 是否识别公共 SDK / 协议仓的事实角色

### advisory
- 需求追溯链与业务规则确认状态已补充完整
