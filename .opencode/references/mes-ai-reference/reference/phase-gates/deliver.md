# 发布交付阶段门禁

## 一、进入门禁

### must-pass
- 测试验证阶段已退出
- P1 全通过，P2 达到约定阈值

---

## 二、退出门禁

### must-pass
- `deploy-plan.md` 已生成且包含回滚方案
- 发布交付阶段完成产物报告已生成，且列出标准产物、已生成文件、文件作用、未生成文件及原因
- `release-note.md` 已生成
- `go-nogo.md` 已完成并给出 Go / Conditional Go 结论
- 若交付结论依赖需求验收，需求项到测试/验收结论的追溯关系已形成
- 并发锁已释放
- `status-tracker.md` 已更新为交付状态
- 阶段评审结论为 ✅通过 / ⚠️有条件通过

### should-check
- 验收报告、部署日志、交接文档、版本治理记录、发布后观察清单已生成
- `acceptance-trace-matrix.md` 已生成，或验收报告中已包含等价追溯章节
- 运维手册、用户手册、交付包审计记录已完成
- 已检查编码前思考结果，包含交付对象、Go/No-Go 标准、风险分层、回滚策略、观察指标与后补动作
- 已检查简洁优先，交付材料服务发布判断、回滚恢复和交接消费，未生成无关长文档
- 已检查精准修改，交付阶段未重新定义需求、契约、Provider、仓边界或测试结论
- 已检查目标驱动执行，部署计划、验收结论、回滚路径和交接状态均可验证
- 若使用 GitNexus 类能力，已说明其辅助核对的发布包对象、调用链、回滚影响面或刷新对象，并与真实发布证据区分
- 若使用 graphify 类能力，已说明交付图谱导读不替代 `go-nogo.md`、`deploy-plan.md`、`handover-doc.md` 或阶段主结论

### advisory
- 交付边界说明、发布后复盘准备项已沉淀

---

## 三、说明

本阶段的通用步骤级门禁、评审结构与 mandatory-record 要求，统一以 `common.md` 为准。
