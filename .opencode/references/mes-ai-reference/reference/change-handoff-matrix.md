# 变更交接与知识回写映射

## 一、用途
- 明确开发阶段的变更对象如何交接给测试阶段。
- 明确开发/交付阶段的事实变更何时需要提示知识刷新阶段回写。

## 二、开发到测试的最小交接对象

当开发阶段改动将进入测试阶段消费时，至少应交接以下内容中的适用项：

| 交接项 | 用途 | 典型产物 |
|---|---|---|
| 变更对象清单 | 说明本次改动覆盖哪些接口、服务、表、页面、配置 | `impact-ledger.md` |
| 验证对象变化清单 | 说明哪些测试对象必须新增、修改或回归 | `verify-object-change-list.md` |
| 验证证据摘要 | 说明开发阶段已执行过的最小验证 | `verification-evidence.md` |
| 设计偏差与限制条件 | 说明与设计、私有契约或预期路径的偏差 | `self-review-report.md` / `impact-ledger.md` |

## 三、开发/交付到刷新的知识回写提示

当代码、配置、契约或交付事实变化会影响知识库时，应形成刷新提示，至少回答：

- 哪些知识文件可能受影响
- 影响来源是什么（代码/配置/数据库/接口/交付状态）
- 是必须刷新、建议刷新还是人工复核

推荐交接产物：
- `refresh-hints.md`

## 四、哪些变更应提示 refresh

| 变更类型 | 是否应提示 refresh | 说明 |
|---|---|---|
| API/契约变化 | 必须 | 影响 api-registry、contracts、frontend-backend-map |
| 服务依赖关系变化 | 必须 | 影响 service-dependencies、business-flow 相关知识 |
| 数据库对象变化 | 必须 | 影响 database-registry、database-index |
| 前端路由/页面与后端映射变化 | 必须 | 影响 frontend-backend-map |
| 仅实现细节小修复且不改变外部事实 | 可选 | 可只记入局部验证与交接，不必强制 refresh |
| 交付结论/验收边界变化 | 建议 | 影响运行期/交付知识边界，建议回写说明 |

## 五、消费规则

### 5.1 test 消费规则
- 若缺少 `verify-object-change-list.md` 或等价信息，test 阶段不得默认认为已覆盖全部变更对象。

### 5.2 refresh 消费规则
- 若存在 `refresh-hints.md`，refresh 阶段应优先将其作为增量刷新范围提示。
- 若开发/交付阶段声明“知识不受影响”，也应说明判断依据，而不是默默跳过。
