# 业务校验规则结构化方法模板（validation-rule-structuring-template）

> 用于将业务语言描述的校验规则转换为结构化、可追溯、可开发的规则表达式。
> 目标是弥合"业务表述"与"技术实现"之间的断层。

## 一、模板定位

- **文件角色**：方法论模板（不依赖具体业务）
- **适用阶段**：需求分析 / 详细设计
- **使用时机**：当需求包含"校验、阻止、拦截、不允许"等业务约束表述时
- **输出产物**：结构化校验规则表（可追加到 requirement-spec 或 design-doc）

---

## 二、结构化方法总览

### 2.1 转换目标

将业务语言描述的校验规则转换为以下结构：

| 维度 | 说明 | 示例 |
|------|------|------|
| 规则ID | 唯一标识 | `VAL-001` |
| 规则名称 | 业务语义名称 | 阻止已发货订单履行 |
| 触发点 | 校验执行的入口位置 | 订单履行API入口 / 订单状态变更Service |
| 判断对象 | 校验的字段/状态/实体 | `order.status` |
| 判断条件 | 具体的判断逻辑表达式 | `status == "已发货"` |
| 校验动作 | 条件满足后的动作 | 阻断 + 返回错误码 + 提示信息 |
| 影响范围 | 校验涉及的层次 | 前端阻断 + 后端阻断 |
| 置信度 | 规则来源的可信程度 | 确认 / 候选 / 待补证 |

### 2.2 转换流程

```mermaid
graph LR
    A[业务语言原文] --> B[识别触发点]
    B --> C[识别判断对象]
    C --> D[识别判断条件]
    D --> E[识别校验动作]
    E --> F[识别影响范围]
    F --> G[标注置信度]
    G --> H[结构化规则表]
```

## 三、详细转换步骤

### 3.1 步骤一：识别触发点

**目标**：确定校验应该在哪个入口执行

**判断维度**：

| 维度 | 判断依据 | 典型触发点 |
|------|----------|------------|
| 用户交互入口 | 业务规则涉及"用户操作时" | 前端页面提交按钮点击 / API请求入口 |
| 数据流转入口 | 业务规则涉及"数据变更时" | Service方法入口 / DAO写入前 |
| 状态变更入口 | 业务规则涉及"状态流转时" | 状态变更Service方法 / 状态机流转点 |
| 跨服务边界 | 业务规则涉及"跨服务数据" | 跨服务调用前 / 数据同步入口 |

**输出**：
```markdown
触发点候选列表：
1. [候选1] - [位置描述] - [置信度]
2. [候选2] - [位置描述] - [置信度]
最终选定触发点：[触发点] - [选择理由]
```

### 3.2 步骤二：识别判断对象

**目标**：确定校验判断的字段/状态/实体是什么

**判断维度**：

| 维度 | 判断依据 | 示例 |
|------|----------|------|
| 业务实体 | 规则涉及的业务对象 | 订单 / 标签 / 工序 |
| 状态字段 | 规则涉及的"已XX/未XX"表述 | `order.shippingStatus` |
| 属性字段 | 规则涉及的具体属性 | `label.printStatus` |
| 关联实体 | 规则涉及的关联对象 | 订单→履行记录 |

**输出**：
```markdown
判断对象识别：
- 主判断对象：[实体].[字段]
- 关联判断对象：[关联实体].[字段]（如需要）
- 对象来源：[数据库表/枚举类/实体类]
- 字段枚举值：[枚举值列表]（如状态字段）
```

### 3.3 步骤三：识别判断条件

**目标**：将业务语言的"已XX/未XX/不允许XX"转换为具体的判断逻辑

**典型转换表**：

| 业务表述 | 逻辑表达式模板 | 说明 |
|----------|----------------|------|
| 已XX | `field == XX_STATUS` | 状态等于指定值 |
| 未XX | `field != XX_STATUS` | 状态不等于指定值 |
| 不允许XX | `field != XX_STATUS` | 同"未XX" |
| 存在XX | `field != null && field.isNotEmpty()` | 对象存在 |
| 不存在XX | `field == null || field.isEmpty()` | 对象不存在 |
| 数量大于N | `count > N` | 数量判断 |
| 首次XX | `count == 0` | 首次操作判断 |
| 补打/重做 | `count > 0` | 非首次操作判断 |

**输出**：
```markdown
判断条件表达式：
- 主条件：[逻辑表达式]
- 组合条件：[AND/OR组合]（如需要）
- 条件来源：[业务表述原文 → 表达式转换依据]
```

### 3.4 步骤四：识别校验动作

**目标**：确定条件满足后应该执行什么动作

**动作分类**：

| 动作类型 | 说明 | 典型实现 |
|----------|------|----------|
| 硬阻断 | 不允许继续执行 | 返回错误响应 + 中断流程 |
| 软阻断 + 预警 | 允许继续但需要用户确认 | 弹出警告提示 + 用户确认后继续 |
| 自动修正 | 自动修正后继续执行 | 自动设置默认值 + 继续执行 |
| 记录日志 | 仅记录不阻断 | 写入日志 + 继续执行 |

**输出**：
```markdown
校验动作定义：
- 动作类型：[硬阻断/软阻断/自动修正/记录日志]
- 阻断方式：[抛异常/返回错误码/弹出提示]
- 提示信息：[用户可见的提示文案]
- 错误码：[错误码（如有）]
- 后续动作：[阻断后的处理流程]
```

### 3.5 步骤五：识别影响范围

**目标**：确定校验涉及的层次（前端/后端/跨服务）

**范围分类**：

| 影响层次 | 说明 | 必要性 |
|----------|------|--------|
| 前端阻断 | 前端页面直接阻断提交 | 用户体验优化，但不可替代后端校验 |
| 后端Controller校验 | API入口层校验 | 必要（防止绕过前端） |
| 后端Service校验 | 业务逻辑层校验 | 必要（业务逻辑一致性） |
| 跨服务校验 | 涉及其他服务的数据校验 | 按需（跨库数据一致性） |

**原则**：
- 前端校验不可替代后端校验
- Controller校验 + Service校验必须同时存在（双重校验）

**输出**：
```markdown
影响范围：
- 前端校验：[是否需要] - [校验位置]
- Controller校验：[是否需要] - [校验方法]
- Service校验：[是否需要] - [校验方法]
- 跨服务校验：[是否需要] - [调用链路]
```

### 3.6 步骤六：标注置信度

**目标**：明确规则来源的可信程度

> **置信度定义见**：`.opencode/references/mes-ai-reference/reference/glossary-confidence-level.md`

**置信度分类**：
- **确认**：已找到明确事实来源（有代码/枚举/数据库字段支撑）
- **候选**：有局部证据但未完全闭合（需补证）
- **待补证**：当前范围内无足够依据（需人工确认）

**输出**：
```markdown
置信度标注：
- 规则置信度：[确认/候选/待补证]
- 证据来源：[代码路径/枚举类/数据库字段]
- 待补证项：[需要补充确认的内容]
```

## 五、使用示例（模板占位）

> 以下示例为占位格式，实际使用时需填充具体项目的真实规则。

### 示例规则转换

**业务原文**：
> 阻止已发货的订单履行

**结构化转换过程**：

1. **触发点识别**：
   - 候选1：订单履行API入口（Controller） - 置信度：候选
   - 候选2：订单履行Service方法 - 置信度：候选
   - 最终选定：Controller + Service双重校验 - 选择理由：防止绕过前端，保障业务一致性

2. **判断对象识别**：
   - 主判断对象：`order.shippingStatus`
   - 对象来源：数据库表 `t_order.shipping_status`
   - 字段枚举值：`SHIPPED(已发货)`、`NOT_SHIPPED(未发货)` - 来源：`ShippingStatusEnum.java`

3. **判断条件识别**：
   - 主条件：`order.shippingStatus == ShippingStatusEnum.SHIPPED`
   - 条件来源：业务表述"已发货" → 枚举值 `SHIPPED`

4. **校验动作识别**：
   - 动作类型：硬阻断
   - 阻断方式：返回错误响应
   - 提示信息："订单已发货，不允许履行"
   - 错误码：`ERR_ORDER_ALREADY_SHIPPED`

5. **影响范围识别**：
   - 前端校验：需要 - 订单履行页面提交前校验
   - Controller校验：需要 - `OrderFulfillmentController.fulfill()`方法入口
   - Service校验：需要 - `OrderFulfillmentService.fulfill()`方法入口
   - 跨服务校验：不需要 - 订单数据在本服务内

6. **置信度标注**：
   - 规则置信度：确认
   - 证据来源：`ShippingStatusEnum.java`、`t_order.shipping_status`字段
   - 待补证项：无

**结构化输出**：

| 维度 | 内容 |
|------|------|
| 规则ID | VAL-001 |
| 规则名称 | 阻止已发货订单履行 |
| 业务原文 | 阻止已发货的订单履行 |
| 触发点 | Controller入口 + Service方法入口 |
| 判断对象 | `order.shippingStatus` |
| 判断条件 | `shippingStatus == SHIPPED` |
| 校验动作 | 硬阻断 + 返回错误码 |
| 提示信息 | "订单已发货，不允许履行" |
| 错误码 | `ERR_ORDER_ALREADY_SHIPPED` |
| 影响范围 | 前端 + Controller + Service |
| 置信度 | 确认 |
| 证据来源 | `ShippingStatusEnum.java`、`t_order.shipping_status` |

## 七、约束规则

### 必须遵守
1. 每条校验规则必须经过完整的六步转换流程
2. 判断条件必须是具体的逻辑表达式，不可保留"已XX"等业务表述
3. 必须标注置信度和证据来源
4. 前端校验不可替代后端校验，必须标注"Controller + Service双重校验"

### 禁止行为
1. ❌ 禁止将业务表述直接写入判断条件（如"已发货"）
2. ❌ 禁止跳过置信度标注
3. ❌ 禁止仅标注前端校验而忽略后端校验
4. ❌ 禁止将推断性结论标注为"确认"置信度

---

## 八、风险提示

- **置信度标注缺失**：会导致后续开发基于错误假设实现
- **前端校验替代后端校验**：会导致校验可绕过，业务约束失效
- **判断条件不具体**：会导致开发阶段需要人工补充逻辑，可能不一致

---

## 九、消费建议

1. 需求分析阶段：使用本模板将业务校验规则结构化，追加到 `spec.md`
2. 详细设计阶段：基于结构化规则设计校验API、校验Service方法、错误码
3. 开发阶段：基于结构化规则实现校验逻辑，禁止自行解读业务表述
4. 测试阶段：基于结构化规则设计校验边界测试用例
