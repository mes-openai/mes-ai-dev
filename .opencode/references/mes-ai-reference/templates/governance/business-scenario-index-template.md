# 业务场景定义索引模板（business-scenario-index-template）

> 用于从代码、枚举、配置、状态字段中提取业务场景的候选定义，形成业务场景定义索引。
> 目标是弥合"业务场景表述"与"代码实现定义"之间的断层，防止AI基于通用常识推断业务语义。

## 一、模板定位

- **文件角色**：方法论模板（指导如何提取和确认业务场景定义）
- **适用阶段**：初始化 / 需求分析
- **使用时机**：当需求涉及业务场景表述（如"补打"、"已发货"、"未排产"）且需要明确定义来源
- **输入**：需求中的业务场景表述 + 代码中的候选定义
- **输出**：业务场景定义索引（含候选定义、定义来源、置信度、待确认事项）

---

## 二、业务场景定义类型分类

### 2.1 定义来源类型

| 来源类型 | 说明 | 置信度 | 典型位置 | 提取方式 |
|----------|------|--------|----------|----------|
| **枚举类定义** | 枚举类中明确定义的场景值 | **高** | `enums/XxxEnum.java` | 读取枚举类注释和枚举值 |
| **状态字段定义** | 数据库表中状态字段的业务含义 | **中** | `t_xxx.status` + 数据字典 | 读取数据库字段注释和数据字典 |
| **配置项定义** | 配置文件中定义的场景开关或参数 | **中** | `application.yml` | 读取配置项注释和参数说明 |
| **人工定义** | 业务人员口头定义或文档定义 | **待确认** | 需求文档、口头沟通 | 人工确认后记录 |
| **推断定义** | 基于代码逻辑推断的场景含义 | **候选** | Service方法逻辑 | 读取Service方法注释和逻辑 |

### 2.2 定义置信度分级

> **置信度定义见**：`.opencode/references/mes-ai-reference/reference/glossary-confidence-level.md`

| 置信度 | 说明 | 定义来源 |
|--------|------|----------|
| 确认 | 已找到明确事实来源 | 枚举类/数据字典/人工定义 |
| 候选 | 有局部证据但未完全闭合 | 状态字段推断/配置项推断 |
| 待补证 | 当前范围内无足够依据 | 需人工补充定义 |

## 三、业务场景定义提取流程

### 3.1 提取流程总览

```mermaid
graph LR
    A[识别业务场景表述] --> B[枚举类候选定义提取]
    B --> C[状态字段候选定义提取]
    C --> D[配置项候选定义提取]
    D --> E[代码逻辑候选定义推断]

```

### 3.2 详细提取步骤

#### 步骤一：识别业务场景表述

**目标**：从需求中识别涉及业务场景的表述关键词

**典型业务场景表述关键词**：

| 关键词类型 | 说明 | 示例 |
|----------|------|------|
| 状态类表述 | "已XX"、"未XX"、"XX状态" | "已发货"、"未排产"、"打印状态" |
| 操作类表述 | "补打"、"重做"、"首次XX" | "补打标签"、"重做订单" |
| 判断类表述 | "允许XX"、"不允许XX"、"阻止XX" | "允许履行"、"不允许流转" |
| 场景类表述 | "XX场景"、"XX情况"、"XX条件下" | "补打场景"、"已发货情况" |

**输出**：
```markdown
业务场景表述识别：
- 业务场景表述：[原文表述]
- 表述关键词：[关键词列表]
- 场景类型：[状态类/操作类/判断类/场景类]
- 涉及业务对象：[实体/字段]
```

#### 步骤二：枚举类候选定义提取

**目标**：从枚举类中提取候选定义

**提取流程**：

| 提取项 | 提取依据 | 提取方式 |
|--------|----------|----------|
| 枚举类路径 | 枚举类文件位置 | `glob("**/enums/*Enum.java")` |
| 枚举值名称 | 枚举值的变量名 | 读取枚举类内容 |
| 枚举值含义 | 枚举值的注释或构造参数 | 读取枚举类注释 |
| 枚举值代码 | 枚举值的数值代码 | 读取枚举类构造参数 |

**提取示例**：
```java
// 枚举类示例
public enum LabelPrintStatusEnum {
    NOT_PRINTED(0, "未打印"),
    PRINTED(1, "已打印"),
    REPRINT(2, "补打");  // ← 候选定义：枚举值 REPRINT 表示"补打"
    
    private final int code;
    private final String desc;
}
```

**输出模板**：
```markdown
枚举类候选定义提取：
- 枚举类路径：[文件路径]
- 枚举值名称：[枚举值变量名]
- 枚举值含义：[枚举值注释]
- 枚举值代码：[数值代码]
- 置信度：高（枚举类明确定义）
- 定义来源：枚举类定义
- 待确认事项：无（枚举类已明确定义）
```

#### 步骤三：状态字段候选定义提取

**目标**：从数据库状态字段中提取候选定义

**提取流程**：

| 提取项 | 提取依据 | 提取方式 |
|--------|----------|----------|
| 数据库表 | 状态字段所属表 | 从 `database-registry.md` 识别 |
| 状态字段名 | 状态字段列名 | 从表结构识别 |
| 字段注释 | 字段的业务含义注释 | 从DDL注释或数据字典识别 |
| 字段值含义 | 字段值的业务含义 | 从数据字典或枚举映射识别 |

**提取示例**：
```sql
-- 数据库字段示例
CREATE TABLE t_order (
    id BIGINT PRIMARY KEY,
    shipping_status INT COMMENT '发货状态：0-未发货，1-已发货',  -- ← 候选定义：字段值1表示"已发货"
    label_print_count INT COMMENT '标签打印次数',  -- ← 候选定义：打印次数>0可能表示"补打"
);
```

**输出模板**：
```markdown
状态字段候选定义提取：
- 数据库表：[表名]
- 状态字段名：[字段名]
- 字段注释：[注释内容]
- 字段值含义：[值含义推断]
- 置信度：中（需确认字段值与业务语义映射）
- 定义来源：数据库字段推断
- 待确认事项：需确认字段值与业务语义的准确映射
```

#### 步骤四：配置项候选定义提取

**目标**：从配置文件中提取候选定义

**提取流程**：

| 提取项 | 提取依据 | 提取方式 |
|--------|----------|----------|
| 配置文件 | 配置文件位置 | `application.yml`、`application.properties` |
| 配置项名称 | 配置项key | 读取配置文件 |
| 配置项含义 | 配置项的注释或命名语义 | 从配置项命名推断含义 |
| 配置项值 | 配置项的值 | 读取配置项值 |

**提取示例**：
```yaml
# 配置文件示例
label:
  print:
    reprint:
      warning:
        enabled: true  # ← 候选定义：配置项暗示存在"补打预警"功能
```

**输出模板**：
```markdown
配置项候选定义提取：
- 配置文件：[文件路径]
- 配置项名称：[配置key]
- 配置项含义推断：[从命名推断的含义]
- 置信度：候选（需人工确认配置项与业务场景的关系）
- 定义来源：配置项推断
- 待确认事项：需确认配置项是否定义了业务场景
```

#### 步骤五：代码逻辑候选定义推断

**目标**：从Service方法逻辑中推断候选定义

**提取流程**：

| 提取项 | 提取依据 | 提取方式 |
|--------|----------|----------|
| Service方法 | 业务逻辑方法位置 | 从 `ownership.md` 或代码搜索识别 |
| 方法逻辑 | 方法中的判断逻辑 | 读取Service方法代码 |
| 逻辑推断 | 从判断逻辑推断场景含义 | 从if/switch语句推断 |

**提取示例**：
```java
// Service方法示例
public void fulfillOrder(Order order) {
    // 从代码逻辑推断："labelPrintCount > 0" 可能表示"补打"
    if (order.getLabelPrintCount() > 0) {
        log.warn("补打标签订单履行：orderId={}", order.getId());
    }
}
```

**输出模板**：
```markdown
代码逻辑候选定义推断：
- Service方法：[方法路径]
- 判断逻辑：[if/switch语句]
- 逻辑推断含义：[从逻辑推断的场景含义]
- 置信度：候选（需人工确认逻辑推断是否正确）
- 定义来源：代码逻辑推断
- 待确认事项：需确认代码逻辑是否准确表达业务场景语义
```

#### 步骤六：汇总候选定义列表

## 四、人工确认流程

### 4.1 人工确认步骤

| 步骤 | 确认内容 | 确认方式 |
|------|----------|----------|
| 1 | 确认候选定义的业务语义是否正确 | 与业务人员沟通或查阅需求文档 |
| 2 | 确认候选定义来源是否准确 | 检查代码/数据库/配置是否准确表达业务语义 |
| 3 | 补充人工定义（如候选定义不足） | 记录人工定义并补充定义文档 |
| 4 | 更新置信度 | 将"候选"升级为"确认" |
| 5 | 更新业务场景索引 | 将确认结果写入业务场景索引文件 |

枚举类候选定义提取：

## 八、约束规则

### 必须遵守
1. 业务场景定义必须标注来源类型和路径
2. 业务场景定义必须标注置信度
3. 候选定义必须标注待确认事项
4. 人工确认后必须更新置信度为"确认"
5. 不得将推断定义标注为"确认"置信度

### 禁止行为
1. ❌ 禁止将推断定义直接写成确认定义
2. ❌ 禁止跳过置信度标注
3. ❌ 禁止跳过来源路径标注
4. ❌ 禁止将通用常识作为业务定义（如"补打"的定义必须来自项目特定来源）

## 九、风险提示

- **推断定义误用**：将推断定义当作确认定义使用，会导致业务语义理解偏差
- **通用常识替代项目定义**：不同项目的"补打"可能有不同定义，不得使用通用常识替代项目特定定义
- **置信度标注缺失**：置信度标注缺失会导致后续使用时无法判断定义可信度

## 十、消费建议

1. 初始化阶段：从枚举类、数据库字段、配置项中提取候选定义，形成候选业务场景索引
2. 需求分析阶段：查阅业务场景索引确认业务场景定义，若缺失则触发人工确认流程
3. 详细设计阶段：基于业务场景定义设计校验规则和预警机制
4. 开发阶段：基于业务场景定义实现校验逻辑和预警提示
