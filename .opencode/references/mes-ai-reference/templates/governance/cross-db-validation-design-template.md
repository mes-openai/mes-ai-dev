# 跨库校验方案设计模板（cross-db-validation-design-template）

> 用于设计跨库数据校验的完整方案，确保符合架构限定通道约束，防止绕过限定通道直接跨库查询。
> 目标是弥合"跨库校验需求"与"架构限定通道实现"之间的断层。

## 一、模板定位

- **文件角色**：方法论模板（不依赖具体业务）
- **适用阶段**：需求分析 / 详细设计
- **使用时机**：当校验需要查询其他数据库的数据时（如现场执行服务校验前处理数据）
- **输入**：结构化校验规则 + 归属决策（来自前三个校验模板）
- **输出**：跨库校验方案设计（含限定通道调用链、禁止路径、风险边界）

---

## 二、跨库校验场景识别

### 2.1 跨库校验典型场景

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| 状态一致性校验 | 校验跨库数据的状态是否一致 | 现场执行校验前处理订单状态是否"已发货" |
| 存在性校验 | 校验跨库数据是否存在 | 现场执行校验前处理物料是否存在 |
| 数量一致性校验 | 校验跨库数据的数量是否一致 | 现场执行校验前处理订单数量与执行数量是否匹配 |
| 时间一致性校验 | 校验跨库数据的时间戳是否一致 | 现场执行校验前处理订单创建时间与执行时间是否匹配 |

### 2.2 跨库校验识别条件

命中以下任一条件时，视为跨库校验场景：

| 条件编号 | 条件描述 | 判断依据 |
|----------|----------|----------|
| CDV-001 | 校验判断对象属于其他数据库的表 | 从 `database-cross-access-rule.md` 识别数据库归属 |
| CDV-002 | 校验需要调用其他微服务获取数据 | 从 `layer-decoupling-channel.md` 识别跨层调用 |
| CDV-003 | 校验涉及架构限定通道约束 | 从架构约束规则识别限定通道要求 |
| CDV-004 | 单库服务需要获取双库数据 | 从服务数据库连接分布识别 |

## 三、跨库校验方案设计流程

### 3.1 设计流程总览

```mermaid
graph LR
    A[识别跨库校验场景] --> B[确认限定通道服务]
    B --> C[设计限定通道调用链]
    C --> D[设计本地二次校验]
    D --> E[标注禁止路径]
    E --> F[标注风险边界]
    F --> G[输出跨库校验方案]
```

### 3.2 详细设计步骤

#### 步骤一：识别跨库校验场景

**目标**：确认当前校验是否属于跨库校验场景

**判断流程**：

| 判断项 | 判断依据 | 输出 |
|--------|----------|------|
| 校验判断对象所属数据库 | 从 `database-cross-access-rule.md` 或 `database-registry.md` 识别 | 数据库名称 |
| 当前服务连接的数据库 | 从服务配置（`application.yml`）或 `ownership.md` 识别 | 当前数据库列表 |
| 是否跨库 | 判断对象数据库 ≠ 当前数据库 | 是/否 |
| 是否命中架构限定通道约束 | 从 `database-cross-access-rule.md` 识别 | 是/否 |

**输出**：
```markdown
跨库校验场景识别：
- 判断对象：[实体].[字段]
- 判断对象所属数据库：[数据库A]
- 当前服务连接数据库：[数据库B]
- 是否跨库：是
- 是否命中限定通道约束：是
- 需要通过限定通道服务：[限定通道服务名（待确认）]
```

#### 步骤二：确认限定通道服务

**目标**：确认承担跨库校验的限定通道服务名称和能力

**确认流程**：

| 确认项 | 确认依据 | 输出 |
|--------|----------|------|
| 限定通道服务名称 | 从 `layer-decoupling-channel.md` 识别或人工确认 | 限定通道服务名 |
| 限定通道服务提供的能力 | 从 `layer-decoupling-channel.md` 能力清单识别 | API列表 |
| 限定通道服务调用方式 | 从 `layer-decoupling-channel.md` 调用方式识别或人工确认 | 同步API/异步MQ |

**输出**：
```markdown
限定通道服务确认：
- 限定通道服务名称：[服务名]
- 提供能力：[API名称列表]
- 调用方式：[同步API/异步MQ]
- 是否提供所需能力：是/否
- 若否，需扩展能力：[待扩展API描述]
```

#### 步骤三：设计限定通道调用链

**目标**：设计跨库校验的限定通道调用链路

**设计要素**：

| 要素 | 说明 | 设计要求 |
|------|------|----------|
| 调用方服务 | 发起跨库校验的服务 | 当前服务 |
| 限定通道服务 | 承担跨库调用的通道服务 | 从步骤二确认的服务 |
| 限定通道API | 限定通道服务提供的API | 从步骤二确认的API |
| 调用方式 | 同步API还是异步MQ | 从步骤二确认的方式 |
| 数据返回格式 | 限定通道服务返回的数据格式 | 明确DTO结构 |

**调用链设计模板**：
```markdown
限定通道调用链：
调用方服务 → 限定通道API → 限定通道服务 → 远程数据库

详细设计：
1. 调用方服务：[当前服务]
2. 调用入口：[校验入口方法名]
3. 限定通道API：[API路径]
4. 限定通道服务：[服务名]
5. 远程数据库：[数据库名].[表名]
6. 数据返回格式：[DTO类名]
7. 调用方式：[同步API/异步MQ]
```

#### 步骤四：设计本地二次校验

**目标**：设计本地服务对限定通道返回数据的二次校验逻辑

**原则**：
- 限定通道服务返回数据后，本地服务必须再次校验
- 不得仅依赖限定通道服务的校验结果
- 本地二次校验保障业务逻辑一致性

**设计要素**：

| 要素 | 说明 | 设计要求 |
|------|------|----------|
| 二次校验触发点 | 本地服务对返回数据执行校验的入口 | Service方法入口 |
| 二次校验逻辑 | 本地服务对返回数据执行的具体校验表达式 | 符合 `validation-logic-expression-template.md` 标准 |
| 二次校验失败动作 | 本地二次校验失败后的阻断动作 | 硬阻断 + 返回错误码 |

**二次校验设计模板**：
```markdown
本地二次校验设计：
1. 二次校验触发点：[Service方法名]
2. 二次校验逻辑：
   ```java
   if (remoteData.statusField == StatusEnum.TARGET_STATUS) {
       throw new BusinessException(ErrorCode.XXX, "校验失败提示");
   }
   ```
3. 二次校验失败动作：硬阻断 + 返回错误码
4. 二次校验必要性说明：保障业务逻辑一致性，防止限定通道数据与本地判断不一致
```

#### 步骤五：标注禁止路径

**目标**：明确标注禁止的跨库访问路径

**禁止路径类型**：

| 禁止路径编号 | 禁止路径 | 禁止原因 | 替代允许路径 |
|--------------|----------|----------|--------------|
| BAN-001 | 调用方服务 → 直接访问远程数据库 | 违反单库服务禁止直接跨库访问规则 | 调用方 → 限定通道 → 远程数据库 |
| BAN-002 | 调用方服务 → 绕过限定通道直接调用远程服务 | 违反分层解耦约束 | 调用方 → 限定通道 → 远程数据库 |
| BAN-003 | 限定通道服务 → 返回未经校验的数据 | 违反限定通道能力定义 | 限定通道必须返回可校验的结构化数据 |

**禁止路径标注模板**：
```markdown
禁止路径：
1. ❌ [当前服务] → 直接访问 [远程数据库]：违反单库服务禁止直接跨库访问规则
2. ❌ [当前服务] → 绕过限定通道直接调用 [远程服务]：违反分层解耦约束
3. ❌ [限定通道服务] → 返回未经校验的数据：违反限定通道能力定义

替代允许路径：
✅ [当前服务] → [限定通道API] → [限定通道服务] → [远程数据库]
```

#### 步骤六：标注风险边界

**目标**：明确标注跨库校验的风险边界和回退策略

**风险类型**：

| 风险编号 | 风险类型 | 风险说明 | 风险等级 |
|----------|----------|----------|----------|
| RISK-001 | 限定通道服务不可用 | 限定通道服务宕机或超时，无法获取跨库数据 | 高 |
| RISK-002 | 限定通道数据与本地数据不一致 | 跨库数据同步延迟导致数据不一致 | 中 |
| RISK-003 | 跨库调用性能风险 | 跨库调用引入额外延迟，影响校验性能 | 中 |
| RISK-004 | 跨库调用失败回退风险 | 跨库调用失败后的回退策略不明确 | 高 |

**回退策略设计**：

| 回退策略编号 | 回退策略 | 适用场景 | 实现方式 |
|--------------|----------|----------|----------|
| FALLBACK-001 | 限定通道服务不可用时阻断校验 | 限定通道服务宕机或超时 | 返回错误码 + 提示"跨库校验服务不可用" |
| FALLBACK-002 | 限定通道服务不可用时允许继续（降级） | 非关键校验场景 | 记录日志 + 允许继续（标注风险） |
| FALLBACK-003 | 数据不一致时阻断校验 | 数据同步延迟导致不一致 | 返回错误码 + 提示"数据不一致" |
| FALLBACK-004 | 跨库调用失败时重试 | 网络抖动导致调用失败 | 重试机制（限制重试次数） |

**风险边界标注模板**：
```markdown
风险边界：
1. 限定通道服务不可用风险：高风险
   - 回退策略：阻断校验 + 返回错误码
2. 数据不一致风险：中风险
   - 回退策略：阻断校验 + 返回错误码
3. 跨库调用性能风险：中风险
   - 回退策略：设置超时阈值 + 降级处理
4. 跨库调用失败回退风险：高风险
   - 回退策略：重试机制（限制重试次数）

风险等级总体评估：高风险
```

## 六、与其他模板的关系

| 关联模板 | 关系说明 | 使用顺序 |
|----------|----------|----------|
| `validation-rule-structuring-template.md` | 校验规则结构化 | 上游模板：先结构化规则，再判断是否跨库 |
| `validation-placement-decision-template.md` | 校验点归属判断 | 上游模板：先判断归属，若命中跨库则使用本模板 |
| `validation-logic-expression-template.md` | 校验逻辑表达式设计 | 上游模板：跨库校验需设计本地二次校验表达式 |
| `validation-fallback-template.md` | 校验失败回退边界设计 | **下游模板**：跨库校验方案完成后，设计校验失败回退边界 |

**链路衔接说明**：
- 本模板为**分支模板A**，处理跨库校验场景
- 上游来源：validation-placement-decision判断命中跨库 → 进入本模板
- 下游去向：validation-fallback-template（跨库校验方案传递给回退边界设计）
- 执行顺序：归属判断 → 跨库方案设计 → 回退边界设计

## 七、约束规则

### 必须遵守
1. 跨库校验必须通过限定通道服务，不得直接跨库查询
2. 限定通道服务返回数据后，本地服务必须再次校验（二次校验）
3. 必须标注禁止路径和替代允许路径
4. 必须标注风险边界和回退策略
5. 限定通道服务名称必须从 `layer-decoupling-channel.md` 确认或人工确认

### 禁止行为
1. ❌ 禁止设计直接跨库查询的校验路径
2. ❌ 禁止绕过限定通道服务直接调用远程服务
3. ❌ 禁止仅依赖限定通道服务的校验结果，忽略本地二次校验
4. ❌ 禁止跳过风险边界标注
5. ❌ 禁止跳过回退策略设计

## 八、风险提示

- **直接跨库查询**：违反架构约束，会导致数据一致性无法保障
- **绕过限定通道**：违反分层解耦约束，会导致架构治理失效
- **忽略二次校验**：会导致业务逻辑不一致，限定通道数据可能过期
- **缺少回退策略**：会导致跨库调用失败时无法正确处理，影响业务稳定性

---

## 九、消费建议

1. 需求分析阶段：识别跨库校验场景后，使用本模板设计跨库校验方案，追加到 `spec.md`
2. 详细设计阶段：基于跨库校验方案设计限定通道调用链、二次校验逻辑、风险边界
3. 开发阶段：基于跨库校验方案实现限定通道调用和二次校验逻辑，禁止实现禁止路径
4. 测试阶段：基于跨库校验方案设计跨库校验测试场景（限定通道可用、不可用、数据不一致等）
