# 业务预警机制设计模板（warning-mechanism-template）

> 用于设计业务预警机制的完整方案，区分"硬阻断校验"与"预警提示"，形成完整的预警触发、展示、处理链路。
> 目标是弥合"预警类需求"与"预警机制实现"之间的断层。

## 一、模板定位

- **文件角色**：方法论模板（不依赖具体业务）
- **适用阶段**：需求分析 / 详细设计
- **使用时机**：当需求包含"预警、提示、提醒、注意、警告"等表述时
- **输入**：结构化校验规则（来自 `validation-rule-structuring-template.md`）
- **输出**：预警机制设计方案（含触发条件、展示方式、处理流程）

---

## 二、预警与阻断的区分

### 2.1 核心区分原则

| 类型 | 说明 | 校验动作 | 用户操作 | 示例 |
|------|------|----------|----------|------|
| **硬阻断校验** | 不允许继续执行 | 阻断流程 + 返回错误 | 无法绕过，必须满足条件 | 阻止已发货订单履行 |
| **预警提示** | 允许继续但需用户确认 | 弹出提示 + 用户确认后继续 | 可绕过，用户确认后继续 | 补打标签预警提示 |

### 2.2 区分判断依据

| 判断依据 | 硬阻断校验 | 预警提示 |
|----------|------------|----------|
| 业务表述 | "阻止、不允许、禁止、拦截" | "预警、提示、提醒、注意、警告" |
| 是否允许继续 | ❌ 不允许 | ✅ 允许（需确认） |
| 用户操作 | 无法绕过 | 用户确认后可继续 |
| 后端校验 | 必须阻断 | 可记录日志，不强制阻断 |
| 前端展示 | 错误提示（阻断型） | 警告提示（确认型） |

### 2.3 预警机制定位

> **预警机制 ≠ 校验机制**
> - 校验机制：硬阻断，强制执行
> - 预警机制：软阻断，用户决策驱动

本模板专门用于设计**预警机制**，硬阻断校验请使用 `validation-logic-expression-template.md`。

## 三、预警机制设计流程

### 3.1 设计流程总览

```mermaid
graph LR

    B --> C[设计预警展示方式]

    D --> E[设计预警记录机制]
    E --> F[设计预警关闭条件]
    F --> G[输出预警机制方案]

### 3.2 详细设计步骤

#### 步骤一：识别预警场景

**目标**：确认当前需求属于预警场景而非硬阻断校验场景

**判断流程**：

| 判断项 | 判断依据 | 输出 |
|--------|----------|------|
| 业务表述关键词 | "预警/提示/提醒/注意/警告" vs "阻止/不允许/禁止/拦截" | 预警/阻断 |
| 是否允许继续执行 | 业务是否允许用户确认后继续 | 是/否 |
| 用户决策权 | 用户是否可以决定是否继续 | 是/否 |

**输出**：
```markdown
预警场景识别：
- 业务原文表述：[原文]
- 表述关键词：[预警/提示/提醒/注意/警告]
- 是否允许继续执行：是（需用户确认）
- 用户决策权：有（用户可以决定是否继续）
- 场景类型：预警提示（非硬阻断校验）
```

#### 步骤二：设计预警触发条件

**目标**：设计预警的触发条件表达式

**设计要素**：

| 要素 | 说明 | 设计要求 |
|------|------|----------|
| 触发对象 | 预警判断的对象 | 明确实体和字段 |
| 触发条件 | 预警触发的判断表达式 | 符合 `validation-logic-expression-template.md` 标准 |
| 触发时机 | 预警触发的入口点 | 明确前端/后端触发位置 |
| 触发频率 | 预警触发的频率控制 | 单次触发/持续触发 |

**触发条件设计模板**：
```markdown
预警触发条件：
- 触发对象：[实体].[字段]
- 触发条件表达式：
  ```java
  if ([字段判断表达式]) {
      triggerWarning([预警ID], [预警内容]);
  }
  ```
- 触发时机：[前端提交前/后端入口/Service方法]
- 触发频率：[单次触发（每次操作触发一次）/持续触发（持续显示）]
```

#### 步骤三：设计预警展示方式

**目标**：设计预警在前端的展示方式

**展示方式分类**：

| 展示方式 | 说明 | 适用场景 | 实现方式 |
|----------|------|----------|----------|
| 弹窗确认 | 弹出确认对话框，用户点击"确认"或"取消" | 关键预警，需用户明确决策 | `this.$confirm()` / `MessageBox.confirm()` |
| Toast提示 | 浛地提示，不阻断用户操作 | 非关键预警，仅提醒 | `this.$message.warning()` |
| 页面内嵌提示 | 页面内嵌警告区块，持续显示 | 持续预警，需用户时刻注意 | `<el-alert type="warning">` |
| 标签标记 | 列表/详情页标签标记（如"补打"标签） | 状态标记型预警 | `<el-tag type="warning">` |
| 颜色高亮 | 字段颜色高亮（如红色/橙色） | 字段级预警 | CSS样式控制 |

**展示方式设计模板**：
```markdown
预警展示方式：
- 展示方式：[弹窗确认/Toast提示/页面内嵌提示/标签标记/颜色高亮]
- 展示位置：[页面顶部/字段旁/列表行/弹窗中]
- 展示时机：[触发条件满足时立即显示]
- 展示内容：[预警提示文案]
- 用户操作选项：[确认继续/取消/忽略/查看详情]
- 展示示例：
  ```javascript
  // 弹窗确认示例
  const confirmed = await this.$confirm(
      '该订单为补打标签，请注意核对信息后继续',
      '补打预警',
      { type: 'warning', confirmButtonText: '确认继续', cancelButtonText: '取消' }
  );
  if (confirmed) {
      // 继续提交
  }
  ```
```

#### 步骤四：设计预警处理流程

**目标**：设计预警触发后的完整处理流程

**处理流程要素**：

| 要素 | 说明 | 设计要求 |
|------|------|----------|
| 用户确认后动作 | 用户点击"确认继续"后的流程 | 继续执行原操作 |
| 用户取消后动作 | 用户点击"取消"后的流程 | 中止操作，返回上一状态 |
| 预警记录动作 | 预警触发后的记录动作 | 记录日志/写入预警表 |
| 预警升级动作 | 预警触发后的升级动作（如需要） | 通知管理员/升级为硬阻断 |

**处理流程设计模板**：
```markdown
预警处理流程：
1. 用户确认继续：
   - 动作：继续执行原操作（如提交订单）
   - 后端动作：记录预警日志（可选）
   - 日志内容：{预警ID, 用户ID, 确认时间, 操作类型}
2. 用户取消：
   - 动作：中止操作，返回上一状态
   - 后端动作：记录取消日志（可选）
3. 预警记录动作：
   - 记录方式：日志记录 / 预警表写入
   - 记录内容：{预警ID, 触发对象, 触发时间, 用户决策, 处理结果}
4. 预警升级动作（如有）：
   - 升级条件：[如预警触发次数超过阈值]
   - 升级动作：[通知管理员/升级为硬阻断]
```

#### 步骤五：设计预警记录机制

**目标**：设计预警触发后的记录机制，用于审计和追溯

**记录机制分类**：

| 记录方式 | 说明 | 适用场景 | 实现方式 |
|----------|------|----------|----------|
| 日志记录 | 后端日志记录预警触发和处理 | 轻量级预警，不需要持久化 | `log.warn()` |
| 预警表写入 | 写入专门的预警记录表 | 关键预警，需要追溯和统计 | INSERT预警表 |
| 状态字段标记 | 在业务表字段中标记预警状态 | 状态型预警，需要持续显示 | UPDATE状态字段 |

**记录机制设计模板**：
```markdown
预警记录机制：
- 记录方式：[日志记录/预警表写入/状态字段标记]
- 记录时机：[预警触发时/用户确认后/用户取消后]
- 记录内容：
  ```json
  {
      "warningId": "[预警ID]",
      "triggerObject": "[触发对象]",
      "triggerCondition": "[触发条件]",
      "triggerTime": "[触发时间]",
      "userId": "[用户ID]",
      "userDecision": "[确认继续/取消]",
      "processResult": "[处理结果]"

  ```
- 记录表（如有）：[预警表名]
- 状态字段（如有）：[业务表].[预警状态字段]
```

#### 步骤六：设计预警关闭条件

**目标**：设计预警何时关闭或消失

**关闭条件分类**：

| 关闭条件 | 说明 | 适用场景 | 实现方式 |
|----------|------|----------|----------|
| 用户确认后关闭 | 用户点击"确认继续"后关闭 | 弹窗确认型预警 | 用户操作后立即关闭 |
| 操作完成后关闭 | 操作完成后关闭 | 页面内嵌型预警 | 操作成功后关闭 |
| 状态变更后关闭 | 状态字段变更后关闭 | 状态标记型预警 | 状态字段更新后关闭 |
| 手动关闭 | 用户手动关闭 | Toast提示型预警 | 用户点击关闭按钮 |

**关闭条件设计模板**：
```markdown
预警关闭条件：
- 关闭方式：[用户确认后关闭/操作完成后关闭/状态变更后关闭/手动关闭]
- 关闭时机：[具体关闭时机描述]
- 关闭动作：[关闭时执行的动作，如清除状态字段/更新预警表]

### 示例：补打标签预警提示

**输入**：规则 VAL-002 结构化结果

| 维度 | 内容 |
|------|------|
| 规则ID | VAL-002 |
| 规则名称 | 补打标签预警提示 |
| 业务原文 | 对于补打标签的订单，进行预警提示 |
| 判断对象 | `order.labelPrintCount` |
| 判断条件 | `labelPrintCount > 0` |
| 校验动作 | 预警提示（软阻断） |

**预警机制方案设计输出**：

```markdown
## 预警机制方案设计：补打标签预警提示

|------|------|
| 触发对象 | `order.labelPrintCount` |

| 维度 | 内容 |

展示示例：
```javascript
async beforePrint() {
    const order = await this.getOrder(this.orderId);
    if (order.labelPrintCount > 0) {
        const confirmed = await this.$confirm(
            '该订单为补打标签，请注意核对信息后继续',
            '补打预警',
            { type: 'warning', confirmButtonText: '确认继续', cancelButtonText: '取消' }
        );
        if (!confirmed) {
            return false;  // 用户取消
        }
    }
    return true;  // 继续打印
}

## 六、与其他模板的关系

| 关联模板 | 关系说明 | 使用顺序 |
|----------|----------|----------|
| `validation-rule-structuring-template.md` | 校验规则结构化 | 上游模板：先结构化规则，再判断是预警还是阻断 |
| `validation-placement-decision-template.md` | 校验点归属判断 | 上游模板：预警优先在前端实现，后端可选补充日志 |
| `validation-logic-expression-template.md` | 校验逻辑表达式设计 | 硬阻断校验使用此模板，预警使用本模板 |
| `validation-fallback-template.md` | 校验失败回退边界设计 | **下游模板**：预警机制方案完成后，设计预警确认后失败回退边界 |

**链路衔接说明**：
- 本模板为**分支模板B**，处理预警提示场景
- 上游来源：validation-rule-structuring判断校验动作为"预警提示" → 进入本模板
- 下游去向：validation-fallback-template（预警机制方案传递给回退边界设计）
- 执行顺序：规则结构化 → 预警机制设计 → 回退边界设计

**与硬阻断模板的区别**：
- 硬阻断校验：直接进入 validation-logic-expression-template → validation-fallback-template
- 预警提示：进入 warning-mechanism-template → validation-fallback-template

## 七、约束规则

### 必须遵守
1. 预警必须明确标注为"软阻断"，用户可绕过
2. 预警展示方式必须明确（弹窗/Toast/页面内嵌等）
3. 预警处理流程必须明确（用户确认后动作、取消后动作）
4. 预警记录机制必须明确（日志记录/预警表写入）
5. 预警关闭条件必须明确

### 禁止行为
1. ❌ 禁止将预警标注为"硬阻断"（预警允许用户确认后继续）
2. ❌ 禁止跳过预警展示方式设计
3. ❌ 禁止跳过预警处理流程设计
4. ❌ 禁止跳过预警记录机制设计
5. ❌ 禁止将预警仅在前端实现而不记录（后端必须补充日志）

---

## 八、风险提示

- **用户绕过预警**：预警为软阻断，用户可能忽略预警继续执行，导致风险操作
- **预警触发频率过高**：预警触发频率过高可能导致用户体验下降，需适度控制
- **预警记录不足**：预警记录不足可能导致无法追溯问题，后端必须补充日志记录

---

## 九、消费建议

1. 需求分析阶段：识别预警场景后，使用本模板设计预警机制方案，追加到 `spec.md`
2. 详细设计阶段：基于预警机制方案设计前端展示和后端日志记录
3. 开发阶段：基于预警机制方案实现前端预警展示和后端日志记录
4. 测试阶段：基于预警机制方案设计预警测试场景（预警触发、用户确认、用户取消）
