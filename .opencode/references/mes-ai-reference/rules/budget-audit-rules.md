# 知识库新鲜度SLA与预算审计规则

> 对应摘要：`.opencode/references/mes-ai-reference/rules/budget-audit-rules-summary.md`
> 若当前任务只需判断新鲜度状态、预算状态与基本处理动作，默认优先读取摘要；仅在需要完整审计细则时进入正文。

> 本文件为硬性规则，定义知识库新鲜度标准和上下文预算审计机制。

---

## 一、知识库新鲜度SLA

### 1.1 刷新频率要求

| 变更类型 | 最大刷新间隔 | 触发方式 |
|---------|-------------|---------|
| 新增服务 | 立即刷新 | 执行 `/mes-refresh-knowledge` |
| 删除服务 | 立即刷新 | 执行 `/mes-refresh-knowledge` |
| API新增/修改 | 7天 | 自动提醒或手动执行 |
| 代码重构（包结构变更） | 7天 | 手动执行 |
| 配置变更（端口/依赖） | 7天 | 手动执行 |
| 仅代码逻辑修改 | 14天 | 可定期批量刷新 |

### 1.2 新鲜度检查规则

`mes-refresh-validate-quality` Skill 在校验时同时检查新鲜度：

| 检查项 | 规则 | 不满足时 |
|--------|------|---------|
| `state.yaml` 的 `sync.last_sync` | 不超过7天 | ⚠️ 警告：建议执行 /mes-refresh-knowledge |
| `state.yaml` 的 `sync.status` | 无 `error` | ❌ 阻止：必须修复错误服务 |
| overview中的服务数 vs 实际目录数 | 一致 | ❌ 阻止：知识库与代码仓不同步 |
| API注册表总数 vs 实际端点数 | 差异<10% | ⚠️ 警告：可能遗漏API |

> **历史遗留**：`.sync-record.json` 仅 mes-verify-state-migration 专项核查时引用，不在新鲜度检查中。

### 1.3 新鲜度评级

| 评级 | 条件 | 影响 |
|------|------|------|
| 🟢 新鲜 | `state.yaml.sync.last_sync` ≤ 7天，覆盖率≥90% | 可正常使用 |
| 🟡 需刷新 | `state.yaml.sync.last_sync` 7-14天，或覆盖率70-90% | 建议刷新，可用但需注意 |
| 🔴 过期 | `state.yaml.sync.last_sync` > 14天，或覆盖率<70% | **阻止进入需求分析阶段** |

> 不允许对历史遗留文件做运行时 fallback；若 `state.yaml.sync` 缺失，应先修复统一状态源，再继续执行新鲜度审计。

## 二、上下文预算审计

## 预算基线引用

本文档负责预算审计流程、预算状态分级与处置要求。

涉及以下内容时，应以 `.opencode/references/mes-ai-reference/rules/context-budget-baseline.md` 为准：

- 常驻预算
- 单阶段预算
- scenario 预算
- governance / summary 预算
- heavy 文档预算
- 超预算回退顺序的基线口径

如本文档与预算基线的数值或分层口径出现冲突，以预算基线为准，并应同步修正审计规则中的相关引用。

### 2.1 审计频率

每个 Command 执行时 `mes-guard-context-budget` Skill 自动审计。

### 2.2 审计记录

每次审计输出到对话中（不写文件），格式：

```
📊 上下文预算审计
阶段：[阶段名] | 需求：REQ-XXX
读取文件数：X个 | 预估消耗：XXK token
预算状态：✅ 在预算内 / ⚠️ 接近上限(>80%) / ❌ 超限
执行策略：直接执行 / 分批处理 / 压缩摘要
```

### 2.3 超限处理

| 超限程度 | 处理 |
|---------|------|
| 80-100% | ⚠️ 警告，建议压缩 |
| 100-120% | 必须压缩或分批 |
| >120% | 拒绝执行，必须拆分需求 |

---

## 三、强制执行规则

1. **新鲜度红色** = 阻止进入需求分析阶段
2. **预算超限** = 必须压缩或拆分，不可强行读取
3. **审计记录不可跳过** = 每个 Command 必须执行
