# 阶段门禁使用标准（phase-gate-usage-standard）

## 一、门禁类型
- 阶段进入门禁：判断能否开始正式执行
- 步骤消费门禁：判断当前输出能否被下一步骤消费
- 阶段退出门禁：判断能否结束并进入下一阶段
- 汇总消费门禁：判断局部结果能否被主控收口

## 二、强制触发时机
1. analyze / design / develop / test / deliver / emergency / refresh 进入前
2. 每个步骤输出被下游消费前
3. 任一阶段退出前
4. 共享知识收口、数据库迁移、高风险发布等关键场景前后

## 三、门禁最小内容
1. 门禁类型
2. 检查目标
3. 检查对象
4. 检查依据
5. 检查项与结果
6. 问题清单
7. 风险评估
8. 门禁结论
9. 证据路径
10. 后续动作

## 三点五、常见专项 must-pass（新增）
- **表真实性专项**：涉及数据库设计时，必须检查表类型、事实来源、证据路径、新增批准依据。
- **Java 引用可解析性专项**：涉及后端代码生成时，必须检查包声明、import、类型、方法、返回值是否真实可解析。
- **MyBatis 映射一致性专项**：涉及 Mapper/XML 生成时，必须检查 `namespace`、`id`、`parameterType`、`resultType`、`resultMap` 是否真实一致。
- **Provider 契约一致性专项**：涉及内部接口调用或跨服务调用时，必须检查 provider 的真实参数、返回值、泛型、包装层与调用方处理方式一致。
- 上述专项一旦命中，均应按 must-pass 处理，不得降级为“建议检查”。

## 四、结论口径
- 通过
- 有条件通过
- 不通过
- 需返工后重检
- 最小可交付，可继续推进但不可视为完整通过

## 五、失败处理
- 当前层级返工
- 不得带病流转
- 必须记录返工点与重检条件
- 命中专项 must-pass 失败时，必须明确记录失败对象、缺失证据和重检前置条件

## 六、统一引用写法
“阶段门禁与步骤消费门禁的触发时机、结论口径与落盘要求，必须符合 `.opencode/references/mes-ai-reference/rules/governance/phase-gate-usage-standard.md`。”

“若命中数据库设计、后端代码生成、MyBatis 生成或内部接口调用场景，必须追加执行对应真实性专项 must-pass 检查。”
