# 跨库访问架构约束规则（database-cross-access-rule）

> 本文件定义跨库访问的架构约束规则，防止绕过限定通道直接跨库查询导致数据一致性风险。
> 本文件当前为**候选结论**状态，待人工确认后方可升级为确认规则。

---

## 一、架构现状说明

### 1.1 数据库分布

| 数据库名称 | 别名 | 说明 |
|------------|------|------|
| 前处理数据库 | pre-process-db | 前处理业务数据存储 |
| 现场执行数据库 | execution-db | 现场执行业务数据存储 |

### 1.2 微服务数据库连接分布

| 服务类型 | 数量 | 数据库连接 | 说明 |
|----------|------|------------|------|
| 只连前处理数据库的服务 | 7个 | pre-process-db | 只能访问前处理数据 |
| 只连现场执行数据库的服务 | 5个 | execution-db | 只能访问现场执行数据 |
| 双库连接服务 | 3个 | pre-process-db + execution-db | 可同时访问前处理和现场执行数据 |

## 二、架构约束规则（候选结论）

### 2.1 跨库访问基本原则

| 规则编号 | 规则名称 | 规则描述 | 置信度 | 来源 |
|----------|----------|----------|--------|------|
| DB-CROSS-001 | 单库服务禁止直接跨库访问 | 只连接单个数据库的微服务，禁止直接连接另一个数据库执行查询 | **候选** | 用户架构说明 |
| DB-CROSS-002 | 跨库数据必须通过微服务调用获取 | 单库服务获取跨库数据，只能通过调用其他微服务的API实现 | **候选** | 用户架构说明 |
| DB-CROSS-003 | 现场执行服务调用前处理服务必须通过限定通道 | 现场执行的微服务调用前处理的微服务，只能通过特定的限定通道服务进行 | **候选** | 用户架构说明 |

### 2.2 跨库访问路径约束

#### 允许路径

| 调用方服务类型 | 被调用方服务类型 | 调用路径 | 约束说明 |
|----------------|------------------|----------|----------|
| 只连execution的服务 | 只连pre的服务 | 现场执行服务 → 限定通道服务 → 前处理服务 | 必须通过限定通道服务 |
| 双库服务 | pre-process-db | 直接访问 | 允许（双库服务可直接访问前处理数据） |
| 双库服务 | execution-db | 直接访问 | 允许（双库服务可直接访问现场执行数据） |

#### 禁止路径

| 禁止路径 | 禁止原因 | 替代方案 |
|----------|----------|----------|
| 只连execution的服务 → 直接访问pre-process-db | 违反单库服务禁止直接跨库访问规则 | 通过限定通道服务调用前处理服务 |
| 只连pre的服务 → 直接访问execution-db | 违反单库服务禁止直接跨库访问规则 | 通过限定通道服务调用现场执行服务 |
| 现场执行服务 → 直接调用前处理服务（绕过限定通道） | 违反限定通道架构约束 | 必须通过限定通道服务中转 |

## 四、架构约束违反风险

### 4.1 违反约束的风险类型

| 风险类型 | 说明 | 影响 |
|----------|------|------|
| 数据一致性风险 | 绕过限定通道直接跨库查询，可能导致数据不一致 | 两个数据库的数据状态可能不同步 |
| 事务边界风险 | 跨库直接访问无法保证事务一致性 | 无法实现跨库事务，可能导致部分成功部分失败 |
| 架构治理风险 | 绕过限定通道破坏分层解耦架构 | 架构约束失效，后续难以维护和扩展 |
| 性能风险 | 跨库直接查询可能引入性能问题 | 无法利用限定通道的缓存、优化策略 |

### 4.2 风险等级标注

| 场景 | 风险等级 | 说明 |
|------|----------|------|
| 只连execution的服务 → 直接访问pre-process-db | **高风险** | 违反架构约束，数据一致性无法保障 |
| 现场执行服务 → 绕过限定通道直接调用前处理服务 | **高风险** | 违反分层解耦约束 |
| 双库服务 → 直接访问任意库 | **低风险** | 符合架构约束 |

## 五、需求分析阶段应用

### 5.1 跨库数据访问场景识别

当需求涉及以下场景时，必须应用本规则：
- 现场执行服务需要查询前处理数据（如订单状态、订单信息）
- 前处理服务需要查询现场执行数据（如执行状态、履行记录）
- 单库服务需要获取另一个库的数据

### 5.2 Provider选择必须考虑架构约束

当多个服务均可提供某项跨库数据时，必须区分：

| Provider类型 | 说明 | 选择原则 |
|--------------|------|----------|
| 技术可达Provider | 技术上可以调通的服务（如直接跨库查询） | **不得直接选择**，需判断是否符合架构约束 |
| 架构允许Provider | 符合架构约束的服务（如通过限定通道） | **优先选择**，符合分层解耦架构 |
| 最终选定Provider | 在架构允许前提下选择的具体服务 | 必须是架构允许Provider中的一个 |

### 5.3 需求分析产物必须标注跨库访问

当需求涉及跨库数据访问时，需求规格必须标注：
- 调用方服务类型（只连单库 / 双库）
- 数据来源数据库（pre-process-db / execution-db）
- 是否需要通过限定通道
- 限定通道服务的候选名称（待确认）

## 七、开发阶段应用

### 7.1 禁止直接跨库查询代码

开发阶段禁止编写以下代码：

```java
// ❌ 禁止：直接跨库查询
// 只连execution的服务，直接查询pre-process-db
String sql = "SELECT status FROM pre_process_db.t_order WHERE id = ?";
jdbcTemplate.query(sql, orderId);

// ❌ 禁止：绕过限定通道直接调用
// 现场执行服务，绕过限定通道直接调用前处理服务
preProcessService.getOrder(orderId);  // 缺少限定通道服务前缀
```

### 7.2 必须通过限定通道调用

开发阶段必须编写以下代码：

```java
// ✅ 正确：通过限定通道调用
// 现场执行服务，通过限定通道服务调用前处理服务
OrderDTO order = channelService.getOrder(orderId);  // 通过限定通道服务

// ✅ 正确：双库服务可直接访问
// 双库服务，可直接查询pre-process-db
Order order = preProcessDao.selectById(orderId);
```
