# 需求状态追踪模板

> 本模板定义 `workspace/status-tracker.md` 的标准结构。
> 用于承接阶段门禁、步骤级门禁、退回原因、阻塞状态与条件通过闭环状态。

---

## 一、状态定义

| 状态 | 含义 | 可转换到 |
|------|------|---------|
| `draft` | 需求已提交，未开始分析 | `analyzing` |
| `analyzing` | 需求分析中 | `designed` / `draft` |
| `designed` | 设计完成，待开发 | `developing` / `analyzing` |
| `developing` | 开发中 | `testing` / `designed` |
| `testing` | 测试中 | `delivering` / `developing` |
| `delivering` | 交付中 | `closed` / `testing` |
| `closed` | 已交付关闭 | — |
| `blocked` | 被阻塞 | 恢复原状态 |

## 三、更新规则

1. 阶段进入门禁通过后，更新 `状态/当前阶段/当前步骤/最近门禁结论`
2. 步骤级门禁通过后，更新 `当前步骤/最近门禁结论`
3. `⚠️有条件通过` 时，必须将 `条件闭环状态` 更新为 `待闭环`
4. `❌不通过` 时，必须更新 `退回原因`
5. 闭环完成后，将 `条件闭环状态` 更新为 `已闭环`
6. 若知识可信度被 refresh 标记为 dirty 或 partial，且命中当前需求范围，必须更新 `是否受refresh影响`
7. 若当前需求关联紧急修复编号，必须更新 `关联EMG`
8. `依赖知识基线` 应记录当前消费所依赖的知识版本、日期或状态说明
9. 阶段退出依赖 `stage-output-report.md` 时，必须同步更新 `阶段产物报告` 字段为对应报告路径或留痕标识
