# 测试可测性矩阵模板

> 用于生成 `knowledge/code-map/testability-matrix.md`。
> 目标是量化服务/模块的可测性，帮助 AI 判断哪些变更需要补测试、哪些区域需要人工验证或 Mock。

## 一、文件定位

- **文件角色**：测试可测性矩阵
- **适用阶段**：设计 / 开发规划 / 测试
- **使用原则**：按服务/模块记录可测性事实，不以主观印象替代证据

## 二、可测性评分规则

| 评分维度 | 说明 | 分值范围 |
|----------|------|----------|
| 自动化测试存在性 | 是否已有测试资产 | 0-5 |
| 数据准备复杂度 | 数据依赖与准备难度 | 0-5 |
| 环境依赖复杂度 | 是否依赖外部系统/环境 | 0-5 |
| 回归入口清晰度 | 是否有明确验证入口 | 0-5 |
| 故障可定位性 | 出问题后是否容易定位 | 0-5 |

## 五、消费建议

1. 设计阶段：优先识别低可测性对象，避免设计出无法验证的方案。
2. 开发阶段：低可测性对象必须补充验证策略，不得默认“之后再测”。
3. 测试阶段：按矩阵优先覆盖低可测性且高风险对象。
