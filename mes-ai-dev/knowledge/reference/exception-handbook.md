# 例外流程手册

> 对应摘要：`knowledge/reference/exception-handbook-summary.md`
> 若当前尚未确认命中标准流程之外的例外场景，优先读取摘要并禁止默认整篇读取正文。

本手册定义标准流程之外的例外场景处理方式，确保特殊情况也有章可循。

## 紧急修复流程 (`/mes-emergency-fix`)

紧急修复场景已有独立 Command 负责执行，本章节仅作为入口说明。

- 适用对象：线上已发生且持续影响业务的高优先级故障。
- 使用方式：进入紧急修复流程时，统一通过 `/mes-emergency-fix` 发起，不在常规开发流程中旁路处理。
- 记录要求：修复原因、影响范围、修复动作、验证结果、回滚预案均需纳入紧急记录。
- 后续要求：紧急修复完成后，仍需补齐测试、文档与复盘信息。

### 紧急修复回流主流程要求

- 紧急修复若关联已有需求，必须记录 `related_requirement`
- 紧急修复完成后，必须明确 `backflow_strategy`：
  - 回流测试
  - 回流交付
  - 仅补知识与文档
  - 直接关闭（仅限独立事故场景）
- 紧急修复若修改了业务规则、热点链路、运行时配置或共享知识资产，必须补充：
  - `refresh_required`
  - `knowledge_backfill_required`
- 若修复结果仍需进入常规需求链路，则必须同步更新 `status-tracker.md`

## 回滚后补文档

### 场景描述

当变更已上线但因故障、风险或业务策略调整执行回滚后，仍需对本次变更与回滚过程补充完整文档，保证后续可追溯。

### 触发条件

- 已发生正式回滚操作。
- 回滚影响到已发布功能、配置、数据库脚本或外部接口行为。
- 原有交付文档无法完整反映回滚事实与原因。

### 补文档范围

- 变更目标与原计划内容。
- 回滚触发原因、影响范围与发现时间。
- 回滚动作、执行人、验证方式与最终状态。
- 后续修复计划、重新发布前置条件与风险提示。

### 补文档时限

- 原则上在回滚完成后 24 小时内补齐。
- 如遇跨团队协查，最长不超过 3 个工作日。

### 责任人

- 需求责任人负责组织补文档。
- 实际执行回滚的人员负责补充操作细节。
- 技术负责人负责审核完整性与准确性。

## 设计缺失先开发

### 适用场景

- 需求紧急，无法等待完整设计文档出齐。
- 改动范围明确且实现路径相对可控。
- 业务窗口极短，先行开发明显优于等待设计完成。

### 前置确认

- 至少需获得架构师口头确认。
- 需同步产品或业务责任人确认目标范围不再扩散。
- 需明确本次先开发的边界、已知风险与不可承诺项。

### 产出要求

- 开发前至少形成最小实现说明，包含目标、范围、风险与回退方式。
- 开发完成后 7 天内补齐设计文档。
- 补齐的设计文档需明确标注“先开发后补设计”的事实与原因。

## 历史脏代码区特殊处理

### 识别标准

- 位于遗留债务目录中已明确标记的区域。
- 已知存在命名不一致、模式混杂、依赖耦合或低可测性问题。
- 团队历史约定中已标识为短期内不做全面治理的代码段。

### 处理规则

- 不强制对齐新模式，避免在单次需求中引入大规模重构。
- 需加注释标记为遗留逻辑，说明本次修改目的与边界。
- 若必须绕过现行规范，应在变更记录中注明原因与后续治理建议。

### 修改范围最小化原则

- 仅修改实现当前目标所必需的代码。
- 不顺手清理无关遗留问题，避免扩大回归面。
- 若发现高风险隐患，可补充风险记录，但不在当前任务中强制治理。

## 初始化锁冲突与收口接管

### 适用场景

- 多 session 同时执行 `/mes-init-project` 或 `/mes-init-enrich`，且命中相同 repo/module/schema scope
- 多 session 同时触发 `/mes-init-converge`，争抢同一个全局收口入口
- 初始化 session 异常退出后，遗留 `init-repo-*` / `init-module-*` / `init-schema-*` / `mes-init-converge.lock`

### 处理原则

1. 初始化阶段锁只保护局部产物、状态片段与知识片段写入
2. 最终共享文件仍统一由 `/mes-init-converge` 收口，不允许因锁冲突而绕过收口规则
3. 锁冲突不得降级为“继续写局部产物试试看”
4. 强制接管必须留痕，且需记录接管原因、接管人、原锁信息与补偿动作

### 初始化锁冲突处理

当 `init-repo-*` / `init-module-*` / `init-schema-*` 发生冲突时，只允许以下选择：

| 处理方式 | 适用场景 | 必须记录项 |
|----------|----------|------------|
| 等待 | 当前 scope 必须执行，且原锁仍有效 | 锁文件内容、等待开始时间、最大等待时长 |
| 跳过当前 scope | 本次可先处理其他不冲突 scope | 被跳过 scope、原因、后续处理计划 |
| 仅执行不冲突 scope | 批量初始化中部分 scope 冲突 | 已执行 scope、跳过 scope、剩余 pending |
| 强制获取 | 原 session 已异常退出、锁超时、经确认可接管 | 原锁信息、接管时间、接管审批人/责任人 |

### 收口锁冲突处理

当 `mes-init-converge.lock` 冲突时：

1. 默认优先等待，不允许并行执行第二个 `/mes-init-converge`
2. 若需强制接管，必须先确认：
   - 原 session 已异常退出或超时
   - 原锁未继续推进状态更新
   - 本次接管不会覆盖未确认的共享文件结果
3. 强制接管后，必须在收口报告与例外记录中说明：
   - 原收口范围
   - 接管范围
   - 未完成片段清单
   - 是否重新执行热点重算

### 死锁/锁泄漏恢复

#### 判定条件

- 锁文件存在，但对应 session 已不存在或长时间无更新
- checkpoint / state.yaml / 收口报告均无后续推进痕迹
- 同一 scope 长时间被锁占用，阻塞后续初始化/深化/收口

#### 恢复步骤

1. 读取锁文件并核对 scope、session、started、expires
2. 检查对应状态片段、知识片段、checkpoint 是否仍在推进
3. 若确认原 session 已失活：
   - 标记原锁为失效
   - 记录强制接管例外记录
   - 由新 session 重新获取锁
4. 若无法确认原 session 状态：
   - 升级给技术负责人
   - 暂停该 scope 的进一步写入

### 必须补偿动作

- 更新 `state.yaml.initialization.recent_execution` / `convergence` 中的待收口状态
- 在 `workspace/refresh/*-converge-report.md` 或对应阶段报告中记录锁冲突/接管事实
- 必要时补充 `workspace/refresh/*-fragment-convergence-checklist.md`，重新核对待收口片段是否受冲突影响
- 若强制接管过 `mes-init-converge.lock`，必须重新核对热点候选结果与最终共享文件是否一致

## 并行任务部分失败处理

### 适用场景

- 多 Agent 并行执行时，部分支路已完成，部分支路失败或待补证
- 已形成局部结果，但尚不满足整体汇总条件

### 处理原则

1. 成功支路结果必须保留，不得因其它支路失败而整体回滚
2. 失败支路仅重派 pending 范围，不得无差别重做全部支路
3. 若支路仍持有锁，必须在 handoff 或例外记录中明确锁状态
4. 若支路结果仅可局部消费，必须显式标注，不得直接汇总为全局结论

### 必须记录项

- 已完成项
- 失败项
- Pending 项
- 建议重派范围
- 当前锁状态
- 当前知识消费口径

### 升级条件

- 无法判断成功支路是否仍可信
- 锁状态异常悬挂且无法确认接管条件
- 局部结果与全局结论边界不清，存在误汇总风险

## 跳过测试的紧急上线

- 仅限 P0 级生产故障。
- 必须获得运维总监与技术负责人双签。
- 上线前需至少完成最小人工验证，并明确回滚方案。
- 上线后 48 小时内必须补齐测试，并补录跳过测试原因、风险评估与补偿结果。

## 例外流程审批矩阵

| 例外类型 | 最低审批人 | 必须记录项 | 后续补偿动作 |
| --- | --- | --- | --- |
| 紧急修复 | 技术负责人 | 故障现象、影响范围、修复方案、验证结果、回滚预案 | 补测试、补文档、补复盘 |
| 回滚后补文档 | 技术负责人 | 回滚原因、执行时间、影响对象、验证结论 | 24 小时内补文档并更新交付记录 |
| 设计缺失先开发 | 架构师 | 口头确认时间、范围边界、已知风险、补设计时限 | 7 天内补设计文档 |
| 历史脏代码区特殊处理 | 模块负责人 | 遗留区域说明、最小修改范围、未治理原因 | 补充遗留标记与风险说明 |
| 跳过测试的紧急上线 | 运维总监 + 技术负责人 | 故障等级、跳测原因、人工验证结果、回滚方案 | 48 小时内补测试并补记录 |
| 初始化锁强制接管 | 技术负责人 | 原锁信息、接管原因、影响 scope、补偿动作 | 补齐例外记录、重新核对待收口状态与收口结果 |
