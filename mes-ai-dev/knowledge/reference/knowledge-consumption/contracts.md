# contracts 消费分片

## 一、什么是契约级知识

契约级知识是指会影响多个模块、多个服务、多个阶段共同理解与实现方式的共享约束，不等同于普通业务字段说明或单接口示例。

典型契约级知识包括：
- 统一响应包装
- 错误码与错误扩展参数
- SDK 请求/响应模型
- 公共异常模型
- MQ 消息体
- 认证与网关透传规则
- 链路透传字段（如 traceId）

## 二、定义点优先原则

契约级知识必须优先从定义点消费，再按需查看使用点。

定义点优先顺序：
1. SDK / common / shared / integration 中的定义类
2. 公共异常处理器、统一封装器
3. 接口签名与服务接口
4. controller/service 使用示例
5. 模板与文档说明

不得用使用点代替定义点。
不得用模板代替项目事实。

## 三、三态规则

契约级知识只允许以下三态：
- 确认结论
- 候选结论
- 明确未知

候选结论必须标注低置信度和待补证项。
明确未知不得用常识补齐。

## 四、空模板不可消费原则

空模板、占位态文档、仅有章节骨架的文档，不得作为契约级知识输入下游阶段。
文件存在不等于知识存在。

## 五、消费建议

遇到以下问题时，应优先进入契约级知识消费：
- 统一响应长什么样
- 成功/失败如何判定
- 错误码从哪里来
- SDK 模型如何传递
- MQ 消息体长什么样
- 认证与网关字段如何透传

## 五点五、模板使用建议

当契约级知识需要正式落盘时，应优先使用对应模板，而不是直接写成零散说明或接口表象总结：

- 统一响应契约 → `templates/reference/response-conventions-template.md`
- 认证契约 → `templates/reference/authentication-conventions-template.md`
- MQ 契约 → `templates/reference/mq-conventions-template.md`
- 网关契约 → `templates/reference/gateway-conventions-template.md`

这些模板统一要求记录：
- 来源类型
- 来源定位
- 获取方式
- 版本一致性说明
- 真实样例或真实使用点

若当前只能观察到局部使用现象，仍不得跳过来源类型与待补证项，直接写成全局确定性契约。

## 六、公共契约源获取顺序

当契约级知识定义在业务仓之外时，消费顺序必须遵循以下优先级：

1. `workspace-source`
   - 直接读取当前 workspace 中的源码定义
2. `repo-source`
   - 读取已纳入当前工作区范围的公共源码仓
3. `sources-jar`
   - 读取与当前依赖版本对应的源码附件
4. `decompiled-jar`
   - 在源码与源码附件均不可得时，作为受控兜底路径
5. `unknown`
   - 若事实源不可得，则明确标记未知，不得补洞

使用点只能辅助理解，不得替代定义点。
jar 反编译只能兜底，不得作为默认路径。

## 七、来源类型与置信度

契约级知识必须记录来源类型。推荐取值如下：

- `workspace-source`
- `repo-source`
- `sources-jar`
- `decompiled-jar`
- `partial-evidence`
- `unknown`

默认置信度建议如下：

- `workspace-source`：高
- `repo-source`：高
- `sources-jar`：中高
- `decompiled-jar`：中/低
- `partial-evidence`：低
- `unknown`：不可消费

若来源类型为 `decompiled-jar`、`partial-evidence` 或 `unknown`，不得输出无风险、无保留的全局确定性结论。

## 八、版本一致性与风险说明

若契约级知识来自源码附件或反编译 jar，必须补充版本一致性说明，至少回答：

- 当前读取对象是否与项目实际依赖版本一致
- 若版本一致性无法确认，是否已标记待补证
- 若来自反编译结果，是否已标记语义缺失风险

以下风险必须显式记录：

- 注释与设计意图缺失
- 泛型、注解或内部结构信息不完整
- 仅能观察结构，难以还原设计边界
- 反编译结果可能与真实源码存在表达差异

来源版本不明时，不得输出高置信度全局契约。

## 九、风险提示

若契约级知识未识别清楚，后续分析、设计、开发、测试会系统性偏移。
契约级知识不清时，应优先补证，而不是继续产出看似完整的下游结论。
