# 阶段门禁通用规则

## 一、使用原则

1. `mes-verify-phase-gate` 默认以 **must-pass** 作为阻断条件。
2. **should-check** 未满足时，必须在评审记录或阶段结果中明确风险与补偿动作。
3. **advisory** 用于持续改进，不得冒充阻断项。
4. 所有阶段的标准化评审要求统一适用本文件。
5. 所有步骤级输出在被下一步骤消费前，必须先通过步骤级门禁。
6. 上一步骤任一 must-pass 不通过，则当前步骤打回重做，下一步骤不得启动。
7. 每个阶段退出前必须生成阶段完成产物报告；未生成或内容不达标不得退出。
8. 每个消费类阶段进入前必须完成阶段记忆消费检查；退出前必须完成阶段记忆更新检查。

## 二、标准化评审要求

### 2.1 must-pass

| 检查项 | 要求 |
|---|---|
| 评审记录存在 | 必须有评审记录文件或评审结论留痕 |
| 评审结论标准化 | 结论必须为：✅通过 / ⚠️有条件通过 / ❌不通过 |
| 详细审查报告完整 | 不得只保留轻量结论，必须形成正式详细审查报告 |
| 详细报告字段齐全 | 至少包含：目标范围、对象输入、依据、检查结果、问题清单、风险评估、整改建议、结论、证据路径、复审状态 |
| 审查时间存在 | 审查报告必须记录明确审查时间 |
| 结论具备证据链 | 禁止猜测性“通过” |
| 不通过不得放行 | 结论为 ❌不通过 时必须整改后重评 |
| 上一步不通过不得进入下一步 | 任一步骤/阶段未通过时，不得消费其产物 |
| 阶段完成产物报告存在 | 阶段退出前必须有阶段完成产物报告 |
| 阶段完成产物报告内容完整 | 必须列出标准产物、已生成文件、未生成文件及原因 |

## 二点五、结论可追溯性要求

共享知识、阶段产物、规范类文档与阶段结论中的关键判断，必须具备可追溯性。

可追溯性至少包括：
- 事实来源文件
- 定义点或主依据位置
- 使用点或消费方位置（如适用）
- 适用范围
- 置信度或未确认说明（如适用）

文件存在不等于知识存在。
模板存在不等于规范存在。
内容完整不等于事实成立。

若关键结论来自模板补全、通用经验、隐式常识或未验证推断，而非项目事实源，则不得视为通过。

### 2.2 should-check

| 检查项 | 要求 |
|---|---|
| 参与角色完整 | 关键阶段应包含编写人、技术评审人、业务/使用方评审人 |
| 有条件通过已闭环 | ⚠️有条件通过 的修改项应在下一阶段前完成 |
| 评审产物清单完整 | 应列出被评审文件与结论 |
| 证据与整改可追溯 | 应能定位问题证据、整改责任和闭环状态 |
| 阶段记忆已消费 | 应能说明上一阶段 handoff 和记忆是否已读取 |

## 三、步骤级强制门禁要求

### 3.1 must-pass

- 每个步骤在输出结果前必须执行一次步骤级 exit gate 审查
- 下一步骤开始前必须确认上一步骤 gate 结论为 `✅通过`
- 若上一步骤为 `⚠️有条件通过`，必须先完成整改闭环，再允许进入下一步骤
- 任一步骤 gate 结论为 `❌不通过`，必须打回当前步骤重做，不得将输出注入下一步骤
- 任一步骤或阶段审查若缺少完整证据链，必须判定为 `❌不通过`
- 任一阶段退出若缺少完整详细审查报告或缺少审查时间，必须判定为 `❌不通过`

### 3.2 should-check

- 步骤级 gate 报告应记录：输入、输出、审查结论、退回原因、重做要求
- 步骤级 gate 报告应按详细审查报告结构记录关键检查项、问题、风险、整改要求和证据路径
- 返工后应重新执行相同步骤 gate，而不是沿用旧结论

### 3.3 mandatory-record

- 任一步骤 gate 结论为 `❌不通过` 时，必须生成门禁审查记录文件
- 任一步骤 gate 结论为 `⚠️有条件通过` 时，必须生成门禁审查记录文件并记录闭环状态
- 任一跨阶段出口 gate 必须生成门禁审查记录文件
- 初始化收拢、发布执行、知识刷新、状态迁移校验相关 gate 必须生成门禁审查记录文件
- 所有强制落盘的门禁记录必须满足详细审查报告最小字段集

## 四、与阶段分片的边界

### 本文件负责什么
- 通用评审要求
- 通用步骤级门禁要求
- 通用 mandatory-record 要求

### 本文件不负责什么
- 各阶段的专属 enter / exit 条件
- 初始化收敛消费状态门禁
- GSD Continue Exit 的完整判断条件

这些内容应继续读取对应分片，不得把当前文件当成单一完整门禁源。

## 五、通用阻断条款（补充）

满足以下任一条件时，应阻断进入下游阶段：

- 关键结论不可追溯
- 空模板/占位文件被当作已知规范消费
- 使用点结论冒充定义点结论
- 项目私有契约被通用框架常识替代
- 候选结论被写成确定事实且未标风险
- 关键结论若依赖业务仓之外的公共定义源，而来源类型未确认、来源定位不可追溯或版本一致性不明
- 关键契约对象仅基于反编译结果得出，但未标注风险、置信度与版本说明
- 来源未知却输出全局统一结论
