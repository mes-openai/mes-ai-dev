# Agent 与工具路由参考

> 本文用于承接 developer 层中与 Agent、工具选择相关的说明性内容。
> 目标是把“何时用谁、为何这样路由”的细节下沉为可按需读取的参考文档，而非长期常驻在高层提示词中。

---

## 一、使用目标

本参考用于回答以下问题：

- 当前请求是直接用本地工具处理，还是交给专门 Agent？
- 什么时候优先 `explore`，什么时候优先 `librarian`？
- 哪些情况需要 `oracle`、`metis`、`momus`？
- 什么时候不应该过度委派？

---

## 二、总原则

- 优先选择“最小且足够”的执行手段。
- 已知文件、已知位置、已知模式的简单问题，优先本地工具。
- 跨模块、结构不熟、需要找模式时，优先探索型 Agent。
- 涉及外部库、外部文档、开源实现示例时，优先外部参考型 Agent。
- 需要高难度架构判断或多次失败后的调试时，再引入高成本顾问型 Agent。

---

## 三、常见路由表

| 场景 | 推荐手段 | 原因 |
|---|---|---|
| 已知文件里的单点修改 | 本地工具 | 范围清晰，直接处理成本最低 |
| 单关键词、单文件搜索 | `grep` / `read` / `glob` | 不必为简单检索额外委派 |
| 跨模块找实现模式 | `explore` | 更适合做结构化探索 |
| 外部库接入方式、官方文档、开源用法 | `librarian` | 更适合查外部资料 |
| 高难架构取舍 | `oracle` | 需要更强推理与只读咨询 |
| 请求本身含糊、规划成本高 | `metis` | 适合先做前置澄清与隐含意图分析 |
| 已保存计划的质量审查 | `momus` | 适合检查计划可执行性与完整性 |

---

## 四、Agent 选择建议

### 4.1 `explore`

适用：

- 需要在代码库中找实现模式。
- 需要快速理解多个模块之间的关联。
- 需要定位“哪里做了某件事”。

不适用：

- 已知文件、已知位置的简单修改。
- 单个关键词即可解决的问题。

### 4.2 `librarian`

适用：

- 外部库接入方式。
- 官方文档确认。
- 开源实现样例。

不适用：

- 纯本地代码结构问题。

### 4.3 `oracle`

适用：

- 架构取舍。
- 高难调试。
- 多次修复失败后的根因判断。
- 安全 / 性能 / 边界复杂问题。

不适用：

- 普通单文件小改。
- 第一次尝试就能解决的显性问题。

### 4.4 `metis`

适用：

- 需求很模糊。
- 计划复杂且容易误判。
- 需要先识别隐藏意图与失败点。

### 4.5 `momus`

适用：

- 已产出计划文档，准备做质量审查。
- 需要检查计划是否清晰、可验证、无明显缺口。

---

## 五、何时不应过度委派

以下情况应避免把简单问题复杂化：

- 已知文件中的单点文案修改。
- 已知位置的 Markdown 结构调整。
- 只需读取一个文件即可回答的问题。
- 单个明确路径的增删改。

原则：

- 不为“看起来专业”而委派。
- 不为“可能更全”而对简单问题过度检索。

---

## 六、避免重复探索

- 已经交给 `explore` / `librarian` 的主题，不应再手工做同样搜索。
- 委派后应继续做不重叠的工作，或等待结果。
- 不要一边等待 Agent，一边自己重复执行同类检索。

---

## 七、与本地工具的关系

本地工具仍然适用于高频、低复杂度场景：

- `read`：已知文件内容读取
- `glob`：按文件名匹配路径
- `grep`：按正则搜索内容
- `apply_patch`：直接做文本改动

当问题从“读取事实”升级为“理解结构、比较模式、找外部依据”时，再切换到 Agent。
