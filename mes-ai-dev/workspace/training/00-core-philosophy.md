# MES-AI-DEV 骨架核心思想

> 本文档用于培训，阐述骨架的核心理念与设计动机。

---

## 一句话概括

> **让 AI 不是"写代码的工具"，而是"可以被纳入正式研发主链路的可控工程单元"。**

骨架围绕三条主线构建：

同时，所有阶段默认遵循四个执行原则：

| 原则 | 骨架内含义 | 解决的问题 |
|------|------------|------------|
| 编码前思考 | 在进入实现或生成前，先暴露假设、困惑、边界、取舍与验证目标 | 错误假设、隐藏困惑、缺少权衡 |
| 简洁优先 | 优先最小必要方案、最少抽象、最短闭环，不为“看起来完整”制造复杂度 | 过度复杂、臃肿抽象 |
| 精准修改 | 只触碰与目标直接相关的规则、文档、代码和测试，不扩大变更面 | 无关编辑、触碰不应碰的代码 |
| 目标驱动执行 | 以可验证成功标准推进，测试、门禁和证据链优先于主观完成感 | 通过测试优先、成功标准不可验证 |

这四个原则不是口号，应在阶段计划、步骤门禁、详细审查与阶段退出结论中被显式检查。

---

## 主线一：知识先行，而非代码先行

传统 AI 辅助开发是"拿到需求直接写代码"。骨架的核心反转是：

```
常规思路：需求 → 直接写代码 → 出问题再改
骨架思路：先建知识底座 → 再消费知识做需求分析 → 再消费分析做设计 → 再消费设计写代码
```

这就是 **规格驱动开发（SDD）**。初始化阶段花大量精力建四层索引、抽取契约、识别热点——不是在"做准备工作"，而是在建一个**后续所有阶段都可以直接消费的结构化知识基线**。没有这个基线，AI 就是在黑盒里猜代码。

---

## 主线二：AI 可治理，而非 AI 自由发挥

骨架不信任 AI 的自由输出。整套治理机制本质上是在回答一个问题：

> **AI 做的每一步，怎么证明它是对的？**

具体手段：

| 治理机制 | 做了什么 |
|---------|---------|
| **三层层级门禁** | 每个阶段有进入/步骤/退出三层检查，不通过就打回 |
| **真实性专项审查** | 不允许 AI 用模板、通用常识、框架默认值补洞——每项都必须有真实代码/契约作为事实来源 |
| **限边规则** | 开发阶段不得重拍需求分析的仓边界决策，不得重拍设计阶段的 Provider 选择——各阶段各管各的决策权 |
| **服务链冻结** | 设计阶段冻结调用路径后，开发阶段只能沿冻结路径走，不允许"技术上可实现就绕路" |
| **空模板阻断** | 文件存在不等于知识存在，占位态模板不得被当作正式规范消费 |
| **产物分类管理** | deliverable / report / evidence / handoff / memory / working 分类存放，不混杂 |

**核心思想是：把 AI 约束在可检查、可复盘、可追责的工程支架（Harness）内。**

---

## 主线三：长期可持续，而非一次性消费

骨架不是为一次需求设计的，而是为**持续几十个需求、跨多个人、跨多个 session** 的长期协作设计的：

| 机制 | 解决的问题 |
|------|----------|
| **阶段记忆持久化** | 坑点台账、决策日志、阻塞记录结构化沉淀，下次需求直接继承 |
| **知识刷新** | 代码变了，知识库增量同步，不需要重新全量初始化 |
| **增量式建设** | 大仓可以先单仓初始化，再 converge 收口，不需要一口气做完 |
| **断点续传** | 初始化中断后从 checkpoint 继续 |
| **四层索引** | L0 总览 → L1 索引 → L1.5 文件摘要 → L2 精准源码，大仓不爆上下文 |
| **契约三态** | 确认/候选/未知，承认"不确定"比假装"都知道"更安全 |

在复杂仓库或跨仓场景中，骨架可按需吸收代码知识图谱经验：

- **GitNexus 类能力**：用于代码结构、依赖关系、调用链、影响范围、重命名/重构风险与架构文档生成的图谱化辅助分析。
- **graphify 类能力**：用于把代码、文档、图示、交接材料与设计依据转成可查询知识图谱、图谱报告或 wiki 化导读。

这些能力在骨架中作为“知识消费与证据增强手段”使用，不替代现有阶段门禁、事实来源标注和人工确认。

---

## 总结成三个关键词

1. **可治理**（Governable）—— 每一步有门禁、有审查、有证据链、有回退路径
2. **可持续**（Sustainable）—— 知识不是用完即弃，而是越做越完整、越做越稳
3. **可追责**（Accountable）—— 决策有 ADR、产物有分类、变更有留痕、错误有阻断

用骨架 README 原话说：

> **MES-AI-DEV 不是单点的 AI 写代码工具，也不是只会生成文档的流程壳子，而是让 AI 能进入正式研发主链路、持续接管更多工程工作单元的治理骨架。**
